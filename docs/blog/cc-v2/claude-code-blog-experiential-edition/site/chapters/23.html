1台のMacで動いた仕組みを、100人のMacへ 最適Architecture

『自分の Mac では完璧に動く Skill が、隣の人の Mac では動かない』。 これは個人の運用ではなく、組織の標準が無いことが原因です。

ほとんどの人は、AIの活用を個人の工夫として済ませています。 ですが、その工夫を社内で再現可能にするには、Architecture（どう配置し、どう同期し、どう更新するか）が要ります。

この章で扱うのは、トレプロハーネスの設計思想を借りて、自社用の Architecture を描く方法です。 正本・配布・強制・個人差分を分離し、Canary→Pilot→Stable で段階展開します。

この章を読み終わる頃には、『一人の成功』を『全社の標準』へ翻訳する目線が手に入ります。

実装リファレンス

23.1 Product定義

なぜここを読むか 全社配布の議論を始める前に、ハーネスが何を提供する製品かを一文で確定させる必要があります。目的が曖昧だと、後続のScope設計と完了判定が揺れてしまうためです。

やる／やらないとどうなるか 「全社員のMacへ同じ実行基盤を配布する」という定義を共有することで、CLAUDE.md・Commands・Hooks・Skillsの一元化と新人30分稼働という具体的なゴールに議論を絞り込めます。

全社員のMacへ、同じClaude Codeの知識、手順、安全制御、外部連携方針を配布し、継続的に改善する社内AI実行基盤。

現行目的：

  • 福岡個人の~/.claude/に滞留する知見をGitで全社員へ。
  • CLAUDE.md、13 Commands、5 Hooks、46 Skillsを共通化。
  • 誤操作をPreToolUse等でBlock。
  • 新人を30分以内で稼働。
  • Kandji/LaunchAgentで毎日更新。

23.2 現行資産

なぜここを読むか 新しい設計を描く前に、現在Gitで管理できている資産と、まだ個人領域に滞留している資産を切り分ける必要があります。土台の解像度がないまま4面構想に進むと、配布漏れと二重管理が発生するためです。

やる／やらないとどうなるか Guidance・手順・制御・Protocol・Reference・Codex・配布・Device・Testという9区分で現状資産を棚卸しでき、不足箇所と移管対象を客観的に並べられるようになります。

領域 資産
Guidance global-config/CLAUDE.md
手順 Commands、Skills
制御 Hooks
Protocol hub-PROTOCOL.md
Reference ref-skills/
Codex .agents/skills/
配布 install/bootstrap/update/sync
Device Kandji/LaunchAgent
Test verify.sh/test-hooks.py

23.3 追加すべき4面

なぜここを読むか 現状資産だけでは「全社強制」「Runtime二重管理」「事故抑制」が成立しないため、追加すべき面を明示する必要があります。Documentに書くだけでは社員の手元で外せてしまい、安全と一貫性が崩れるためです。

やる／やらないとどうなるか Managed Policy・Canonical Skill Source・Release Channel・Telemetry/Auditの4面が定義され、macOSのmanaged-settings.json等の固定領域に置くものが具体パスで決まります。

  1. Managed Policy — ユーザーが外せないPermission/Sandbox/MCP。
  2. Canonical Skill Source — Claude/Codexの二重編集を廃止。
  3. Release Channel — Canary → Pilot → Stable。
  4. Telemetry/Audit — Version、Update、Block、Failure、Cost。

macOS管理領域：

これは何か macOSの組織管理領域に置くファイルパスを示し、Managed PolicyとManaged MCPの設置場所を物理的に確定させます。

実行するとどうなるか 個人ユーザーが書き換えられない領域で組織CLAUDE.md・設定・MCP定義が固定され、Safety解除や設定改変を端末側で行えなくなります。

/Library/Application Support/ClaudeCode/CLAUDE.md
/Library/Application Support/ClaudeCode/managed-settings.json
/Library/Application Support/ClaudeCode/managed-mcp.json

23.4 推奨Repository

なぜここを読むか Skillの正本と配布生成物を分けないと、片方だけ修正される事故や、Build前のFileをそのまま配ってしまう事故が起きます。両Runtime向けに同じ正本から再現可能な成果物を生成するための骨格を先に決めておく必要があります。

やる／やらないとどうなるか skill-srcを正本、global-config/skills と .agents/skills を生成先とするRepository構造が共有され、scripts/build-skills.sh など整備すべきPath一覧が手戻りなく見えるようになります。

これは何か 推奨Repositoryの完全構造を示し、skill-srcを正本に global-config と .agents/skills を生成物として分離する配置を提示します。

実行するとどうなるか 正本と生成物の境界が明確になり、Skillの二重編集や生成物への直接コミットといった事故をRepository構造の段階で防げます。

23.5 Release Flow

なぜここを読むか main最新を全社員へ無条件に流すと、不具合の範囲がゼロ秒で全社に広がります。段階配布を入れ込まないと、止血の余地がない配布になってしまうためです。

やる／やらないとどうなるか PR→Test→Evaluator→Canary→Pilot→Stableの段階を経て承認Tagだけ配る運用が描けるので、事故が起きてもCanary数名規模で封じ込められるようになります。

これは何か PRから承認済みTag配布までの段階展開フローを一行で示し、各段階を通った成果だけが全社員Updateへ進める運用を定義します。

実行するとどうなるか Canary 2〜3名・Pilot部署・Stable全社の順で配布が進むため、不具合の影響範囲をPilot到達前に小さく封じ込められます。

PR → Static/Test/Evaluator → Canary Tag → 2〜3名 → 観測
→ Pilot部署 → Stable Tag → 全社員Update → Metrics

main最新を無条件配布せず、承認済みTagを配る。

23.6 Guidance/Enforcement Mapping

なぜここを読むか 「日本語で書く」「.env禁止」のようなルールが文章のお願いなのか機械強制なのかを揃えないと、守られないルールが量産されます。Guidance/Enforcementの境目を表で見える化する必要があります。

やる／やらないとどうなるか 各要求に対してCLAUDE.md・Hook・managed-settings・managed-mcpのどれで実装するかが対になり、文書任せにせず実行時に止める仕組みへ落とし込めるようになります。

要求 実装
日本語 CLAUDE.md
Test必須 CLAUDE.md + Hook/CI
.env禁止 Managed deny
Deploy承認 ask + Hook
危険DB名 PreToolUse Hook
TS編集後Check PostToolUse
MCP制限 managed-mcp/Allowlist
Bypass禁止 managed-settings
顧客文脈 SessionStart

23.7 個人Layer

なぜここを読むか 配布で個人カスタマイズを上書きすると、社員ごとの私物Skill・私物Promptが消えて配布への信頼が崩壊します。中央Manifestが「自分が配ったものだけ」を扱う約束を明文化する必要があります。

やる／やらないとどうなるか ~/.claude/local/ 配下の保護領域と apply-local-overlay.sh の運用が決まり、中央配布と個人差分が衝突した場合のLogと優先順位も合意できる状態になります。

~/.claude/local/を保護するだけでは自動発見されるとは限らない。運用を明文化する。

~/.claude/local/
├── CLAUDE.local.md
├── skills/
├── agents/
└── apply-local-overlay.sh

中央Manifestは、自分が配ったPathだけを削除する。名前衝突は中央優先でLogへ出す。

23.8 完了条件

なぜここを読むか Architectureの議論は、合格条件を箇条書きにしないと「だいたい出来た」で止まります。後で評価し直せる完了Checklistを置く必要があります。

やる／やらないとどうなるか 30分導入・Managed Policy優先・Skill正本1箇所・1 Command Rollback等の10項目が並び、Architecture章のゴール到達を客観的に判定できるようになります。

これは何か Architecture章の完了条件を10項目のチェックリストで列挙し、Skill正本1箇所・Claude/Codex同Fixture等の達成基準を並べます。

実行するとどうなるか 実装後にこのChecklistへ突き合わせれば、「設計が紙だけで終わっていないか」を機械的に判定できるようになります。

[ ] 30分導入
[ ] 承認Release配布
[ ] 全端末Version把握
[ ] Managed Policy優先
[ ] Bypass禁止
[ ] MCP統制
[ ] Skill正本1箇所
[ ] Claude/Codex同Fixture
[ ] 1 Command Rollback
[ ] 個人Layer保持
