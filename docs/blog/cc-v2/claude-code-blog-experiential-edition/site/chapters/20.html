AIが、寝てる間に走る `/goal`、Headless、定期実行

『朝の調査を、毎朝自分でやっている』。 これは AI の手抜きではなく、自律実行の設計をしていないだけです。

ほとんどの人は、AIを『呼んだら動くもの』として使っています。 ですが、明確なゴールを渡せば、AIは寝ている間にも走れます。

`/goal` と Headless 実行、定期実行は、AIに目的だけを渡し、停止条件と最大回数を決めて自走させる仕掛けです。 朝起きたら、調査・要約・草案の3つが机の上に揃っている──そういう状態を作ります。

ただし、無条件で走らせれば事故も無条件です。成功条件・タイムアウト・人の承認地点を必ず併せて設計します。

この章の概観 ／ 何を扱い、何ができるか

この章で扱う機能と、それで何ができるようになるか。実装に入る前に、全体像をここでつかみます。

  • /goal（成功条件と停止条件）

    成功条件・最大ターン・Timeout・許可範囲・失敗時停止を仕様化し、自律実行が安全に終わるよう設計するコマンド。

    入力
    自動化したい仕事、成功判定の機械条件
    出力
    完了 or 安全停止のどちらかで終わる自律実行

  • Headless実行

    対話画面なしで定型タスクを走らせる実行形態。CI・夜間バッチ・サーバー常駐に組み込む。

    入力
    実行内容、引数、許可範囲
    出力
    Headlessジョブの実行ログとExitステータス

  • 安全なWrapper（重複防止・ロック・ログ）

    /goalやHeadlessを実運用に乗せるための重複防止・ロック・ログ・通知・再試行を包んだラッパースクリプト。

    入力
    ベース実行コマンド、運用要件
    出力
    Wrapperスクリプトと運用ログ

  • Scheduled必須要件

    定期実行で必ず満たすべき重複防止・冪等性・通知の要件チェック。配信時刻のブレも管理する。

    入力
    スケジュール対象ジョブ、頻度
    出力
    必須要件チェック通過記録

  • LaunchAgent例

    macOS LaunchAgentによる定期実行設定の具体例。個人Mac〜開発機での自動化導入に使う。

    入力
    実行コマンド、頻度、ログ出力先
    出力
    動作するLaunchAgent plist例

  • 自動化しない領域の明示

    公開・送信・課金・削除など、最後の一歩を必ず人間に残す領域を明確化するルール。

    入力
    自動化候補リスト、副作用の有無
    出力
    自動化可/人間承認必須の振り分け表

実装リファレンス

20.1 自律化の4要素

なぜここを読むか 自律実行は始め方より止め方が難しく、Goal・Verifier・Budget・Stopの4要素が揃わないと事故になります。

やる／やらないとどうなるか 達成判定・上限・停止条件をそれぞれ明文化する設計テンプレが手に入り、後段の/goal仕様の骨格になります。

これは何か 自律化に必要な4要素を1行ずつ並べた最小フレームで、達成状態・判定方法・上限・停止を分けて書いています。

実行するとどうなるか 個別のジョブを設計する前に、何を欠けば事故になるかが視覚化され、/goal仕様の骨組みとして再利用できます。

それぞれの意味

Goal
何が達成された状態をゴールと呼ぶかを定義する要素です。これが曖昧だとAIは『どこまでやれば終わりか』を判断できず走り続けます。
Verifier
達成したかどうかを機械的に判定する方法です。テスト合格や型チェック通過など、人間の主観を挟まない条件を1個以上置きます。
Budget
ターン数・実行時間・トークン・費用などの上限です。設定し忘れると、達成できないまま無限ループで料金だけ加算され続けます。
Stop
成功・失敗・異常それぞれで停止する条件です。安全停止の出口が決まっているおかげで、達成不能でも事故にせず終わらせられます。
Goal      達成状態
Verifier  判定方法
Budget    Turn/時間/Token/費用上限
Stop      成功/失敗/異常停止

20.2 良いGoal

なぜここを読むか 『テストが通るまで直して』のような曖昧な目標は永遠に走り続ける原因なので、機械的に検証できる成功条件が要ります。

やる／やらないとどうなるか テスト・型・ビルド・E2E・Issue件数を全部満たすという形でGoalが書け、未達時の残課題保存先まで決まります。

これは何か テスト・型・ビルド・E2E・Issue件数の全充足を成功条件にし、最大15ターン30分の上限を付けた/goalの宣言文です。

実行するとどうなるか 曖昧な『直して』を機械検証可能な目標に置換でき、未達時にgoal-report.mdへ残課題が落ちる安全停止が成立します。

それぞれの意味

npm test成功
ユニットテスト全件パスを成功条件にします。AIに『直して』と曖昧に頼まず、機械的に合否を切れる第一の関門になります。
npm run typecheck成功
型エラー0を成功条件にします。テストは通るが型が壊れているといった部分修復の自己満足を排除できます。
npm run build成功
本番ビルド成功を成功条件にします。テスト環境では動くがリリースできないという落とし穴を、Goal側で塞ぐ役割です。
主要3画面のE2E成功
ユーザーが触る代表3画面のE2Eを通すことを必須にします。ユニットだけ緑で実画面が壊れる事故を、自律実行の合格条件に組み込めます。
Critical/High Issue 0
重大・高優先度のLint/解析指摘ゼロを条件にします。テストは通っても危険コードが残るパターンを、Goal側で排除できます。
/goal 次をすべて満たすまで修正。ただし最大15ターン、30分。
- npm test成功
- npm run typecheck成功
- npm run build成功
- 主要3画面のE2E成功
- Critical/High Issue 0
未達時は `goal-report.md` に残課題と最後のErrorを保存。

20.3 Headless

なぜここを読むか 対話画面を持たないHeadless実行は承認ダイアログがないので、許可範囲を絞らないと副作用付き操作も通ってしまいます。

やる／やらないとどうなるか Read-onlyとAllowed Toolを絞ったうえで、ファイル化された機械可読JSONを得るパターンが安全に組み込めます。

これは何か Headless実行でREADMEを読みJSONで返すコマンド例で、--output-format jsonにより機械処理しやすい形を要求します。

実行するとどうなるか 対話なしで構造化結果が得られ、後段スクリプトがそのまま読める前提でCI・夜間バッチへ組み込めます。

claude -p "READMEを読み、起動Commandと前提をJSONで返す" \
  --output-format json > /tmp/project-summary.json

Flagはclaude --helpで確認する。Headlessは対話承認できないためRead-only、Allowed Tool、Sandboxを絞る。

20.4 安全なWrapper

なぜここを読むか 自律実行を本番運用に乗せるには、二重起動・途中失敗・ログ欠落といったありがちな事故への防御層が必要です。

やる／やらないとどうなるか Lockディレクトリ・trapによる解放・tmpファイル経由のatomic差し替えを備えたWrapperで、運用に耐える形に格上げできます。

#!/usr/bin/env bash
set -euo pipefail
ROOT="$(cd "$(dirname "$0")/.." && pwd)"
OUT="$ROOT/output/daily-summary.json"
LOCK="$ROOT/.locks/daily-summary.lock"

mkdir -p "$ROOT/output" "$ROOT/.locks"
if ! mkdir "$LOCK" 2>/dev/null; then
  echo "already running" >&2
  exit 0
fi
trap 'rmdir "$LOCK" 2>/dev/null || true' EXIT

cd "$ROOT"
claude -p "Read-onlyで昨日の変更を要約しJSONで返す。File変更禁止。" \
  --output-format json > "$OUT.tmp"
mv "$OUT.tmp" "$OUT"

20.5 Scheduled必須要件

なぜここを読むか 定期実行は『動いている／止まっている』が見えにくいため、最低要件を明文チェックリスト化しないと劣化に気付けません。

やる／やらないとどうなるか 冪等・Lock・Timeout・JST/UTC明記・Alert先などの項目を埋めたかが運用判定の根拠として残ります。

[ ] 冪等
[ ] Lock
[ ] Timeout
[ ] 有限Retry
[ ] 前回成功を壊さない
[ ] 0件を扱う
[ ] JST/UTC明記
[ ] Working Directory固定
[ ] stdout/stderr保存
[ ] Alert先

20.6 LaunchAgent例

なぜここを読むか macOSで定期実行を組むときの設定は地味に書き方依存で、PATHやWorkingDirectoryの抜けが事故源になります。

やる／やらないとどうなるか ProgramArguments・StartCalendarInterval・StdOut/Errの最小セットを揃えたplist例が、そのまま雛形として使えます。

<?xml version="1.0" encoding="UTF-8"?>
<!DOCTYPE plist PUBLIC "-//Apple//DTD PLIST 1.0//EN"
  "http://www.apple.com/DTDs/PropertyList-1.0.dtd">
<plist version="1.0">
<dict>
  <key>Label</key><string>com.example.daily-claude-summary</string>
  <key>ProgramArguments</key>
  <array>
    <string>/bin/zsh</string>
    <string>/Users/REPLACE_ME/project/scripts/daily-summary.sh</string>
  </array>
  <key>StartCalendarInterval</key>
  <dict><key>Hour</key><integer>8</integer><key>Minute</key><integer>0</integer></dict>
  <key>WorkingDirectory</key><string>/Users/REPLACE_ME/project</string>
  <key>StandardOutPath</key><string>/Users/REPLACE_ME/Library/Logs/daily-summary.log</string>
  <key>StandardErrorPath</key><string>/Users/REPLACE_ME/Library/Logs/daily-summary.error.log</string>
</dict>
</plist>

会社では個人Agentを乱立させずHarnessへ集約する。

20.7 自動化しないもの

なぜここを読むか 公開・送信・課金・削除のような戻せない操作まで自動化すると、得られる効率より事故損失のほうが大きくなります。

やる／やらないとどうなるか 最後の一歩を人間承認に残す境界線が明文化され、それ以外を堂々と自動化できる構造に切り替わります。

人間承認なしで行わない：本番Deploy、外部送信、Payment、契約、Data削除、権限変更、公開範囲変更、高Risk判断。

こんな場面で使う

章の内容が実務でどう活きるか。架空の場面・行動・効果の三点で示します。

場面

ある社内開発者（架空）が、AIに『テストが通るまで直して』と頼み、無限ループで朝までCPUが回り続けて停止条件もなく止まれない。

こう使う

成功条件・最大ターン・Timeout・許可範囲・失敗時停止を/goal仕様として先に書き出す。

得られるもの

夜間自動実行が安全に走り、達成 or 安全停止の二状態で終わる。

場面

情シス（架空）が、毎日のレポート自動配信を構築したいが、誤送信時の責任の所在が決まらず承認が出ない。

こう使う

公開・送信・課金・削除は完全自動化せず、最後の一歩を人間承認に残す設計にする。

得られるもの

副作用のある操作だけ責任者が承認し、それ以外は自動で進む構造になり、承認が下りる。
