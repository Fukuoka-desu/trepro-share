第13章

一人で同時に、3つの調査を走らせる Subagents、Agent View、並列作業

『調査・実装・レビューを一人で順番にやる』。 これでは、いくらAIを使っても作業のシリアル化が解けません。

ほとんどの人は、AIを『一対一の相棒』として使っています。 ですが、別々の目的を別々のサブエージェントに分担させると、3つの調査が同時に進みます。

Subagents と Agent View は、独立した文脈で並列に作業させる仕掛けです。 親文脈を汚さずに、調査担当・実装担当・レビュー担当を別々に走らせ、結果だけを束ねます。

この章を読み終わる頃には、『1人で同時に3つ』を当たり前にする発想が手に入ります。

この章の概観 ／ 何を扱い、何ができるか

この章で扱う機能と、それで何ができるようになるか。実装に入る前に、全体像をここでつかみます。

  • Subagent（文脈を分ける仕組み）

    役割ごとに別Agentを立てて文脈を分離し、過去会話に引きずられない調査・実装・レビューを可能にする機構。

    入力
    親セッションの作業、役割ごとの責務
    出力
    Subagentの完了報告、独立した中間成果物

  • Reviewer Agent（独立コンテキストの評価者）

    成果物だけを渡して評価させ、迎合バイアスを排した指摘を得るための専任エージェント。

    入力
    成果物、評価観点（要件/安全/保守性）
    出力
    独立コンテキストからの指摘リストと優先度

  • Research Agent

    調査専任で出典・確度・反証を分けて整理するエージェント。実装者と分離して情報収集の質を確保する。

    入力
    調査テーマ、検索範囲、評価軸
    出力
    出典付き調査結果と未確認項目リスト

  • Agent View と並列実行

    複数Agentの動きを俯瞰し、独立した仕事へ分担させる運用画面。同じファイルへの同時編集は避ける指針付き。

    入力
    並列化したいタスク群、各Agent責務
    出力
    Agent View上の並列実行ログと統合結果

実装リファレンス

13.1 Subagentを使う理由

なぜここを読むか 1つのSessionで全部やると、過去会話の文脈・権限・バイアスが混ざります。Subagentを使う理由を3点で押さえておかないと、ただの並列化に終わります。

やる／やらないとどうなるか Context分離・Tool/権限限定・作成者と評価者分離の3軸を意識すれば、Subagentは単なる人数増ではなく品質装置になります。理由を理解せず並列化すると、同じBlind Spotを共有したまま動きます。

  1. Contextを分離する。
  2. Toolと権限を限定する。
  3. 作成者と評価者を分ける。

13.2 Reviewer Agent

なぜここを読むか 実装したAgentに自分のコードをレビューさせると、迎合バイアスで甘い採点になります。Reviewer Agentを別定義し、編集権限も渡さない構成が必要です。

やる／やらないとどうなるか Read/Grep/Glob/Bashだけ持つ独立Reviewerを置けば、要件・安全・データ消失・テスト欠落・保守性の観点で本物の指摘が返ります。Write権限を渡すと「直しておきました」で原因が分からなくなります。

.claude/agents/code-reviewer.md：

これは何か Reviewer Agentの定義例です。toolsをRead/Grep/Glob/Bashに限定し、Write権限を渡さない構成にしています。

実行するとどうなるか この定義で起動すれば、編集できない独立Reviewerが要件・安全・データ消失・テスト・保守性の観点で指摘を返します。BlockerとSuggestionの分離も強制されます。

---
name: code-reviewer
description: Independently review changes for correctness, security, data loss, missing tests, and maintainability.
tools: Read, Grep, Glob, Bash
model: sonnet
---

You are an independent reviewer. Do not edit files.
Read requirements, diff, and test evidence.
Prioritize security, data loss, incorrect behavior, edge cases, missing tests,
and operational risk. For each finding include severity, file, symbol or line,
evidence, and minimal remediation. Separate blockers from suggestions.

Review AgentにWrite Toolを渡さない。

13.3 Research Agent

なぜここを読むか 実装者が調査も兼ねると、初期仮説に都合のよい情報だけ集めてしまいます。一次情報・出典・確度を切り離して扱うResearch Agentが必要です。

やる／やらないとどうなるか 公式Doc・原典論文・First-party発表を優先し、公開日・取得日・支持される主張・不確実性を記録する設計にすれば、調査の質が説明可能になります。最終Reportは作らせない分業も重要です。

これは何か Research Agentの定義例です。一次情報優先・公開日や取得日の記録・最終Report作成禁止を含む調査専任設定です。

実行するとどうなるか この役割で動かすと、出典付きの主張と不確実性が指定ファイルに保存されます。実装者と分離されるため、仮説に引きずられない調査結果が得られます。

---
name: primary-source-researcher
description: Collect primary-source evidence for a narrowly defined research question and save structured claims.
tools: WebSearch, WebFetch, Read, Write
model: sonnet
---

Research only the assigned subquestion.
Prefer official documentation, original papers, and first-party announcements.
Record publication date, event date, retrieval date, exact supported claim,
and uncertainty. Save findings to the requested file. Do not synthesize the final report.

13.4 作成者と評価者

なぜここを読むか 作成者と評価者を同一Agentにすると、同じPromptと同じContextを共有するため、同じBlind Spotを再生産します。役割と入力の分離が肝心です。

やる／やらないとどうなるか Main・Research・Implementer・Fresh Reviewerと役割を切り分ければ、人間承認を挟んだ健全なフィードバックループが組めます。全員に同じ前提を渡すと並列化しても品質は上がりません。

これは何か 作成者と評価者の役割分担表です。Main・Research・Implementer・Fresh Reviewerの責務を縦に並べたフロー記述になります。

実行するとどうなるか この役割マップに沿って委任すれば、人間承認後修正までを含む健全なReview Loopが組めます。役割が曖昧だと全員が同じ視点で動く事故が起きます。

Main Agent: 要件と統合
Research Agents: 独立範囲を調査
Implementer: 変更
Fresh Reviewer: 欠陥探索
Main Agent: 人間承認後に修正

同じPromptとContextを全Agentへ配ると、同じBlind Spotを共有する。役割と入力を分ける。

13.5 Agent View

なぜここを読むか Agent Viewで複数Sessionを管理できるからといって、同一Fileを複数Agentへ同時編集させると衝突します。割当の安全パターンを持っておく必要があります。

やる／やらないとどうなるか Read-only調査・tests/のみ・src/のみ・docs/のみ・Read-only Reviewのように担当領域を物理的に分ければ、Agent間の編集競合が起きません。曖昧な割当は破壊的なマージ事故を招きます。

現行Versionでclaude agentsを確認する。複数Sessionを管理できるが、同一Fileの同時編集は避ける。

安全な割当：

これは何か Agent Viewでの安全な担当割当パターンです。読み取り専用と書き込み領域を物理的に分けた構成を示します。

実行するとどうなるか この割当を採用すれば、複数Agent並列でもFile編集衝突が起きません。同じファイルを複数Agentへ同時編集させる失敗を構造的に防げます。

A: Read-only調査
B: tests/のみ
C: src/のみ
D: docs/のみ
E: Read-only Review

13.6 並列化に向く仕事

なぜここを読むか Subagent並列化は万能ではありません。向く仕事と向かない仕事を見分けないと、依存関係のあるTaskを並列化して整合性を失います。

やる／やらないとどうなるか 独立情報源の調査・非依存テスト追加・複数Option Prototype・別観点Reviewは向きます。同一File編集・仕様未確定・外部Write・小Taskは向きません。線引きを持てば並列化の効果が安定して出ます。

  • 独立した情報源の調査。
  • 互いに依存しないTest追加。
  • 複数OptionのPrototype。
  • Code、Security、UXの別Review。

向かない：

  • 同じFileを頻繁に編集。
  • 仕様が固まっていない。
  • 外部Writeを伴う。
  • 1人で数分の小Task。

13.7 委任指示

なぜここを読むか Subagentへ「いい感じに調べて」と頼むと、範囲が広がり成果物の所在も曖昧になります。委任指示には決まった項目セットが必要です。

やる／やらないとどうなるか Role・Scope・Allowed tools・Inputs・Deliverable path・Done conditions・Prohibited actions・Budget/timeoutの8項目を埋めれば、誤動作と暴走を防げます。指示が緩いとAgentは余計なことを始めます。

これは何か Subagent委任指示の必須項目チェックリストです。Role・Scope・Allowed tools・Inputs・Deliverable path・Done conditions・Prohibited actions・Budget/timeoutの8点を並べます。

実行するとどうなるか この項目を埋めて委任すれば、Subagentが範囲外を始めるリスクと終わらないリスクを同時に抑えられます。空欄が残るほど暴走確率が上がります。

公式Documentだけを調べるResearch Agentとして動いてください。

こんな場面で使う

章の内容が実務でどう活きるか。架空の場面・行動・効果の三点で示します。

場面

ある個人開発者（架空）が、自分のコードを自分でレビューさせるとAIが甘く採点し、毎回バグを見落としている。

こう使う

実装セッションを閉じ、背景を知らない独立コンテキストのReviewer Agentへ成果物だけ渡す。

得られるもの

迎合バイアスが消え、要件・安全・保守性の三観点で本当の指摘が返ってくる。

場面

中堅企業のリサーチチーム（架空）が、調査・検証・統合を一人のAIに任せ、結論が初期仮説に引きずられる傾向が抜けない。

こう使う

Research・Verify・Synthesizeを別Agentに分け、各役割の入力と期待出力を明示する。

得られるもの

仮説バイアスが分離され、反証も合わせて統合できる。結論の信頼度が説明可能になる。
