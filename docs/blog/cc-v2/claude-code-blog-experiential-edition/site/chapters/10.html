第10章

間違えても、1分前に戻れる仕事場 GitとCheckpoint

『AIに修正を任せたら、なぜか動かなくなった』。 原因を辿りたいのに、どこから何が変わったか分からない。これは記憶の問題ではなく、保存設計の問題です。

ほとんどの人は、AIの出力をそのままファイルへ上書きしています。 便利ですが、壊れた瞬間に戻る場所を失います。

この章で扱うのは、Gitと Checkpoint を使った『戻れる仕事場』の作り方です。 意味のある単位で保存し、壊れたら 1 分前に戻り、誰が何のために変えたかを履歴に残す。 それだけで、AI と作業する不安の大半は消えます。

不可逆な操作ほど、可逆な仕掛けで包む。Git は AI 時代の安全網です。

この章の概観 ／ 何を扱い、何ができるか

この章で扱う機能と、それで何ができるようになるか。実装に入る前に、全体像をここでつかみます。

  • Checkpoint（会話中の素早い巻き戻し）

    セッション内で直前の変更を一手で戻せる機能。試行錯誤の安心担保として軽量な保存地点を作る。

    入力
    AIに任せる大胆な変更、実験的な編集
    出力
    巻き戻し可能なCheckpoint、戻したあとの状態

  • 最小Git運用

    意味のある単位でCommitし、変更理由と検証結果を残すことで、セッションや人をまたぐ正式な履歴を作る。

    入力
    完了した小さな変更、検証結果
    出力
    意味あるCommitメッセージとレビュー可能なdiff

  • ClaudeへCommitさせるRule

    AIがCommitメッセージを書く際の体裁・粒度・禁止事項を定めるルール。履歴の読みやすさを保つ。

    入力
    完了した変更、検証ログ
    出力
    Rule準拠のCommit、PR説明文

  • Worktreeによる並列作業

    別ブランチを別ディレクトリとして展開し、同じファイルへの編集衝突なしに並列実装を進める仕組み。

    入力
    並行して進めたい複数機能
    出力
    機能別Worktreeディレクトリと衝突なしの差分

実装リファレンス

10.1 両方必要

なぜここを読むか CheckpointとGitはどちらか一方だと穴が残ります。守備範囲の違いを理解し、両方を並列で持つ必要があります。

やる／やらないとどうなるか Checkpointは会話中の素早い巻戻し、GitはSessionや人を跨ぐ正式履歴と役割を分けると、短時間と長期間の両方で『戻れる』状態が作れます。

仕組み 強み 限界
Checkpoint /rewind Session内の素早い巻戻し 外部副作用、手動変更、長期履歴は限定
Git Sessionを跨ぐ履歴、Review、Branch Commit前の細かい対話状態は持たない

10.2 Checkpoint

なぜここを読むか Checkpointは便利ですが、外部副作用や手動変更まで戻せるわけではありません。範囲を理解せずGit代わりにすると壊れた状態で詰みます。

やる／やらないとどうなるか /rewindで戻すのはCodeかConversationかを毎回確認することで、Checkpointを正しく軽量な安心装置として使えます。

これは何か Checkpointへ巻き戻すコマンド。Esc2回でも履歴選択に入れる場合があり、CodeかConversationかを選んで戻します。

実行するとどうなるか セッション内の試行錯誤を一手で戻せるようになり、AIへ大胆な変更を任せる心理的ハードルが下がります。

/rewind

またはEscを2回押して履歴選択へ入れる場合がある。CodeとConversationのどちらを戻すか確認する。

CheckpointはGitの代わりではない。

10.3 最小Git運用

なぜここを読むか AIへ大胆な変更を任せる前後で良い状態を保存していないと、壊れた時に戻る場所がありません。最小限のGit操作は身に付けておく必要があります。

やる／やらないとどうなるか init→add→baseline commit→検証後の意図あるcommit、を癖にすると、AI作業の前後に必ず復帰可能点が残ります。

これは何か 最小Git運用の典型フロー。Baseline Commitを作り、変更後にStatusとDiffを確認してから意図あるCommitを刻みます。

実行するとどうなるか AI作業の前後に復帰可能点が必ず残り、壊れたら直前のCommitに戻すという退避路が常時開通します。

git init
git add .
git commit -m 'chore: baseline before agent changes'

git status --short
git diff
# 検証後
git add <intentional-files>
git commit -m 'feat: add meeting task extraction'

10.4 ClaudeへCommitさせるRule

なぜここを読むか AIにCommitを任せると、メッセージが冗長になったり粒度がばらついたりして、後日のレビューや巻き戻しが難しくなります。書き方のRuleを与える必要があります。

やる／やらないとどうなるか Diff提示・目的を1つ・Secrets Scan・事実だけ書く、などをRule化すると、人と同じ品質の履歴がAI作業でも維持できます。

これは何か AIにCommitを任せる際のRule7か条。Diff提示・目的1つ・Secrets Scan・Force Push禁止・事実記述などを縛ります。

実行するとどうなるか AIが書くCommitでも履歴が読みやすく保たれ、後日のレビューと巻き戻しが現実的な作業として成立します。

それぞれの意味

Commit前にDiffを見せる
実行前に変更全体を人へ提示する縛りです。提示なしのCommitは意図と実体のズレに気付けません。
変更目的を1つにする
1Commitに1つの意図しか含めない縛りです。混ぜると後日の巻き戻しと原因切り分けが難しくなります。
Secrets Scan
鍵やTokenが含まれていないか必ず走らせる縛りです。AIに任せると気付かず混入し、履歴に永続化します。
Test Evidence
検証実行Logや結果を添える縛りです。事実の裏付けなしのCommitは『動いたっぽい』を履歴化してしまいます。
他人の変更を含めない
他者の修正やマージ差分を巻き込まない縛りです。混入すると履歴の責任所在が曖昧になります。
Force Pushしない
履歴を上書きするpushを禁じる縛りです。共有Branchで実行すると他人のCommitを消し飛ばします。
Commit Messageへ事実だけを書く
推測・誇張・自賛を排し、変更内容と理由のみ書く縛りです。後日の読み手が事実だけで判断できるようにします。
- Commit前にDiffを見せる
- 変更目的を1つにする
- Secrets Scan
- Test Evidence
- 他人の変更を含めない
- Force Pushしない
- Commit Messageへ事実だけを書く

10.5 Worktree

なぜここを読むか Agentを並列に動かす時に同じWorking Treeを共有すると、編集が衝突して上書きし合います。物理的に作業場所を分ける仕組みが要ります。

やる／やらないとどうなるか worktree addで機能ごとに別ディレクトリへ展開すると、並列実装が衝突なしで進み、後で人間が安全に統合できます。

Agentを並列化する時は、同じWorking Treeを共有しない。

これは何か Worktreeで作業ツリーを物理分離するコマンド例。テスト用とドキュメント用に別ディレクトリ・別Branchを切ります。

実行するとどうなるか 複数Agentに並列実装させても同じFileを奪い合わず、人間が後で整合させてマージできる構造になります。

それぞれの意味

../project-agent-tests (agent/tests)
テスト追加担当Agent専用の作業ツリーです。本体ディレクトリと物理分離するため、同じFileの奪い合いが起きません。
../project-agent-docs (agent/docs)
ドキュメント整備担当Agent専用の作業ツリーです。Branchも別なので、人間が後でレビューしながら安全に統合できます。
git worktree add ../project-agent-tests -b agent/tests
git worktree add ../project-agent-docs -b agent/docs

各Agentへ編集範囲を割り当て、人間が統合する。

こんな場面で使う

章の内容が実務でどう活きるか。架空の場面・行動・効果の三点で示します。

場面

中小Webサイト運営者（架空）が、AIに『大胆に作り直して』を依頼するたび、戻したい時に戻せず古い版を失っている。

こう使う

毎回の作業開始前に良い状態をCommitし、Checkpoint＋Gitの二重保存で『戻れる』を物理的に作る。

得られるもの

AIへ強気に試行させても、いつでも前に戻れる安心が手に入る。

場面

ある個人開発（架空）が、複数機能を同時に試したくて同じファイルを並列編集し、お互いの変更を上書きして混乱した。

こう使う

Worktreeで作業ツリーを物理的に分け、機能ごとに別ディレクトリで並列に動かす。

得られるもの

AIに並列実装させても衝突が起きず、後でCherry-pickで良い変更だけを統合できる。
