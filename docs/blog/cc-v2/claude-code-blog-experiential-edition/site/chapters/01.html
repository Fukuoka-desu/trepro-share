第1章

チャットで止まる人と、走り出す人の差 Claude Codeは何が違うのか

『この資料、まとめて』とAIに頼んで、文章だけ返ってきて止まる。 これ、AIの限界じゃありません。使ってる入口を間違えてるだけです。

ほとんどの人は、AIを『チャット画面で会話する相手』だと思ってます。 でも実は、目的を渡すとフォルダを読み・ファイルを作り・コマンドを実行し・結果を確かめるところまで、自分で走る入口（Claude Code）があります。

この章では、相談・業務処理・実装・情報設計という4つの入口を整理して、いまの仕事をどこで始めてどこへ渡すかを決めます。 読み終わる頃には、AIに渡す『仕事の単位』が10倍くっきり見えるようになります。

実装リファレンス

1.1 チャットAIからエージェントへ

なぜここを読むか チャットAIはコードと文章を渡してくるだけで、実行と検証は人間に丸投げ。同じ作業を何度も貼り付けるロスがある

やる／やらないとどうなるか Claude Codeは探索・編集・実行・検証を同じ場所で回す。人間はタイピング作業から目的・権限・品質の設計側へ仕事が移る

従来のチャットAIでは、人間がコードをコピーし、実行し、エラーを再度貼り付けていた。Claude Codeは同じ作業場所で次を行う。

  1. Projectを探索する。
  2. 関係ファイルを読む。
  3. 方針を立てる。
  4. ファイルを編集する。
  5. コマンドを実行する。
  6. エラー、Test、Diffを読む。
  7. 修正する。
  8. 完了条件まで繰り返す。

したがって、強い依頼は「コードを書いて」ではなく、目的、文脈、制約、成果物、完了条件、証拠を渡す。

1.2 AIコーディングの3世代

なぜここを読むか Copy時代・Editor時代・Agent時代では、人間に求められる仕事が違う。今いる世代を勘違いするとAIの使い方を間違える

やる／やらないとどうなるか 自分が第2世代の使い方で止まっていることに気づける。次に投資すべきは入力速度ではなく目的設計だと分かる

世代 形 人間 AI
第1世代 Chat Copy、実行、Error転記 断片Code生成
第2世代 AI Editor 実行、統合、判断 補完、局所編集
第3世代 Agent 目的、権限、評価 探索、編集、実行、修正、検証

人間が不要になるのではない。人間の仕事が、入力作業から目的設計、権限設計、品質判定へ移る。

1.3 Chat、Cowork、Code、Design

なぜここを読むか 全部の仕事を1つの入口に押し込むと、相談まで重くなり実装は浅くなる。仕事ごとに最適な入口を分ける必要がある

やる／やらないとどうなるか Chatで要件整理→Coworkで素材処理→Designで情報設計→Codeで実装、と渡せる。1つの仕事が無理なく前へ進む

Surface 得意な仕事 選ぶ目安
Chat 壁打ち、文章、要件整理、単発分析 ファイルを直接変えない相談
Cowork 議事録、週報、整理、定期業務、Artifacts 非開発の複数ステップ業務
Code 実装、Terminal、Test、Git、Harness Code/Config/CLIを扱う
Design Wireframe、Prototype、Design System 情報構造と見た目を先に固める

典型フロー：

これは何か Chat→Cowork→Design→Codeの典型ワークフロー。仕事をどの入口から渡し、どこで実装に着地させるかの順序

実行するとどうなるか 案件をどこから始めてどこへ渡すかが固定される。入口を間違えて手戻る事故が減り、成果物の質が安定する

それぞれの意味

Chatで要件化
ファイルを触らず壁打ちで要件と論点を固める段階。ここを飛ばすと曖昧な依頼のまま実装に入って手戻りが起きる
Coworkで素材処理
議事録・週報・整理など非開発の複数ステップ業務を回す段階。Artifactsで成果物を都度確認しながら進める
Designで情報設計
情報構造と見た目を先に固める段階。実装に入る前にWireframe/Prototypeで合意を取り、後の戻し作業を防ぐ
Codeで実装・検証・公開準備
Terminal・Test・Git・Harnessを使う実装着地点。前段で定義が固まっているから実装が一発で通る
Chatで要件化
  ↓
Coworkで素材処理
  ↓
Designで情報設計
  ↓
Codeで実装・検証・公開準備

1.4 「育てる」の正しい意味

なぜここを読むか 成功した会話を会話のまま残しても、翌日には同じ説明が必要。これを『育てた』とは呼べない

やる／やらないとどうなるか 知見をCLAUDE.md・Rule・Skill・Hook・MCPへ昇格させる。個人の工夫が再呼び出し可能な資産に変わり、属人化が消える

会話を長く続けることではない。成功した知見を適切なレイヤーへ外部化する。

知見 保存先
Project概要・常設方針 CLAUDE.md
Path別のRule .claude/rules/
繰り返す業務手順 Skill
独立した専門家 Subagent
必ず実行・拒否する制御 Hook / Permissions
外部System能力 MCP / Connector
Claudeが発見した短い知識 Auto Memory
全社員共通資産 Git管理Harness / Plugin
