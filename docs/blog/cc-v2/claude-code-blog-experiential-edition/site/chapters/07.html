うっかり rm -rf を、5秒手前で止める Permission Mode、Sandbox、危険操作

『うっかり rm -rf』『git push で main を上書き』。 AIに作業を任せたとき、こうした取り返しのつかない事故は、油断した数秒に起きます。

ほとんどの人は、AIに『全部許可』のまま動かしています。 便利だから、確認が面倒だから。 ですが、その『便利』と『事故』は同じスイッチで繋がっています。

Permission Mode は、操作ごとに『許可・拒否・確認』を切り替える仕組みです。 読み取りは自動で許可、書き換えは確認、削除と外部送信は必ず止める。それだけで、暴走の大半が手前で止まります。

この章を読み終わる頃には、自分が安心して任せられる範囲を、自分で決められるようになります。

この章の概観 ／ 何を扱い、何ができるか

この章で扱う機能と、それで何ができるようになるか。実装に入る前に、全体像をここでつかみます。

  • Permission Mode と Permission Rule

    読む・編集する・実行する・ネットワーク・削除の5分類で許可と確認を切り分け、責任の境界を明示する権限設計。

    入力
    扱うProject、許可してよい操作と確認したい操作
    出力
    Project単位の許可ポリシー表とPermission Rule定義

  • Sandbox と Bypass運用

    Bypassは隔離された使い捨て環境のみ、本番Projectは確認モード固定とする運用ルール。便利さと事故半径を分離する。

    入力
    実験用ワークスペース、本番Projectの一覧
    出力
    Bypass許可Pathと禁止Pathを明示したCLAUDE.mdルール

  • Secrets管理ルール

    APIキー・トークンをチャットに貼らず、環境変数や専用ファイルへ置きGitから除外する3点ルール。

    入力
    使う外部サービス、必要な鍵・トークン
    出力
    Secret保存場所、.gitignore更新、CLAUDE.md記載

  • 削除の標準手順

    対象一覧→件数確認→Dry-run→バックアップ→実行→結果確認の6段で、削除事故をワンクリックで起こさない手順化。

    入力
    整理したいフォルダ、削除候補ファイル
    出力
    削除Dry-run結果、バックアップ、削除完了レポート

実装リファレンス

7.1 Permission Mode

なぜここを読むか 『全部許可』と『全部禁止』しかない運用では、便利さと事故半径が同じスイッチで繋がってしまいます。利用シーンごとに振る舞いを変える軸が必要です。

やる／やらないとどうなるか default・plan・bypassPermissionsなどModeを使い分けると、未知Projectは慎重、隔離環境は素早く、と状況に応じた設計ができます。

Mode 役割 推奨用途
default Toolごとに通常の確認 初心者、未知Project
acceptEdits File編集を円滑化 Local開発
plan 読む・計画する 要件化、高Risk変更
auto Classifierが行動を判定 対応環境で慣れた利用者
dontAsk 確認できない操作を拒否 Headless/限定Workflow
bypassPermissions 確認をほぼ飛ばす 使い捨て隔離環境のみ

Mode名と提供範囲はVersionで変わり得る。/permissionsと現行公式情報を確認する。

7.2 Permission Rule

なぜここを読むか Modeだけでは個別操作の細やかな許可を表現できず、危険コマンドが紛れ込みます。コマンドPattern単位の許可表を別に設ける必要があります。

やる／やらないとどうなるか deny→ask→allowの順で評価される設定にすると、機密Fileの読み出しや破壊コマンドが入口で弾かれ、安全な操作だけ自動許可できます。

Ruleの評価は、基本的に deny → ask → allow の順である。広いdenyに狭いallowを重ねても例外にはならない。

これは何か Permission RuleのJSON設定例で、.env読み取り禁止・rm -rf禁止・git push確認・testコマンド自動許可を組み合わせています。

実行するとどうなるか deny→ask→allowの順で評価されるため、機密Fileや破壊コマンドが入口で止まり、安全コマンドだけ自動進行できます。

それぞれの意味

deny
実行を必ず拒否する区分です。.envや破壊コマンドなど、入口で物理的に止めたい操作をここへ並べます。
ask
実行前に人へ確認を求める区分です。git pushやdeployなど、自動で進めると取り返しがつかない操作に使います。
allow
確認なしで自動許可する区分です。git statusやtest実行など、安全で頻度の高い操作をここに置いて作業を速めます。
{
  "permissions": {
    "deny": [
      "Read(./.env)",
      "Read(./.env.*)",
      "Read(./secrets/**)",
      "Bash(rm -rf *)"
    ],
    "ask": [
      "Bash(git push *)",
      "Bash(gcloud run deploy *)"
    ],
    "allow": [
      "Bash(git status)",
      "Bash(git diff *)",
      "Bash(npm run test *)"
    ]
  }
}

Bash文字列Patternだけで完全なSecurity境界を作れない。HookとSandboxを併用する。

7.3 Bypassを使ってよい条件

なぜここを読むか Bypassは便利さの裏で確認を飛ばすため、本番Projectで使うと取り返しがつきません。利用してよい条件をはっきり線引きする必要があります。

やる／やらないとどうなるか 使い捨て環境・本番Credentialなし・Network制限など全条件を満たした時だけ許可することで、便利さと事故半径を物理的に切り離せます。

すべて必要：

これは何か Bypassを使ってよい条件のチェックリスト。隔離環境・本番Credentialなし・Network制限・Timeout等の必須条件を列挙しています。

実行するとどうなるか 全項目を満たさない環境ではBypassを使わない判断ができ、便利さと事故半径のトレードオフが明示的になります。

それぞれの意味

使い捨てContainer/VM/Codespace
壊れても捨てれば終わる隔離環境であることを担保します。本物のMacで使うと事故半径が無制限に広がります。
本番Credentialなし
本番APIキーやDB認証がその環境に存在しないことを担保します。あると暴走時に本番データへ即到達してしまいます。
Hostの重要DirectoryをMountしない
~/Documents等のホスト資産を露出させない条件です。Mountするとサンドボックスの意味が失われます。
Networkを制限
外部送信先を絞っておく条件です。無制限だと機密の外部送信や踏み台化を止められません。
Gitで復元可能
破壊操作の後でもCommitから戻せる状態を担保します。Git管理外だと壊れた瞬間に詰みます。
Timeout/Kill Switchあり
暴走を物理的に止める手段を用意します。なければAIが延々と料金や副作用を積み上げる事故を止められません。
Side Effectなし
外部送信・課金・通知が発生しない構成であることを担保します。あると『試しただけ』が本番事故に直結します。
[ ] 使い捨てContainer/VM/Codespace
[ ] 本番Credentialなし
[ ] Hostの重要DirectoryをMountしない
[ ] Networkを制限
[ ] Gitで復元可能
[ ] Timeout/Kill Switchあり
[ ] Side Effectなし

全社MacではManaged Settingsで無効化する。

7.4 Sandbox

なぜここを読むか Permission Ruleは『実行してよいか』までしか守れず、実行後にどこへ触れるかは別問題です。実行先の境界を物理的に縛る仕組みが要ります。

やる／やらないとどうなるか Sandboxを有効化し、~/.ssh・~/.aws・~/.kubeなどを読み取り禁止にしておくと、許可された操作の影響範囲が機密領域へ伸びなくなります。

Session内で：

/sandbox

厳格なManaged例：

これは何か Managed設定のSandbox例で、有効化必須・未sandbox実行禁止・~/.ssh等の読み取りDenyを束ねた厳格構成です。

実行するとどうなるか Permissionで許可された操作でも、機密領域への到達が物理的に止まり、実行後の影響範囲を切れます。

それぞれの意味

enabled
Sandbox自体を有効化するスイッチです。trueでない限り他の設定は全て無効で、物理境界はかかりません。
failIfUnavailable
Sandboxが使えない環境では起動自体を失敗させる設定です。trueにすると『なし崩しに素通り実行』を防げます。
autoAllowBashIfSandboxed
Sandboxed状態ならBashを自動許可するかの設定です。falseのまま運用すると、隔離されていても確認を残せます。
allowUnsandboxedCommands
Sandbox外でのコマンド実行を許すかの設定です。falseにしておくと未隔離操作の抜け道を塞げます。
filesystem.denyRead
~/.ssh・~/.aws・~/.kubeなど読み取りすら禁じるPathを列挙します。許可済み操作でも機密へ手が伸びるのを止めます。
{
  "sandbox": {
    "enabled": true,
    "failIfUnavailable": true,
    "autoAllowBashIfSandboxed": false,
    "allowUnsandboxedCommands": false,
    "filesystem": {
      "denyRead": ["~/.ssh", "~/.aws", "~/.kube"]
    }
  }
}

SandboxはPermission Modeとは別物である。Permissionは「実行してよいか」、Sandboxは「実行後に何へ触れられるか」を制御する。

7.5 Secrets

なぜここを読むか APIキーやTokenはチャットや.envに無頓着に置くと、Log・Test Fixture・履歴経由で外へ漏れます。Secretsだけは置き場所と扱いを別建てにする必要があります。

やる／やらないとどうなるか Chat禁止・.env非読み取り・Keychain/Secret Manager利用・Logへ出さない、を徹底することで、漏洩経路を最初から塞げます。

これは何か Secrets管理6原則のリスト。Chat禁止・.env非読み取り・Keychain利用・Log出力禁止・例示は名前だけ・Scope最小と期限。

実行するとどうなるか 鍵やTokenの漏洩経路が網羅的に塞がれ、組織ポリシーとしてそのまま採用できる粒度になります。

それぞれの意味

Chatへ貼らない
APIキーやTokenを会話本文に貼らない原則です。一度貼ると履歴・Log・要約経由で外へ漏れ続けます。
.envを読ませない
Permission Ruleで.env読み取りを禁じる原則です。実装の便宜で読ませると、出力やテストへ平文流出します。
Keychain/Secret Manager/CI Secretを使う
鍵は専用Storeで管理する原則です。File直書きは履歴・バックアップ経由で永続的に残り続けます。
Logへ出さない
実行Log・例外Trace・Debug出力に鍵を含めない原則です。Log集約基盤経由で広く共有される事故を防ぎます。
.env.exampleには名前だけ
サンプル設定には変数名のみ書き、値は空にする原則です。実値混入のままGitへ入る事故を未然に防ぎます。
Scope最小、期限付き、Rotation可能
権限は必要最小限・期限切れ・差し替え可能で発行する原則です。漏洩時の被害範囲と回復時間を小さく保てます。
- Chatへ貼らない
- `.env`を読ませない
- Keychain/Secret Manager/CI Secretを使う
- Logへ出さない
- `.env.example`には名前だけ
- Scope最小、期限付き、Rotation可能

Prompt：

これは何か AIに渡すSecrets取り扱いの指示Prompt例。環境変数経由で読み、値の表示・保存・Fixture混入を禁じます。

実行するとどうなるか 実装段階でSecretsがコードや出力に残らない構造になり、後段のSecret Scanで指摘される事故を未然に防げます。

認証情報は環境変数 INTERNAL_API_TOKEN から読み、値を表示・保存・Test Fixtureへ含めない実装にしてください。

7.6 削除の標準手順

なぜここを読むか 削除はワンクリックで進むと不可逆な事故になります。実行までに人の判断を挟む段階を強制する必要があります。

やる／やらないとどうなるか List→Dry-run→Human Review→Quarantine→Retention→Delete Approvalの順を踏むと、実行前のリストで必ず止められ、誤削除が消えます。

これは何か 削除の6段標準手順。実行までにDry-run・Human Review・Quarantine・Retention・Delete Approvalを挟みます。

実行するとどうなるか ワンクリックでの削除が物理的にできなくなり、最悪でも実行前リストの段階で止められます。

それぞれの意味

List
削除候補を全件一覧化する段階です。件数と中身を可視化しないと、AIの『良しなに』判断で誤対象が混ざります。
Dry-run
実際には削除せず予定だけ表示する段階です。挟まないとワンクリックで不可逆な削除が走ります。
Human Review
人が一覧と予定差分を目視確認する段階です。AI任せでは契約PDFのような重要Fileの混入に気付けません。
Quarantine
対象を一時退避領域へ移す段階です。即削除ではなく退避を挟むことで、誤判定時の復元路を残せます。
Retention
退避後に一定期間保持する段階です。直後に発覚しない問題でも、期間内であれば取り戻せます。
Delete Approval
最終削除前に承認を取る段階です。一連の段階を踏んでも、最後の人の承認で物理削除を分離できます。
List → Dry-run → Human Review → Quarantine → Retention → Delete Approval
