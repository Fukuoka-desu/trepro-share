第8章

最上位モデルを常用して、月初に上限を吹き飛ばす ModelとEffort

『常に最上位モデルを選んでおけば安心』。 これが、月初に利用上限を吹き飛ばす最もよくあるパターンです。

ほとんどの人は、AIのモデルを『強さで選ぶ』と考えています。 ですが実際には、計画・実装・整理・評価という役割で選ぶのが正解です。簡単な整理は軽量モデル、重要なレビューだけ深く考えさせる（高Effort）。 このスイッチを身につけると、月の利用上限が来なくなる上に、難所の品質も上がります。

加えて、モデル名や料金は変わりやすいので、本文に書かず Feature Registry に分離しておきます。これで教科書本体は古びません。

設定は15分。コスト効率と品質、両方が同時に上がります。

この章の概観 ／ 何を扱い、何ができるか

この章で扱う機能と、それで何ができるようになるか。実装に入る前に、全体像をここでつかみます。

  • Model選択（役割で使い分け）

    計画・実装・軽い整理・評価という役割でモデルを選び、常に最上位を使わない使い分けの考え方。

    入力
    業務の難易度、失敗コスト、必要なスピード
    出力
    業務カテゴリ別のモデル割り当て表

  • Effort（思考量の調整）

    深く考える価値がある仕事だけEffortを上げ、定型変換は最小Effortで回すコスト最適化の設定軸。

    入力
    対象タスクの判断重要度
    出力
    タスク別Effort設定とコスト見積

  • Advisor と ultrathink

    難しい判断にだけ高Effort思考を呼び出し、相談役として併走させる使い方。常用ではなく要所投入のパターン。

    入力
    設計分岐、複雑な原因分析、重要レビュー
    出力
    Advisorからの追加観点、判断根拠の補強

  • Feature Registry（変動情報の分離）

    変動するモデル名・利用条件は本文ではなくRegistryへ外出しし、教科書原則を特定モデル依存から切り離す仕組み。

    入力
    現行モデル名・料金・利用上限
    出力
    Feature Registry更新エントリ（確認日付き）

実装リファレンス

8.1 Model名ではなく役割で選ぶ

なぜここを読むか モデルを『強さ』で常に最上位選択にすると、月初に利用上限を吹き飛ばします。仕事の役割で選び分ける軸が必要です。

やる／やらないとどうなるか 軽量・標準・高性能・長時間最上位を役割で割り当てると、日常は標準で回り、難所だけ高性能を投下できる構造になります。

役割 選び方
軽量 定型変換、分類、短い要約
標準 通常実装、文章、調査、Test
高性能 Architecture、難しいDebug、重要Review
長時間最上位 複雑な探索・Workflow。ただしCostと提供状況を確認

日常は標準Modelを使い、計画、難所、最終Reviewだけ高性能へ切り替える。

8.2 Effort

なぜここを読むか Effortを常に最大にしても、曖昧な依頼は曖昧なまま長く考えるだけです。判断の難しさに応じて深さを変える必要があります。

やる／やらないとどうなるか low/medium/high/最大を仕事ごとに使い分けると、定型変換は速く・難所は深く考えさせられ、時間とコストの両方が締まります。

Effortは、考える深さ、時間、利用量のTrade-offである。

これは何か Effortの4段階を仕事種別に対応させた目安表。定型から長時間探索まで、どこに深さを投下するかを示します。

実行するとどうなるか タスクごとのEffort割り当てが具体化し、最大Effort常用による利用上限の早期到達を避けられます。

low/medium: 定型、軽い修正
high: 複数File、分析、重要文書
最大: 難しいArchitecture、長時間探索

深く考えるほど常に良いわけではない。曖昧な依頼を長く考えても、曖昧なままである。

8.3 ultrathink

なぜここを読むか 難しいArchitectureや原因分析では、通常の指示だけだと比較や代案が浅くなります。徹底的に考えさせるSignalを意図的に使う場面があります。

やる／やらないとどうなるか ultrathinkを比較軸・制約・証拠とセットで使うと、最小Riskの計画が引き出せ、Magic Word頼みの過剰起動も避けられます。

難しい計画・Reviewで「徹底的に考える」Signalとして使える場合がある。

これは何か 高難度Plan/Reviewで徹底的に考えさせるSignalの使用例。比較軸・制約・証拠を併記して呼び出します。

実行するとどうなるか 最小Riskな計画案が比較表で返ってくるようになり、難所の判断品質が一段上がります。

ultrathink
既存Architecture、Data Migration、Failure Modeを比較し、最小Riskの計画を作ってください。

Magic Wordだけに依存せず、比較軸、制約、証拠を明示する。

8.4 Advisor

なぜここを読むか 全工程を高性能Modelで回すとコストが膨らみ、軽くするとレビュー品質が落ちます。要所だけ助言を借りる併走設計が要ります。

やる／やらないとどうなるか 標準Modelを主、Advisorを要所投入にすると、品質を保ったままコストを抑えられ、判断根拠の補強だけを得られます。

標準Modelを主に使い、重要な局面だけ高性能Modelの助言を得る設計は、品質とCostのBalanceがよい。現行の /advisor と提供条件を確認する。

8.5 Fable 5の扱い

なぜここを読むか 教材が特定モデル前提だと、提供停止や名称変更で本文ごと古びてしまいます。Workflowを特定モデルから切り離して書く必要があります。

やる／やらないとどうなるか Fable 5のような変動Modelは必須手順にせず、Registryでsuspendedとして扱うことで、教材本体の寿命を延ばせます。

素材中では長時間Taskに強い最上位Modelとして紹介されるが、基準日にはアクセス停止中である。

  • 教材の必須手順にしない。
  • Feature Registryでsuspendedを表示する。
  • 再開後もCost、Safety、対象Planを再検証する。
  • Workflow設計はModel非依存にする。

こんな場面で使う

章の内容が実務でどう活きるか。架空の場面・行動・効果の三点で示します。

場面

個人開発者（架空）が、すべての作業を最上位モデル＋最大Effortで回した結果、月初に利用上限に達して残り3週間を低性能モデルで耐えている。

こう使う

業務を軽量処理・標準処理・高難度判断の三つに分類し、Effortを上げるのは判断系のみにする。

得られるもの

利用上限の発火が遅れ、難所だけに高性能を集中投下できる。コスト効率が大きく改善する。

場面

ABC SIerのテックリード（架空）が、レビュー指摘の質に悩み、レビュー専用モデルを高Effortで分けたいが手順が決まっていない。

こう使う

Plan/Implement/Reviewそれぞれにモデル設定を分け、変動するモデル名はFeature Registryへ外出しする。

得られるもの

重要レビューだけ深く考えさせる構造になり、後でモデル名が変わってもRegistry更新で済む。
