第9章

1日繋ぎっぱなしのチャットが、判断を曇らせる SessionとContext

『朝から続けているこのチャット、AIが急に変なことを言い出した』。 これは AI の故障ではなく、古い前提を引きずっているだけです。

ほとんどの人は、AIとのセッションを『途切れたら不便だから』と1日繋ぎっぱなしにしています。 午前は予算検討、午後は Slack 文面、夕方はバグ修正。机に資料を積み続けたら、AIも混乱します。

この章で扱うのは、仕事の単位でセッションを切る運用と、再開に必要な状態を RESUME.md というファイルへ外に出しておく仕掛けです。 会話履歴に頼らず、ファイルだけで翌日続けられる状態を作ります。

セッションを切ることは、忘れることではなく、必要なものだけを次へ渡すことです。

この章の概観 ／ 何を扱い、何ができるか

この章で扱う機能と、それで何ができるようになるか。実装に入る前に、全体像をここでつかみます。

  • Session分離（仕事単位で切る）

    目的が変わるたびに新しいセッションを開き、古い前提を引きずらせない運用。文脈混入の事故を構造的に防ぐ。

    入力
    現在の作業内容、次に着手する別タスク
    出力
    目的別に分かれたセッション、引き継ぎファイル

  • /clear・/compact・/context

    会話履歴の圧縮・全消去・状態確認のコマンド群。文脈節約と再開のための日常操作。

    入力
    肥大化した会話、進行中のタスク状態
    出力
    圧縮済みコンテキスト、または新しい起点

  • RESUME.md（状態の外部化）

    再開に必要な現在地・決定・次の一操作をファイルへ書き出し、会話履歴に依存せず復帰できるようにする。

    入力
    進行中の判断、未完了タスク、TODO
    出力
    再開可能なRESUME.mdと参照Path一覧

  • Resume運用と履歴を残さない実行

    Resumeは作業再開の補助、独立Stateを正本にする運用方針。画像入力・割り込みも文脈を保つ範囲で扱う。

    入力
    中断中のセッション、機密性のある作業
    出力
    正本としての作業State、必要に応じた履歴非保持実行

実装リファレンス

9.1 Sessionを分ける基準

なぜここを読むか 1つのセッションで複数の仕事を続けると、AIは古い前提を引きずって判断を曇らせます。セッションを切る基準を明確に持つ必要があります。

やる／やらないとどうなるか 目的が変わる時・別Projectへ移る時に新Sessionを開く運用にすると、文脈混入の事故が起きにくく、回答が現状に追従します。

新しいSessionにする：

  • 目的が変わる。
  • 別Project/別Issueへ移る。
  • 前提を一度捨ててReviewしたい。
  • 長い探索後に実装へ移る。
  • 同じ会話が過去の誤りへ引っ張られる。

同じSessionを続ける：

  • 同じ完了条件へ向けた修正。
  • 直前のTest結果を受けたDebug。
  • 承認済みPlanの実装。

9.2 /clear、/compact、/context

なぜここを読むか ContextはAIの作業机のようなもので、肥大化すると判断が鈍ります。圧縮・全消去・確認の3操作を日常的に使い分ける必要があります。

やる／やらないとどうなるか /clearで仕事切替・/compactで同テーマ圧縮・/contextで内訳確認、を場面で選べると、無理な閾値Ruleに頼らず安定運用できます。

これは何か セッション制御コマンド3種の使い分け早見表。clearは切替、compactは同テーマ圧縮、contextは内訳確認です。

実行するとどうなるか 場面ごとにどれを使うかが即決でき、固定閾値Ruleに頼らず文脈量を意図的にコントロールできます。

/clear   違う仕事へ切り替える
/compact 同じ仕事の履歴を圧縮
/context 何がContextを占めるか確認

「40%を超えたら必ず性能低下」のような固定閾値を一般Ruleにしない。出力品質、Context内訳、Task境界で判断する。

9.3 Contextを節約する設計

なぜここを読むか CLAUDE.mdや長文Logを丸ごと積むと、すぐにContextが膨らみます。詰め込みではなく節約の設計を別建てで考える必要があります。

やる／やらないとどうなるか CLAUDE.mdを短く・手順はSkillへ・詳細はreferences/へ分離するなどの設計で、Contextを軽く保ったまま情報の参照性は維持できます。

  • CLAUDE.mdを短くする。
  • 長い手順をSkillへ移す。
  • 詳細をreferences/へ分離する。
  • 調査結果をClaim LedgerやSummary Fileへ保存する。
  • 大きなLogを丸ごと貼らず、必要範囲だけ読む。
  • Subagentへ独立範囲を渡す。

9.4 Resume

なぜここを読むか Resumeは便利ですが、過去Sessionの前提と現在のFile状態がずれているリスクがあります。再開時にも事実確認の工程が要ります。

やる／やらないとどうなるか --continue/--resume後にGit状態・Issue・完了条件を確認する習慣を付けると、古いContextに引きずられた誤判断を防げます。

これは何か 前回Sessionの再開に使うCLIコマンド。--continueは直前、--resumeは選択再開を担います。

実行するとどうなるか セッションを切っても作業を引き継げますが、再開時にGit状態と完了条件の再確認が必要なことを意識づけられます。

claude --continue
claude --resume

再開時も、現在のGit状態、Issue、完了条件を確認する。過去SessionのContextと現在のFileがずれている可能性がある。

9.5 履歴を残さない実行

なぜここを読むか 機密性の高い作業や自動化Jobでは、Session履歴自体が漏洩リスクや混乱要因になります。履歴を残さない実行モードが要る場面があります。

やる／やらないとどうなるか 利用Versionの--helpと組織Policyを確認して非対話モードでPersistenceを切ると、機密や自動化の要件に沿った実行ができます。

機密性や自動化要件により、非対話実行でSession Persistenceを無効化する選択肢がある。利用Versionの--helpと組織Policyを確認する。

9.6 画像と割り込み

なぜここを読むか 画像入力や作業中の追加指示は、便利な反面、機密漏洩や指示の取り違えにつながりやすい入口です。扱い方を先に決めておく必要があります。

やる／やらないとどうなるか 個人情報のMask・割り込み反映タイミング・音声指示の復唱を運用に組み込むと、UI共有や即時指示でも事故を抑えられます。

  • UI崩れ、Diagram、Error画面は画像で伝える。
  • 個人情報・秘密を先にMaskする。
  • 作業中の追加指示は次の安全な区切りで反映されることがある。
  • 緊急停止はEsc等の停止操作を使う。
  • 音声入力のPath、Command、数値は実行前に復唱させる。

こんな場面で使う

章の内容が実務でどう活きるか。架空の場面・行動・効果の三点で示します。

場面

ある経営企画（架空）が、午前は予算検討・午後はSlack文面・夕方はバグ修正を全部同じセッションで進め、AIが古い前提を引きずって変な提案を出している。

こう使う

仕事の単位ごとにセッションを切り直し、RESUME.mdへ現在地・決定・次の一操作を残す。

得られるもの

AIの回答が現状に合うようになり、翌日の再開も会話履歴ではなくファイルだけで成立する。

場面

兼業ライター（架空）が、長時間セッションで記事執筆→AIが急に話題を取り違える→/compactしても直らない、を繰り返している。

こう使う

話題切替時は新セッションを開き、必要な状態をresumeメモへ外部化する運用に変える。

得られるもの

AIの混乱が消え、文脈の引き継ぎが意図したファイルのみに限定される。
