第22章

AIに調べさせて、嘘ごと信じる事故をやめる 信頼できる調査とレポート

『AI に調べさせて、嘘ごと信じる』。 これは AI の不誠実ではなく、検証を仕組み化していないだけです。

ほとんどの人は、AIの調査結果をそのまま使っています。 ですが、出典の確認・複数経路の確認・反証の確認をしないままでは、AIの便利さは事故率を上げてしまいます。

この章で扱うのは、調査と検証を別の AI、別の文脈で走らせる仕掛けです。 情報源の独立性・反証の明示・確信度の段階を、毎回の調査に組み込みます。

この章を読み終わる頃には、AI の調査結果に対して、安心して『信じる』『保留する』を分けられるようになります。

この章の概観 ／ 何を扱い、何ができるか

この章で扱う機能と、それで何ができるようになるか。実装に入る前に、全体像をここでつかみます。

  • Source優先度（公式→一次→実測→解説→SNS）

    情報源の重みを序列化し、変動情報には確認日を付ける調査品質の土台。SNSの勢いと事実を分ける。

    入力
    集めた情報源、それぞれの種別
    出力
    重み付き情報源リストと確認日メタ

  • Claim Ledger

    主張・根拠・日付・確度・反証を1行ずつ記録する追跡可能な台帳。後で『この数字の出所は？』に即答できる。

    入力
    レポート上の主張、対応する出典
    出力
    5列構造のClaim Ledger

  • Research Skill

    対象・期間・除外条件・評価軸を内蔵した調査Skill。テーマを渡すと出典付き調査結果を返す。

    入力
    調査テーマ、期間、除外条件
    出力
    Research Skillによる出典付きレポート

  • 動画/SNS統合（発言と事実の区別）

    同じ内容を重複カウントせず、発言と事実を区別して統合する処理。SNS発の話題を業務情報に変える。

    入力
    動画・SNS投稿の集合
    出力
    重複統合済みの発言/事実分離レポート

  • 完了条件（未確認事項の明示）

    未確認事項と次に検証すべき点を明示してレポートを閉じる完了基準。隠れた不確実性を残さない。

    入力
    調査ドラフト
    出力
    未確認事項リストと次の検証アクション

実装リファレンス

22.1 Source優先度

なぜここを読むか SNSや解説記事をそのまま事実扱いすると、勢いのある誤情報まで取り込んでしまい、訂正で信頼を失います。

やる／やらないとどうなるか 公式→一次→専門媒体→解説→SNSの順に重みを下げる序列を持ち、SNSは発見と反応の窓口に限定できます。

これは何か 情報源を5段階で序列化したリストで、SNS・掲示板・Commentは最下位として扱うことを明示しています。

実行するとどうなるか どのSourceを仕様確定の根拠に使ってよいかが揃い、SNSは発見と反応に限定する運用がそのまま定着します。

それぞれの意味

公式Document/Changelog/原論文/法令/一次Data
もっとも重みが高い一次ソースです。仕様確定や法的判断はここだけを根拠にし、他段は補強・発見用と切り分けます。
開発者・当事者発表
公式ではないが当事者本人による発信です。発言の文脈と日付が確認でき、二次解釈が入る前の一次に近い情報として扱えます。
信頼できる専門媒体
編集・査読プロセスを持つ専門メディアの記事です。一次資料が読みにくい分野で、出典付きの解釈を借りるのに使えます。
高品質解説
個人ブログや専門家解説など、出典が明示されているまとめです。理解の足場として有用ですが、これ自体を仕様確定の根拠にはしません。
SNS/掲示板/Comment
もっとも重みが軽い反応・噂層です。話題発見と感触把握には使いますが、ここを事実として引用すると訂正対応で信頼を失います。
1. 公式Document/Changelog/原論文/法令/一次Data
2. 開発者・当事者発表
3. 信頼できる専門媒体
4. 高品質解説
5. SNS/掲示板/Comment

SNSは発見と反応に使い、仕様確定に使わない。

22.2 Claim Ledger

なぜここを読むか 完成したレポートだけ見ても『この数字の出所は？』に即答できなければ、役員プレゼンで矛盾を突かれます。

やる／やらないとどうなるか 主張・根拠・日付・確度・反証を行単位で記録した台帳が残り、どのClaimもSourceまで逆引きできます。

これは何か Claim LedgerをCSVで実装した1行サンプルで、status・source_type・published_at・retrieved_atを分離しています。

実行するとどうなるか 公開日と取得日が別カラムで残り、変動情報の鮮度判定や反証追加が後から差分で行える台帳になります。

それぞれの意味

claim_id
個々の主張に振る一意の識別子です。レポート本文から逆引きできる連番で、後で『この数字の出所』を即座に検索する起点になります。
claim
検証対象となる主張そのものを1文で書く列です。長い段落ではなく原子化した1主張に分けるのが、ステータス管理の前提になります。
status
verified・supported・disputed・unverified・obsoleteの5段階で確度を示す列です。未検証と検証済みを混ぜず、引用可否の判断材料にします。
source_type
official・primary・専門媒体・解説・SNSなど、出典の階層を示す列です。Source優先度の5階層と対応させ、軽い出典に依存しすぎを検知できます。
source_title
出典文書のタイトルを記録する列です。URLが切れても文書名で再検索でき、リンク切れ時の復元手段として効きます。
source_url
出典への直接リンクを記録する列です。レビュー時に1クリックで原典に飛べ、Claimの妥当性検証の手数を最小化します。
published_at
出典そのものが公開された日付の列です。情報の鮮度判定に使い、古い仕様を最新と取り違える事故を防ぎます。
retrieved_at
こちらが情報を取得した日付の列です。公開日と分けて持つことで『取得時点で正しかった』を主張でき、後の仕様変更にも追従できます。
evidence
出典内の該当箇所や具体的な数値を抜粋しておく列です。あとで原文に戻らなくても、ここを見るだけで真偽の再判定ができます。
notes
前提条件・例外・要確認事項を添える列です。Claimだけでは伝わらない注意点を残し、後任のレビュアーが同じ罠を踏まないようにします。
claim_id,claim,status,source_type,source_title,source_url,published_at,retrieved_at,evidence,notes
C001,"Dynamic Workflows is available on paid plans",verified,official,"Dynamic workflows",...,2026-06-...,2026-06-22,"...","Check minimum version"

Status：verified、supported、disputed、unverified、obsolete。

22.3 Research Skill

なぜここを読むか Research・Verify・Draftの手順を毎回口頭で伝えていると、人によって出典の扱いも反証の扱いもブレます。

やる／やらないとどうなるか 対象・除外条件・評価軸を内蔵したSkillとして固定でき、テーマを渡すだけで出典付きレポートが返ってきます。

これは何か 一次情報優先・日付分離・SNS未検証扱い・推測の明示を内蔵したResearch Skillの定義Markdownです。

実行するとどうなるか テーマを渡すとreport.md・claims.csv・sources.json・open-questions.mdまで揃った成果物が返ってきます。

それぞれの意味

Primary sources first
一次情報を最優先する原則の宣言です。Skillに内蔵することで、依頼者がうっかりSNSベースの調査を要求しても挙動が変わりません。
Publication date / event date / retrieval date
出典公開日・出来事の発生日・取得日を別物として扱うルールです。日付が1つに丸められて鮮度判定を間違える典型事故を防ぎます。
Social posts are not fact without verification
SNS投稿は未検証では事実にしないという固定ルールです。Skillの外で『SNSも入れて』と頼んでも、Skill内で検証フラグが付きます。
Mark inferences
推測には必ず推測である印を付ける原則です。検証済みClaimと推測を地続きの文章で書き、読者が事実と混同する事故を防ぎます。
research/report.md
結論と根拠をまとめた最終レポートの成果物です。読者向けの本文で、Claim Ledgerと一対で発行されることが前提になります。
research/claims.csv
Claim Ledger本体の成果物です。report.md内の各主張がここの行に対応し、status・出典まで遡れる監査経路を担います。
research/sources.json
参照した出典の構造化リストです。URL・公開日・タイトルを機械可読で持ち、後から鮮度確認や差分追跡に再利用できます。
research/open-questions.md
未確認のまま残った論点と次にすべき検証アクションのリストです。不確実性を隠さず引き渡すための完了条件として機能します。
---
name: source-grounded-research
description: Research a current topic using primary sources, build a claim ledger, verify dates, and produce a cited report.
argument-hint: "<question>"
---

# Rules
- Primary sources first.
- Publication date, event date, retrieval date separately.
- Social posts are not fact without verification.
- Mark inferences.
- Preserve credible disagreement.

# Workflow
1. Split into subquestions.
2. Define freshness.
3. Collect primary sources.
4. Build claims.csv.
5. Fresh reviewer verifies high-impact claims.
6. Draft only from verified/supported claims.
7. Add uncertainty and missing evidence.

# Deliverables
- research/report.md
- research/claims.csv
- research/sources.json
- research/open-questions.md

22.4 動画/SNS統合

なぜここを読むか 動画やSNSは同じ話題を複数の発信者が繰り返すため、件数で押し切ると『複数経路で確認した』という錯覚が生まれます。

やる／やらないとどうなるか 重複Clusterで畳んでから公式検証する流れが固まり、発言と事実、原則と例を分けて再構成できます。

これは何か 動画とSNSの統合手順を矢印で並べたフローで、重複Cluster→Claim分解→公式検証→再構成という順序を定義します。

実行するとどうなるか 同じ話題の重複カウントを潰してから公式検証に回せ、最後にCoverage Matrixで漏れの有無まで確認できます。

それぞれの意味

収集
対象の動画・SNS投稿を集める最初の段階です。母集団を決めずに次へ進むと、後段のCoverage Matrixで漏れの有無が判定できなくなります。
完全文字起こし
要約せず全文を文字化する段階です。発話のニュアンスや但し書きを取りこぼさず、後段で原則と例の分離に効いてきます。
動画別Topic
各動画ごとに扱った話題を列挙する段階です。発信者単位で論点が把握でき、次段の重複Cluster化の入力データになります。
重複Cluster
複数の動画で同じ話題が出ているものを束ねる段階です。件数で押し切ると複数経路で確認したと錯覚するため、ここで重複を畳みます。
Claim分解
Clusterごとに検証可能な単位の主張へ分解する段階です。長い文を原子化することで、Claim Ledgerの1行に落とせる粒度になります。
公式検証
分解したClaimを公式情報源と突き合わせる段階です。動画内の主張が現行仕様と一致するかをここで初めて事実認定します。
原則/手順/例/注意へ再構成
検証済みClaimを4区分に並べ替える段階です。発言順や動画順ではなく、読者が使える知識構造に組み直します。
Coverage Matrix
対象テーマ × 出典の網羅状況を表で確認する最終段階です。どの観点に出典が不足しているかが一目で見え、追加調査の的が絞れます。
収集 → 完全文字起こし → 動画別Topic → 重複Cluster → Claim分解
→ 公式検証 → 原則/手順/例/注意へ再構成 → Coverage Matrix

22.5 完了条件

なぜここを読むか 未確認のまま閉じたレポートは、後で読む人に確認済みかどうかが判別できず、誤って意思決定に使われます。

やる／やらないとどうなるか 未確認事項リストと次の検証アクションをレポート末尾で明示し、不確実性を隠さず閉じる完了基準になります。

[ ] 質問へ直接回答
[ ] 重要Claimに根拠
[ ] 現在/過去を分離
[ ] 投稿日/Event Dateを分離
[ ] 反対情報
[ ] 推論の明示
[ ] 数値の単位/母数
[ ] 未確認事項
[ ] Sourceと本文の対応
[ ] 次Action

こんな場面で使う

章の内容が実務でどう活きるか。架空の場面・行動・効果の三点で示します。

場面

経営企画（架空）が、AIに業界レポートを作らせたが、根拠不明の数字や古い情報が混じり、役員プレゼンで矛盾を指摘された。

こう使う

Claim Ledgerに主張・根拠・日付・確度・反証を必須項目で記録するSkillを作る。

得られるもの

プレゼン前に根拠を逆引きでき、『この数字の出所は？』に即答できる。

場面

ある個人ブロガー（架空）が、最新機能の解説を書こうとして、SNSの噂と公式情報が混在し、結果として誤情報を発信して訂正に追われた。

こう使う

出典の優先度を公式→一次→解説→SNSの順で重み付けし、変動情報には確認日を付ける。

得られるもの

発信前にSNS情報を切り分けでき、訂正リスクが大幅に減る。
