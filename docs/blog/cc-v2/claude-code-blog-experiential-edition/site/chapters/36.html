30日後、AIを使える組織になっている 30日導入計画

『AIを社内に導入したいが、何から始めれば 30 日で形になるのか分からない』。 これは戦略の不足ではなく、計画の粒度の問題です。

ほとんどの人は、AI導入を『年単位の構想』として描いてしまいます。 ですが、本書の範囲なら、30 日で組織の地肩を上げることは十分にできます。

この章で扱うのは、Week1〜Week4 の具体的なアクションを並べた 30 日計画です。 誰が・何を・どの順序で・どこまで実行すれば、組織が AI を扱える状態になるかを、現実的な粒度で並べます。

30 日後、AI を使える組織になっている。これは過剰な約束ではなく、計画と実行の話です。

実装リファレンス

Week 1

なぜここを読むか 正本と禁止事項を決めないままCanaryを始めると、何を測ればよいか分からず、後工程で必ず差し戻しになります。

やる／やらないとどうなるか 本書Review・46 Skillの棚卸し・Naming Migration・Managed Settings Canary・Bypass検証・Release Policyを1週目に固めます。

これは何か Week1のチェックリスト。本書Review・Skill棚卸し・Naming Migration・Managed Settings Canary・Bypass検証・Release Policyの6項目です。

実行するとどうなるか 1週目の終わりに、現有Skillの正本と禁止事項・配布ポリシーが文書として揃い、2週目以降のBuildへ安全に渡せます。

- 本書Review
- 46 Skill Inventory
- Naming Migration
- Managed Settings Canary
- Bypass/Secret/Sandbox検証
- Release Policy

Week 2

なぜここを読むか 正本ソースとFixtureが揃わないまま配布すると、誰が何を更新したかが追えなくなり、Rollbackも実行できません。

やる／やらないとどうなるか skill-srcの正本化・build/validate・Hook Fixture・syncのDry-run/Delete/Preserveテスト・Rollback手順を2週目に整えます。

これは何か Week2のチェックリスト。正本化・build/validate・Hook Fixture・syncのDry-run/Delete/Preserveテスト・Rollback整備の5項目を並べています。

実行するとどうなるか Canary配布前に必要なビルド・検証・Rollback経路が一通り揃い、配布中の事故時に元の状態へ戻せる準備が完了します。

- skill-src正本化
- build/validate
- Hook Fixture
- sync Dry-run/Delete/Preserve Test
- verify/Rollback

Week 3

なぜここを読むか 管理者だけで完結した配布は現場で必ず破綻します。失敗を口頭で言ってくれる小集団で先に当てる必要があります。

やる／やらないとどうなるか 管理者+開発者でCanary・Bronze研修・Meeting Assistant運用・誤検知や教材修正の収集を3週目で並走させ、小さく失敗を見つけます。

これは何か Week3のチェックリスト。Canary対象・Bronze研修・Meeting Assistant運用・誤検知やSandbox観測・教材修正までを5項目に圧縮しています。

実行するとどうなるか Canary実績ログとBronze認定者リストが残り、設計上の問題を全社展開前に検出した状態でWeek4へ引き継げます。

- 管理者+開発者Canary
- Bronze研修
- Meeting Assistant
- 誤検知/Sandbox/Skill観測
- 教材修正

Week 4

なぜここを読むか Canaryで止まると価値が組織に届きません。Exit条件を満たした時点でPilotとStable Releaseへ進ませる仕組みが要ります。

やる／やらないとどうなるか Pilot部署選定・Incident/Help Desk・Version Dashboard・Stable Release・全社員研修・月次Reviewを4週目で同時に立ち上げます。

これは何か Week4のチェックリスト。Pilot拡大・Incident/Help Desk・Version Dashboard・Stable Release・全社員研修・月次Reviewの6項目です。

実行するとどうなるか Stable Releaseと配布完了レポートが揃い、運用Loopを月次で回し続ける常設体制に移行できます。

- Pilot部署
- Incident/Help Desk
- Version Dashboard
- Stable Release
- 全社員研修
- 月次Review

30日後

なぜここを読むか 「やった感」だけで30日を終えると、経営報告に使える数字が残らず次の予算も取れません。最初から計測指標を宣言しておく必要があります。

やる／やらないとどうなるか 新人30分・Bronze完了・Stable 95%更新・Rollback Drill・Skill二重編集0・Critical Gap 0・Topic Coverage 100%を到達条件として並べます。

これは何か 30日後の到達条件チェックリスト。新人30分・Bronze完了・Stable 95%更新・Rollback Drill・Skill二重編集0などの計測指標です。

実行するとどうなるか 経営報告に使える定量指標が揃い、AI推進の費用対効果をCEOへ数字で説明できる状態になります。

実務に落とすときの補足

4週間で詰めすぎないコツ
1週目のInventoryは最低でも3営業日、Policyドラフトに2営業日確保しないと、後半のCanaryで必ず差し戻しが発生します。最初の週は『動かさない作業しかしない』ことを宣言し、その分2週目以降のテンポを上げる設計にすると失敗が減ります。
Canary対象の選び方
Canary対象は『AI習熟度が高い人』ではなく、『業務頻度が高く、失敗を口頭で言ってくれる人』を3〜5名選びます。習熟度の高い人は自分で迂回路を作ってしまうため、設計上の問題が検出されにくくなります。
30日後指標の運用
指標は5〜7個に絞り、毎週月曜にダッシュボードへ自動転記します。指標を10個以上並べると誰も見なくなり、結果として導入の費用対効果がCEOに伝わりません。指標を絞り切る判断が、組織展開の成否を分けます。
次の章への接続
4週間ロードマップが回り始めると、次章のFeature/原則分離が真価を発揮します。配布物が増えれば増えるほど、長く使える原則と頻繁に変わる製品情報を別々の更新周期で管理する必要が高まるからです。
