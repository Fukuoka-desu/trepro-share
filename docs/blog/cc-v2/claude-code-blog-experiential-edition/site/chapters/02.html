便利と事故は、同じスイッチで繋がっている 安全を最初に設計する

『rm -rf』をAIにうっかり打たせて、心臓が止まる。 『大丈夫だろう』で本番フォルダを触らせて、戻し方が分からない。 こういう事故は、油断した10秒に起きます。

ほとんどの人は、安全を『気をつけよう』というメモで済ませてます。 でも、メモは読まれません。本当に止めたい操作は、最小権限・隔離・履歴・検証・監査の5要素で物理的に止めるのが正解です。 便利な『お願い』（Guidance）と、確実に止める『強制』（Enforcement）の境界を、最初に決めます。

設定は5分。一度作った学習用フォルダの3禁（削除しない／外部公開しない／秘密情報を貼らない）が、その後ずっと自分を守ってくれます。

実装リファレンス

2.1 安全の基本式

なぜここを読むか 『気をつけよう』は読まれないし守られない。事故は油断した10秒で起きる。安全は要素を掛け算で重ねる必要がある

やる／やらないとどうなるか 最小権限×隔離×可逆性×検証×監査の5層で守れば、どれか1つが破れても被害は限定される。事故が起きても戻せる

これは何か 安全の基本式。最小権限・隔離・可逆性・検証・監査の5要素を掛け算で重ねる安全設計の土台

実行するとどうなるか どの要素で何を守っているかを言語化できる。穴があれば即座に補強箇所が見える

安全性 = 最小権限 × 隔離 × 可逆性 × 検証 × 監査

Claude Codeは優秀だが、初日の新人と同じく、Projectの暗黙知を知らない。最初から本番権限を渡さない。

2.2 GuidanceとEnforcement

なぜここを読むか CLAUDE.mdの『お願い』は守られないことがある。本当に止めたい操作はPrompt層ではなく権限・Hook層で物理的に止める必要がある

やる／やらないとどうなるか 守って欲しいだけならGuidance、絶対に止めたいならEnforcementと判断軸が分かれる。重大事故は仕組みで阻止できる

内容 Guidance Enforcement
日本語で返す CLAUDE.md 不要
.envを読まない CLAUDE.md Permission deny / Sandbox denyRead
削除前に確認 CLAUDE.md Ask Rule / Hook
本番Deploy禁止 CLAUDE.md Ask/Deny + Hook + CI権限
Typecheck必須 CLAUDE.md PostToolUse Hook / CI

重要な禁止事項をPromptだけに頼らない。

2.3 初心者が最初に禁止すること

なぜここを読むか 初日に踏みやすい地雷は決まっている。これを最初に潰さないと、便利な機能を試す前に事故で学習が終わる

やる／やらないとどうなるか 権限スキップ・本番Folder・APIキー貼付・無承認操作・テスト弱体化・出所不明Plugin・Diff未確認Commitを禁止と決められる

これは何か 初心者が最初に禁止する7項目。権限スキップ・本番練習・APIキー貼付・テスト弱体化・出所不明Pluginなど

実行するとどうなるか やってはいけない操作リストが手元に残る。初日に踏みやすい地雷をすべて先回りで潰せる

- --dangerously-skip-permissions
- 本番・顧客Folderでの練習
- APIキーの貼り付け
- 無承認の削除・送信・公開・Deploy
- Test/Lint/Typecheckの弱体化
- 出所不明MCP/Pluginの追加
- Git diffを見ないCommit

2.4 安全な練習場

なぜここを読むか 本番フォルダや顧客データの上で練習すると、失敗が即事故になる。学習用の物理境界が要る

やる／やらないとどうなるか 個人用Copyだけを置いたLabが完成する。秘密情報と本番接続は持ち込まれず、失敗してもLabの中で完結する

これは何か 学習用Labを作るbashコマンド。フォルダ作成→git init→README commit→claude起動まで一気通貫

実行するとどうなるか 本番データから物理的に切り離された練習場が完成する。安全に失敗できる場所を5分で用意できる

mkdir -p ~/claude-lab/meeting-assistant
cd ~/claude-lab/meeting-assistant
git init
printf '# Claude Lab\n' > README.md
git add README.md
git commit -m 'chore: initialize safe lab'
claude

個人用のCopyデータだけを置き、Secretsと本番接続を入れない。

2.5 外部副作用はCheckpointで戻らない

なぜここを読むか Email送信・公開・Deploy・課金は、ファイルの巻き戻しでは元に戻らない。CheckpointとGitだけでは守れない領域がある

やる／やらないとどうなるか 外部副作用はDraft→Preview→人間承認→Executeに分離する。誤送信・誤公開・誤課金の事故が物理的に起きにくくなる

CheckpointやGitで戻せるのはローカルファイル中心である。次は別のRollbackが必要。

  • 送信したEmail。
  • 公開したWebページ。
  • Production Deploy。
  • DB更新。
  • Calendar作成・削除。
  • Payment、注文、契約。

外部副作用は Draft → Preview → Human Approval → Execute に分離する。
