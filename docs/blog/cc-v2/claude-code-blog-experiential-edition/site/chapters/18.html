AIと2人で、同じ画面を編集する CoworkとLive Artifacts

『AIと一緒に作業しているはずなのに、結局画面を行き来している』。 これは慣れの問題ではなく、共有空間が無いだけです。

ほとんどの人は、AIの出力をコピーし、エディタへ貼り、戻ってきて指示し直しています。 そのたびに、文脈の半分が抜け落ちます。

Cowork と Live Artifacts は、AIと人が同じファイルを同時に編集できる場所です。 AI が直したところがそのまま反映され、人が直したところを AI が拾います。

この章を読み終わる頃には、『AI に頼む』から『AI と並んで書く』へ、距離感が変わります。

この章の概観 ／ 何を扱い、何ができるか

この章で扱う機能と、それで何ができるようになるか。実装に入る前に、全体像をここでつかみます。

  • Cowork（非開発業務の複数ステップ）

    Projectで文脈を固定し、複数ステップの業務処理を継続実行する作業形態。Codeとは別の入口として扱う。

    入力
    繰り返し発生する業務、参照する文脈ファイル
    出力
    継続実行可能なCowork Project

  • Live Artifact 5層構造

    データ取得・整形・表示・AI要約・キャッシュの5層に分けて設計する、開くたびに最新化する業務インターフェース。

    入力
    対象データソース、表示したい指標
    出力
    5層分離されたLive Artifact実装

  • 会議Dashboard Prompt

    Driveの会議ファイルを開くたびに取り込み・要約・キャッシュするダッシュボード用プロンプトのテンプレ。

    入力
    Drive上の会議ファイル群、要約観点
    出力
    リロード時に最新化される会議Dashboard

  • Scheduled設計

    定期実行で重複防止・ロック・ログ・通知・再試行を持たせる業務スケジュール設計。

    入力
    実行頻度、失敗時の挙動、通知先
    出力
    Scheduledジョブ設定と運用ログ

  • Acceptance（更新/キャッシュ/出典）

    Live Artifactの完成判定。更新時刻・キャッシュ振る舞い・出典表示の3点をAcceptance基準にする。

    入力
    完成したArtifact、運用シナリオ
    出力
    3点Acceptance通過記録、Stale時の挙動確認

実装リファレンス

18.1 Cowork向き

なぜここを読むか Codeに何でも持ち込むと、議事録要約や朝のニュース整理のような軽い業務まで重装備になり続きません。

やる／やらないとどうなるか Cowork向き・Code向きの境目を持つと、Git・認証・Deployが要らない仕事はProjectの中で軽く回せます。

  • 議事録→週報/Task。
  • Drive資料の整理・要約。
  • 毎朝News Dashboard。
  • 社内文書View。
  • Scheduled Task。

Codeへ移す目安：Git履歴、独自Library/Test、認証/DB、本番Deploy、複雑な監視が必要。

18.2 主要領域

なぜここを読むか Project・Artifacts・Scheduled・Dispatch・Customizeを名前で覚えると、Beta更新ごとに教材が陳腐化します。

やる／やらないとどうなるか 役割で整理しておけば、UI名が変わっても『どこで何をする領域か』が崩れず、新人にも説明し直しやすくなります。

領域 役割
Project Folder、指示、素材
Artifacts 永続的なHTML等
Scheduled 定期実行
Dispatch 離れた端末から指示
Customize Skills/Connectors

UI名より役割を教材化する。

18.3 Live Artifactの5層

なぜここを読むか 毎回フルでデータ取得・要約をやり直すArtifactは、コスト・速度・失敗時の挙動のどれも安定しません。

やる／やらないとどうなるか Source→Fetch→Transform→Cache→Viewの5層に分けて設計すれば、変更分だけ再処理・失敗時はStale表示と切り分けられます。

これは何か Live Artifactを5層で表現した設計図で、取得から表示までの責務分割を1行で示しています。

実行するとどうなるか どの層で失敗したかが切り分けやすくなり、再要約や再取得の最小範囲を判断できます。

それぞれの意味

Source
データの源を指す層です。Driveや外部APIなど『どこから取るか』を1箇所に固定することで、変更時の影響範囲を限定できます。
Fetch
Sourceから実際にデータを取得する層です。MCP等の経路と認証をここに閉じ込めることで、上位の表示処理が認証の詳細を知らずに済みます。
Transform
取得した生データを整形・要約する層です。表示用の構造に変える処理をここに集めると、表示と取得の責務が混ざりません。
Cache
整形済みデータを一時保存する層です。変更がない部分の再処理を省き、コストと表示速度の両方を安定させられます。
View
Cacheを読んで利用者に見せる層です。Filterや検索もここで完結させると、データ取得を介さず軽く操作できます。
Source → Fetch → Transform → Cache → View
Drive     MCP     Parse/Summary  Hash    Filter/Search/Ask

18.4 会議Dashboard Prompt

なぜここを読むか 会議要約を毎週手作りすると、属人化と更新遅延が同時に進み、週末まで状況が見えなくなります。

やる／やらないとどうなるか Driveを参照するLive Artifactのプロンプトテンプレを持てば、開くたびに最新状態が出る業務インターフェースに置き換えられます。

これは何か 会議Dashboardを生成するためのテンプレで、表示項目・Performance・Safetyを並列で指示しています。

実行するとどうなるか source_id+content_hash+prompt_versionのキャッシュ設計まで含んだ、運用に出せる粒度のArtifactが得られます。

Google Driveの「全社会議文字起こし」を参照するLive Artifactを作る。

- 最新File一覧
- 日付、題名、参加者、決定、担当、期限
- 全文検索/担当Filter
- 原文Link
- 短いAI要約
- 会議ごとの質問欄
- 失敗/権限不足/空Data状態

Performance:
- source_id + content_hash + prompt_versionでCache
- 変更Fileだけ再要約
- 30秒Timeout、Retry Button

Safety:
- Read-only
- 権限変更なし
- 原文全文/個人情報をLocal Storageへ保存しない

動画で紹介されたArtifact内API名はBetaで変化し得る。作成時の公式Helpを正とする。

18.5 Scheduled設計

なぜここを読むか 定期実行を雑に組むと、重複起動・タイムゾーン違い・失敗の沈黙が運用事故として表面化します。

やる／やらないとどうなるか 対象期間・Lock・Retry上限・通知・0件時挙動を先に決めておけば、Scheduledを安心して放置できます。

これは何か Scheduledジョブを設計する際の8項目で、重複起動Lockや0件挙動まで網羅したチェック観点です。

実行するとどうなるか 定期実行に起きやすい『黙って失敗』『二重起動』を、設計段階で潰せます。

それぞれの意味

対象期間
ジョブが処理対象とする時間範囲を決めます。曖昧だと『どこからどこまで』が実行ごとにずれ、抜けや二重処理の原因になります。
既処理判定
既に処理済みのデータを判別する仕組みです。なければ同じデータを何度も処理し続け、外部APIコストや通知重複を招きます。
Timezone
実行時刻の解釈に使う時間帯を固定します。サーバーと利用者でずれると、深夜実行のはずが昼に走るなどの事故が起きます。
Retry上限
失敗時の再試行回数の上限です。上限を決めないと、外部障害時に無限再試行で他処理まで巻き添えにします。
重複起動Lock
前回の実行が終わる前に次が走るのを防ぐ仕組みです。なければ同じ処理が二重に走り、出力が壊れたり外部APIを過剰に叩いたりします。
失敗通知
ジョブ失敗時に人へ知らせる経路です。設計しないと、止まっていることに気付かず数日経つ『黙って失敗』が起きます。
保存先
処理結果を書き出す場所をあらかじめ決めます。決めないと出力が散らばり、次のジョブから参照できません。
0件時の挙動
処理対象が0件だった時の振る舞いです。エラー扱いか正常終了かを決めておかないと、運用通知が誤報で埋まります。
- 対象期間
- 既処理判定
- Timezone
- Retry上限
- 重複起動Lock
- 失敗通知
- 保存先
- 0件時の挙動

18.6 Acceptance

なぜここを読むか Artifactは見た目が動けば完成に見えてしまい、Cacheや出典の不備が後から効いてきます。

やる／やらないとどうなるか 更新・キャッシュ・出典の3点をAcceptance基準として明文化すると、運用に渡せる完成判定が機械的に下せます。

これは何か Live ArtifactのAcceptance基準10項目で、更新・キャッシュ・出典・モバイル・権限まで含めたゲートです。

実行するとどうなるか 見た目が動くだけのデモから、運用に渡せるDashboardへの卒業判定が一定にできます。

それぞれの意味

Add/Deleteで更新
データの追加・削除が画面に反映されることの確認です。固定スナップショットのままだと、最新状況に基づく判断ができません。
未変更Dataを再処理しない
中身が変わっていないデータには再要約を走らせないことの確認です。コストと所要時間を、変更分だけに抑えられます。
権限不足説明
アクセス権がない時にその旨を明示することです。黙って空表示にすると、利用者が『データがない』と誤解します。
0件/部分失敗/Timeout
0件・一部失敗・Timeoutといった非常時の見え方を確認することです。失敗状態が正常と区別できないと、判断ミスにつながります。
原文へ戻れる
要約や集計から元の原文へ辿れる導線です。怪しい記述があった時、原典に当たれないと信頼性が崩れます。
AI要約と原文を区別
AIが要約した部分と原文そのままの部分が見分けられることです。区別がつかないと、AIの解釈ミスが事実として扱われます。
更新日時
画面のデータがいつ時点のものかを表示することです。古い情報を最新と誤認することによる意思決定ミスを防げます。
Mobile
スマートフォン幅でも操作・閲覧できることの確認です。移動中や会議室の手元で開いた時に使えない状態にしません。
Write権限不要
閲覧用Dashboardが書き込み権限を要求しないことです。読み取りに不要な強権限を渡さないことで、誤操作の被害面を狭められます。
Browser Storageへ機密なし
ブラウザのLocal Storage等に機密情報を残さないことの確認です。共有端末や開発者ツール経由の漏えい経路を物理的に閉じられます。

[ ] 未変更Dataを再処理しない
[ ] 権限不足説明
[ ] 0件/部分失敗/Timeout
[ ] 原文へ戻れる
[ ] AI要約と原文を区別
[ ] 更新日時
[ ] Mobile
