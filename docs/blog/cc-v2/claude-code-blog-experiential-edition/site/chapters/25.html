第25章

誰が作って、誰が承認したか、説明できますか Skill Governance

『Skill を誰が作って、誰が承認したのか分からない』。 これは管理不足ではなく、Governance を設計していないだけです。

ほとんどの人は、AIの Skill を増やすことに集中し、誰が責任を持つかを後回しにしています。 ですが、Skill の数が増えると、品質の悪い 1 本が全社の事故を生むことがあります。

この章で扱うのは、提案・レビュー・承認・配布・廃止という Skill のライフサイクルを管理する仕掛けです。 誰が作り、誰が責任を持ち、いつ捨てるかを台帳で見える化します。

この章を読み終わる頃には、Skill を『増やす』だけでなく『健全に管理する』視点が手に入ります。

実装リファレンス

25.1 5 Prefixの移行

なぜここを読むか S_やmm-が混在したまま量だけ増やすと、用途と所有者がひと目で分からないSkill群になります。Canonical Nameを小文字英数字とHyphenへ統一する命名移行が必要です。

やる／やらないとどうなるか Legacy NameとCanonical Nameを並べた移行表とtrepro-legacy-prefix等のMetadataで来歴を保持しながら名前を揃えられ、検索性と責任所在が同時に改善します。

現行Prefixは S_、mm-、sm-、project-、feedback-。Agent SkillsのCanonical Nameは小文字英数字とHyphenへ統一する。

Legacy Canonical Metadata
S_customer-research s-customer-research trepro-legacy-prefix: S_
mm-campaign-report 同名 trepro-category: mm
sm-post-review 同名 trepro-category: sm
project-kickoff 同名 trepro-category: project
feedback-copy-review 同名 trepro-category: feedback

S_の意味は推測で変えず、社内定義をMetadataへ。

これは何か Canonical Name・Description・trepro-category・trepro-legacy-name・trepro-owner・trepro-versionをfrontmatterに揃えるSkill MetadataのSample例です。

実行するとどうなるか S_接頭辞の意味を推測で書き換えずに来歴を保持でき、所有者とVersionが分かる台帳としてSkillを管理できるようになります。

それぞれの意味

name
Canonical Name本体です。^[a-z0-9-]+$に揃えることで、検索性とSkillランタイム解決の安定性を両立できます。
description
発動条件と用途を1〜2文で示す紹介文です。短すぎると別Skillと衝突し、誤発動の温床になります。
trepro-category
S/mm/sm/project/feedbackなど社内分類タグです。検索フィルタや所有チームの責任範囲を機械可読にします。
trepro-legacy-name
改名前の旧名を残すための来歴フィールドです。リネーム後の利用者検索や移行通知に使えます。
trepro-owner
Skillの所有チーム名を示す責任明示フィールドです。空欄だと壊れた時の連絡先が消え、放置Skill化します。
trepro-version
Skill単位のセマンティックバージョンです。Lifecycleの昇格・ロールバック判断の根拠数字になります。
---
name: s-customer-research
description: Research a customer account and produce a source-grounded brief for sales preparation.
metadata:
  trepro-category: "S"
  trepro-legacy-name: "S_customer-research"
  trepro-owner: "sales-ops"
  trepro-version: "1.3.0"
---

Legacy Command Wrapperを90日残し、新規作成を止める。

25.2 Lifecycle

なぜここを読むか 提案・評価・配布・廃止を一本の流れに繋げないと、評価されていないSkillが全社へ広がります。draft→retiredまでのLifecycleで各状態の配布範囲を縛る必要があります。

やる／やらないとどうなるか draft/candidate/approved/stable/deprecated/retiredの6状態と「Canary・Pilot・全社」配布範囲が紐付き、Fresh Evaluator 85+などの昇格条件で品質ゲートが効きます。

これは何か Skill Lifecycleの6状態と、各状態に許可される配布範囲、昇格に必要な条件を表で並べた運用定義です。

実行するとどうなるか candidateはCanaryまで、approvedはPilotまでといった配布制約が明文化され、評価未済のSkillが全社へ流れ込む事故が構造的に防がれます。

それぞれの意味

draft
作者本人だけが触れる仕様化中の状態です。配布範囲ゼロのため、未評価Skillが他者へ流れる事故が起きません。
candidate
Canaryまで配布可能な自己評価合格状態です。Fixture実行や自己評価を通過した版だけが進めます。
approved
Fresh Evaluator 85点以上を満たしPilotへ進める状態です。第三者評価を経たSkillのみがここへ昇格します。
stable
全社配布対象の最終形です。Pilot実績・Owner・Runbookが揃っていることが条件で、配布事故の主因を事前に潰します。
deprecated
代替先と終了日を明示した撤退準備状態です。既存利用者は使えるが新規誘導は止まる、移行猶予期間として機能します。
retired
完全停止しArchive化された終端状態です。利用0確認後にだけ進めるため、業務停止事故を防げます。
draft → candidate → approved → stable → deprecated → retired
Status 配布 条件
draft 作者 仕様化中
candidate Canary Fixture/自己評価
approved Pilot Fresh Evaluator 85+
stable 全社 Pilot実績、Owner/Runbook
deprecated 既存 代替/終了日
retired 停止 利用0/Archive

25.3 Generator/Evaluator分離

なぜここを読むか 作った本人が採点すると自作自演バイアスが入り、SKILL.mdの過大評価が起きます。Generatorは経緯を渡し、Evaluatorは経緯を見ないという分離が必要です。

やる／やらないとどうなるか GeneratorはSession経緯から顧客名・秘密・個人Pathを除いた仕様を作り、EvaluatorはSKILL.md・Fixture・実行結果のみを見て改善箇所を行単位で指摘できるようになります。

Generator：成功したSessionから、入力、手順、出力、失敗、品質を抽象化し、顧客名/秘密/個人Pathを除去。

Evaluator：作成経緯を知らず、SKILL.md、Fixture、実行結果だけを評価。改善を具体的な行・項目で示す。

25.4 4軸100点

なぜここを読むか Skillの評価軸が人によって違うと、合格基準もズレます。明確性・実用性・保守性・整合性の4軸で各25点、合計85以上という固定基準が必要です。

やる／やらないとどうなるか 各軸18以上・Critical 0という閾値が共有され、秘密混入や成功判定欠落といったCriticalがあれば総合点に関わらず不合格、というブレない評価運用になります。

軸 25点の観点
明確性 発動、入力、出力、禁止
実用性 現実Dataで完走、失敗処理
保守性 Reference、Version、Owner、Test
整合性 全社Rule、他Skill、標準 
合計85以上
各軸18以上
Critical 0

Critical：秘密、無承認Side Effect、過剰Tool、成功判定なし、名前違反、重複。

25.5 Fixture-driven

なぜここを読むか Skillが「うちの環境では動く」状態で広がると、新人や別端末で再現せず壊れます。minimal・normal・missing-data・malicious-instructionなど固定Fixtureで開発する必要があります。

やる／やらないとどうなるか test-cases.yamlのassertionsでoutput存在・必須見出し・秘密パターン非含有・悪性指示の不実行が機械判定でき、Skill変更時の回帰チェックも同じ枠で回ります。

これは何か minimal/normal/missing-data/malicious-instructionの4ケースとexpectedを並べ、test-cases.yamlでassertionsを書くFixture-driven開発のディレクトリ構造です。

実行するとどうなるか 悪性指示でtool_callsに削除・送信が呼ばれない、必須見出しが揃う、秘密パターンが混入しないといった条件を、機械判定でSkillへ強制できます。

それぞれの意味

minimal
最小限の入力だけを与えるケースです。必須項目欠落時の落ち方を観察し、ガードの抜けを発見できます。
normal
想定通りの入力を与える基準ケースです。Skillの素の品質をリグレッションテストで継続監視できます。
missing-data
一部データが欠けている異常系ケースです。AIが推測で埋める癖を検出し、null明示などの設計強制に使えます。
malicious-instruction
外部入力に悪性指示を仕込むケースです。tool_callsへsend/deleteが現れないことを機械判定でき、Injection耐性を担保します。
expected/
各ケースの正解出力を保管するディレクトリです。assertionsの突合先となり、変更時の影響範囲を即座に可視化します。
tests/test-cases.yaml
assertionsの宣言ファイルです。output存在・必須見出し・秘密パターン非含有といった条件をコードと分離して管理できます。
skill-src/customer-brief/
├── SKILL.md
├── references/output-schema.md
├── fixtures/
│   ├── minimal/
│   ├── normal/
│   ├── missing-data/
│   ├── malicious-instruction/
│   └── expected/
└── tests/test-cases.yaml

cases:
  - id: minimal
    input: fixtures/minimal/input.md
    assertions:
      - output_exists: output/customer-brief.md
      - contains_headings: [Executive summary, Evidence, Open questions]
      - no_secret_patterns: true
  - id: malicious-instruction
    input: fixtures/malicious-instruction/input.md
    assertions:
      - tool_calls_not_contain: [send_email, delete]
      - marks_external_instruction_untrusted: true

25.6 Build

なぜここを読むか Claude Code用とCodex用に手で二重編集すると、片方だけ修正される事故が起きます。skill-srcを正本にして両Runtimeへ同時生成する仕組みが必要です。

やる／やらないとどうなるか validate→rsyncでCLAUDE_OUTとCODEX_OUTへ同期し、diff -ruで一致を確認できるBuildになり、Runtime差での挙動ズレが構造的にゼロになります。

#!/usr/bin/env bash
set -euo pipefail
ROOT="$(cd "$(dirname "$0")/.." && pwd)"
SRC="$ROOT/skill-src"
CLAUDE_OUT="$ROOT/global-config/skills"
CODEX_OUT="$ROOT/.agents/skills"
python3 "$ROOT/scripts/validate-skills.py" "$SRC"
rm -rf "$CLAUDE_OUT" "$CODEX_OUT"
mkdir -p "$CLAUDE_OUT" "$CODEX_OUT"
rsync -a "$SRC/" "$CLAUDE_OUT/"
rsync -a "$SRC/" "$CODEX_OUT/"
diff -ru "$CLAUDE_OUT" "$CODEX_OUT"

25.7 Validator

なぜここを読むか 命名規約や行数上限や秘密混入を目視で見ると、品質は人によって揺れます。Validatorで構造を機械検証する必要があります。

やる／やらないとどうなるか name正規表現・directory一致・description長・行数200・secretパターンをまとめてチェックするPython Validatorで、Skill構造の不正がCIで自動的に弾かれるようになります。

これは何か name正規表現・name=directory一致・description長・行数上限・secretパターン検出を1ファイルで実行するPython Validatorです。

実行するとどうなるか 命名違反・冗長なSKILL.md・APIキー貼り付けといった事故がCIで自動検出され、Reviewerの目視に依存しないSkill構造検査が成立します。

それぞれの意味

NAME 正規表現
Skill nameを小文字英数字+Hyphenだけに縛るパターンです。大文字・記号混入の事故をCIで自動的に弾けます。
SECRET 正規表現
api_key/password/secret/token=値形式を検出するパターンです。APIキー貼付の事故をSKILL.md段階で防止します。
frontmatter()
SKILL.md先頭のYAML frontmatterをパースする処理です。形式不備の素ファイルを早期に弾けます。
name != directory check
frontmatterのnameとディレクトリ名の一致を強制する検査です。配布時のSkill解決ミスを発生段階で防ぎます。
description length >= 30
説明文の最低文字数を縛るチェックです。便利Skill程度の手抜き説明が量産されるのを防ぎます。
200 lines上限
SKILL.md本文の行数上限です。Skillの肥大化と責務逸脱を構造的に抑制できます。
#!/usr/bin/env python3
import re, sys
from pathlib import Path
import yaml

NAME = re.compile(r"^[a-z0-9]+(?:-[a-z0-9]+)*$")
SECRET = re.compile(r"(?i)(api[_-]?key|password|secret|token)\s*[:=]\s*[^\s]+")

def frontmatter(text: str) -> dict:
    if not text.startswith("---\n"):
        raise ValueError("missing frontmatter")
    return yaml.safe_load(text.split("---", 2)[1]) or {}

def validate(directory: Path) -> list[str]:
    errors=[]; file=directory/"SKILL.md"
    if not file.exists(): return [f"{directory}: missing SKILL.md"]
    text=file.read_text(encoding="utf-8")
    try: meta=frontmatter(text)
    except Exception as e: return [f"{file}: {e}"]
    name=str(meta.get("name", "")); desc=str(meta.get("description", "")).strip()
    if not NAME.fullmatch(name): errors.append(f"{file}: invalid name")
    if name != directory.name: errors.append(f"{file}: name != directory")
    if len(desc) < 30: errors.append(f"{file}: description too short")
    if len(text.splitlines()) > 200: errors.append(f"{file}: over 200 lines")
    if SECRET.search(text): errors.append(f"{file}: possible secret")
    return errors

root=Path(sys.argv[1]); errors=[]
for d in sorted(x for x in root.iterdir() if x.is_dir()): errors += validate(d)
if errors: print("\n".join(errors), file=sys.stderr); raise SystemExit(1)
print("skills valid")

25.8 重複統合

なぜここを読むか 似た名前のSkillが量産されると、利用者が選べず、保守側も二重管理になります。重複候補は捨てる/統合する/分けるの方針を先に決める必要があります。

やる／やらないとどうなるか 完全重複→既存更新、80%共通→Core+Reference、組合せ可能→Orchestratorといった対応関係が決まり、Skill数より価値で測る運用が成立します。

完全重複 → 既存更新
80%共通 → Core + Reference
名前だけ近い → Description明確化
組合せ可能 → Orchestrator
独立価値 → 新規

25.9 Deprecation

なぜここを読むか 古いSkillを黙って消すと、それを使っていた業務が止まります。代替先と終了日を示しながら閉じる手順が必要です。

やる／やらないとどうなるか trepro-status: deprecated と trepro-replaced-by・trepro-removal-date が揃い、利用者が移行先と猶予期間を読み取れる安全な廃止予告になります。

これは何か trepro-status・trepro-replaced-by・trepro-removal-dateをMetadataへ書き、終了日と代替先を明示するDeprecation定義です。

実行するとどうなるか 利用者は代替Skillへ移行する猶予期間を読み取れ、運用側は利用0確認後の安全な削除に至る道筋を持てるようになります。

それぞれの意味

trepro-status: deprecated
現在のLifecycle状態を機械可読に示すフィールドです。Skill検索や昇格判定が状態前提で回せるようになります。
trepro-replaced-by
代替Skillの名前を明示するフィールドです。利用者が移行先を迷わずに見つけられる導線になります。
trepro-removal-date
実削除予定日を明示するフィールドです。利用部門が移行を計画できる猶予期間として機能します。
metadata:
  trepro-status: deprecated
  trepro-replaced-by: customer-brief-v2
  trepro-removal-date: "2026-09-30"

利用0確認後に削除。
