第5章

『いい感じに』で頼んで、いい感じに失敗する 強い指示の書き方

『ちゃんと指示したのに、ぜんぜん違うものが返ってきた』。 たぶん99%、指示の書き方の問題です。

ほとんどの人は、AIへの依頼を1〜2行で済ませてます。 『〇〇についてまとめて』『××してください』 これだと、AIは『あなたが何を期待しているか』を毎回ゼロから推測することになる。

強い指示には型があります。目的・背景・入力・出力・制約・完了条件、6つ。 これを揃えるだけで、返ってくる答えの精度が一段変わる。

この章の最後、自分の仕事1つを、この型で書き直してください。 たぶん『あれ、AI怒涛に賢くなった』と感じる瞬間が来ます。

実装リファレンス

5.1 魔法のPromptより仕事の定義

なぜここを読むか 魔法のPromptを探す方向に行くと底なし沼にハマる。本当に効くのは仕事自体の定義の精度

やる／やらないとどうなるか Goal・Context・Constraints・Deliverables・Done when・Evidenceの6軸で仕事を定義できる。指示は短くても通る

これは何か 強い指示の6軸。Goal・Context・Constraints・Deliverables・Done when・Evidenceで仕事自体を定義する

実行するとどうなるか AIに何を判断させ、何を判断させないかが分かれる。指示の精度がPromptテクニックではなく定義で決まる

それぞれの意味

Goal
達成したい状態の一文。これが曖昧だと他5軸が全部推測になり、出力が毎回違うものになる
Context
AIが先に読むべきファイル・既存挙動・利用者などの背景。前提を渡さないとAIは想像で穴埋めする
Constraints
守ること・してはいけないことの一覧。AIの善意による暴走（外部API追加・入力改変等）を仕様で縛る
Deliverables
成果物として何を作るかの宣言。ファイル名やArtifact種別を明示し、納品形が毎回変わるのを防ぐ
Done when
完了の判定条件。テスト・型・画面確認など機械的に確かめられる基準で書き、感覚での合格判定を排除する
Evidence
完了の証拠として出させるもの。変更File・コマンド出力・画面など、後追い確認に使う物的証跡を指定する
Goal       何を達成するか
Context    何を読めばよいか
Constraints 守ること・しないこと
Deliverables 何を作るか
Done when どう判定するか
Evidence   何を証拠として出すか

5.2 汎用Template

なぜここを読むか 毎回ゼロから書くと、書き手によって粒度がバラバラになる。Templateで型を固定する必要がある

やる／やらないとどうなるか 6項目の汎用Templateが手元に残る。曖昧な口頭依頼でも、テンプレに流し込むだけで構造化された依頼文に変わる

これは何か 汎用依頼Template。Goal・Context・Constraints・Deliverables・Done when・EvidenceのMarkdownひな形

実行するとどうなるか 曖昧な依頼を機械的にこのテンプレへ流し込める。書き手による粒度差が消える

それぞれの意味

Goal
達成したい状態を1行で書く欄。曖昧な動詞ではなく『〇〇できる状態』と結果で書くと判定可能になる
Context
Read・Existing behavior・Usersの3要素で背景を渡す欄。AIが現状を推測せず事実から始められる
Constraints
Must/Must not/Out of scopeの3区分で縛る欄。やる/やらない/対象外を分けて書き、暴走の余地を消す
Deliverables
成果物のファイル名やArtifact種別を並べる欄。何を納品物として扱うかを先に固定する
Done when
machine-verifiable checksを並べる欄。テスト通過・型通過などコマンドで判定できる条件だけ書く
Evidence
完了報告に添えさせる証拠の列挙。変更file・コマンド・exit code・画面・残riskを返させる
# Goal
[達成したい状態]

# Context
- Read: [files]
- Existing behavior: [現状]
- Users: [対象]

# Constraints
- Must: [必須]
- Must not: [禁止]
- Out of scope: [対象外]

# Deliverables
- [file/artifact]

# Done when
- [machine-verifiable checks]

# Evidence
- changed files
- commands and exit codes
- screenshots/output samples
- remaining risks

やる／やらないとどうなるか 利用者・必要機能・対象外・完了条件まで具体化された依頼文に書き直せる。一発で意図に近い出力が返る

5.4 Claudeに質問させる

なぜここを読むか 未知の要件があるのにAIが沈黙のまま実装に入ると、勝手な仮定で暴走する。実装前に質問させる仕組みが要る

やる／やらないとどうなるか 最大5問・選択肢提示・工数Risk説明という制約付き質問が返る。仮定の暴走を実装前に止められる

これは何か Claudeに質問させるPrompt。最大5問・選択肢提示・各選択肢の工数Risk説明という制約付き

実行するとどうなるか 未知要件を勝手に仮定して暴走しない。質問への回答が次の依頼の素材になり、仕様が固まる

実装前に、不足情報を最大5問まで質問してください。
選択肢を提示し、各選択肢が工数・Riskへ与える影響を短く説明してください。

質問に答えられない場合は、仮定を明示して最小版を選ぶ。

5.5 完了報告の型

なぜここを読むか 『完了しました』だけを受け取ると、何が変わって何が確認済みか分からない。後で別の問題で揺り戻しが来る

やる／やらないとどうなるか 結果・変更File・Check結果・画面・残Risk・次Actionの6区分で報告が返る。確認可能な証拠付きで完了判定できる

これは何か 完了報告の型。結果・変更File・Check結果・画面/Artifact・残Risk・次Actionの6区分で報告させる

実行するとどうなるか 『完了しました』が確認可能な証拠付き報告に変わる。残Riskと次Actionも同じ報告で見える

それぞれの意味

結果
依頼が達成されたかの一文。Yes/Noで答えさせ、『だいたい動いた』のような曖昧表現を排除する
変更File
今回触ったファイル一覧。後追いレビュー時にDiffを見るべき箇所を即特定できる
実行したCheckとExit Status
走らせたテスト・lint・typecheckとその終了コード。緑なら本当に通ったことの物的証拠になる
画面・Artifact
出力されたHTMLやスクリーンショット等の証拠物。文字だけでは伝わらない動作確認を視覚で残す
残Risk/未対応
今回潰しきれなかった問題と未着手項目。完了報告に隠さず書かせて、次の依頼の素材にする
次の推奨Action
AIから見た次の一手。残Riskを踏まえた優先度付き提案で、人が次に何をすべきか即決できる
1. 結果
2. 変更File
3. 実行したCheckとExit Status
4. 画面・Artifact
5. 残Risk/未対応
6. 次の推奨Action

「完了しました」だけを受け入れない。
