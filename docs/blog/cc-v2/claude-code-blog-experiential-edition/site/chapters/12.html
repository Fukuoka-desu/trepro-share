第12章

うまくいった一回を、毎回呼べる形にする Skills設計

『うまくいった一回のプロンプトを、また呼び出せない』。 これは記憶力の問題ではなく、Skill 化していないだけです。

ほとんどの人は、成功した手順を頭の中、あるいは古いチャット履歴にしまっています。 翌週、似た仕事が来たときに、同じ品質では再現できません。

Skill は、成功したやり方を再利用可能な部品にまとめる仕組みです。名前・前提・手順・成果物・検証の5点を1ファイルへ閉じ込め、呼ぶたびに同じ品質で走らせます。

この章を読み終わる頃には、自分の仕事の中から最も使い回す3つを、Skill 化する目で見られるようになります。

この章の概観 ／ 何を扱い、何ができるか

この章で扱う機能と、それで何ができるようになるか。実装に入る前に、全体像をここでつかみます。

  • Skill（再現可能な業務パッケージ）

    名前・条件・入力・処理・出力・品質基準・参照資料を持つ、必要時だけ読み込まれる再利用単位。

    入力
    成功した業務手順、入力例、合格基準
    出力
    次回から呼び出せるSKILL.mdと参照資料

  • Custom Commands と Canonical Name

    SkillsへCommandsを統合する正本命名規約。/コマンドで呼び出せる名前と所有者を定義する。

    入力
    Skill名候補、所有者、利用シーン
    出力
    lowercase-hyphenのCanonical Nameと所有Owner

  • Progressive Disclosure（references分離）

    長文・コード例・FAQはreferencesへ逃がし、SKILL.md本体を短く保つ設計原則。

    入力
    Skill関連の長文素材、コード例
    出力
    短いSKILL.mdとreferences/配下の補助資料

  • 4軸評価（明確性/実用性/保守性/整合性）

    Skillを4観点で採点し、合格点未満は本番配布しない品質ゲート。Generator/Evaluator分離の前段で使う。

    入力
    完成したSkill、評価Fixture
    出力
    4軸スコアと改善優先度のレポート

実装リファレンス

12.1 Skillとは

なぜここを読むか Skillとは何かの定義が共有されていないと、単なるプロンプト断片と再利用可能な業務Packageの区別がつきません。CLAUDE.mdとの責務分担も曖昧になります。

やる／やらないとどうなるか 「知識・手順・出力契約・Reference・Script・Fixtureをまとめた業務Manual」と定義を揃えれば、何をSkill化すべきかが判定可能になります。定義無しではSkill乱立か死蔵かの両極端になります。

Skillは、特定Taskを実行するための知識、手順、出力契約、Reference、Script、Fixtureをまとめた再利用Packageである。

Claude.mdが「この職場の基本方針」なら、Skillは「業務Manual」である。ただし、SkillもPrompt資産であり、Security強制はHook/Permissionへ置く。

12.2 Custom Commands

なぜここを読むか 旧来の`.claude/commands/`は互換目的で動きますが、新規もそこへ書き続けるとSkillsとCommandsの二系統が並走し、所有者と更新ルートが分裂します。

やる／やらないとどうなるか Inventory化→Skill移行→互換Wrapper残し→90日後Deprecate、の順で揃えれば呼び出し口を一本化できます。両建てを続けるとどちらを直せば効くか分からない混乱が長期化します。

旧 .claude/commands/ は互換目的で動く場合があるが、新規はSkillsへ統合する。Commands 13個を持つ現行Harnessは、次の順で移行する。

  1. 呼び出し名と利用実績をInventory化。
  2. 業務手順はSkillへ移す。
  3. UI操作だけの短いものは互換Wrapperを残す。
  4. 90日後に旧CommandをDeprecate。

12.3 Canonical Name

なぜここを読むか Skill名がlowercase-hyphen・日本語・スネークケース等で混在すると、Claude Code/Codex共通発動や検索で取りこぼしが起きます。命名規約を明文化する必要があります。

やる／やらないとどうなるか lowercase letters・digits・hyphensのみ、ディレクトリ名=frontmatter nameと統一すれば、どのエージェントからも同じ名前で確実に呼べます。Legacy名はMetadataへ残し移行を追跡可能にします。

- lowercase letters

12.4 標準Template

なぜここを読むか Skillごとに見出し構成が違うと、メンテナンスのたびに読み解き直す手間が発生します。標準Templateを持っておくと書き手も評価者も迷いません。

やる／やらないとどうなるか Objective・Inputs・Preconditions・Workflow・Output contract・Safety・Quality checks・Failure behavior・Referencesを揃えれば、書き漏れと評価漏れが構造的に防げます。

これは何か 標準SKILL.md Templateの全体像です。Frontmatterから9セクションまでの構造を最小例で示します。

12.5 Progressive Disclosure

なぜここを読むか Skill本体に長いAPI仕様・例・FAQまで詰めると、毎回の読み込みが重くなり、肝心のWorkflowが埋もれます。Progressive Disclosureで段階開示する設計が必須です。

やる／やらないとどうなるか SKILL.mdは発動条件と主要Workflowに絞り、長文はreferences/、決定的処理はscripts/、入出力例はfixtures/へ分ければ、本体200行以内を維持できます。

これは何か Progressive Disclosureのファイル分割ガイドです。SKILL.md本体と補助ディレクトリの役割分担を示します。

実行するとどうなるか このレイアウトに揃えれば、本体は短く保たれ、長文資料は必要なときだけ読まれます。Skillの読み込みコストと保守コストが下がります。

SKILL.md: 発動条件、主要Workflow、契約、安全、参照条件
references/: 長いAPI仕様、例、部門Rule
scripts/: 決定的処理
fixtures/: 入力と期待結果
tests/: 回帰Test

Root SKILL.mdは200行以内を目標にする。

12.6 Skillを作る最良の手順

なぜここを読むか 会話で一度うまくいったからといってそのままSkill化すると、別の入力で破綻したり固有情報を抱えたまま配布したりします。Skill化には正しい順序があります。

やる／やらないとどうなるか 手動完走→記録→固有情報除去→抽象化→Fixture化→Fresh Evaluator→Canary→改善、の8段を踏めば再現性のあるSkillになります。「今のをそのままSkillに」だけでは全社配布水準に到達しません。

これは何か Skillを作る8段階の手順です。手動完走から利用実績による改善までの順番を示します。

実行するとどうなるか この順を踏めば、固有情報の混入や再現性不足を避けながら配布水準のSkillへ到達できます。会話のメモ取りだけでSkill化する事故を防げます。

1. 手動でTaskを完走
2. 成功/失敗を記録
3. 固有情報を除去
4. 入力/出力/停止条件を抽象化
5. Fixture化
6. Fresh Evaluator
7. Canary配布
8. 利用実績で改善

会話中に「今の内容をそのままSkillにして」だけで全社配布しない。

12.7 4軸評価

なぜここを読むか Skillの出来不出来を主観で判定すると、配布後に「動かない」「曖昧」と苦情が出ます。4軸の機械的な採点ゲートが必要です。

やる／やらないとどうなるか 明確性・実用性・保守性・整合性で総合85点以上、各軸18点以上、Critical 0件を合格条件にすれば、配布前に品質が担保されます。基準が無いと品質ドリフトが進みます。

軸 観点
明確性 発動、入力、出力、禁止が明確
実用性 現実Dataで完走、失敗を扱える
保守性 短い正本、Reference、Test、Owner
整合性 全社Rule、他Skill、標準と矛盾なし

合格は総合85点以上、各軸18点以上、Critical 0件とする。

12.8 呼び出し

なぜここを読むか Skillには明示呼び出しと自然言語発動の2方式があり、Descriptionの書き方で発動精度が変わります。重要業務ほど明示呼び出しが安全です。

やる／やらないとどうなるか 「何をするか」だけでなく「いつ使うか」をDescriptionへ書けば、自然言語発動の的中率が上がります。明示コマンドを併用すれば重要業務は確実に発動します。

明示呼び出しと自然言語発動の両方がある。重要業務は明示呼び出しを推奨する。

/meeting-minutes inputs/meeting.md

Descriptionは「何をするか」だけでなく「いつ使うか」を書く。

補助図の制作指示

こんな場面で使う

章の内容が実務でどう活きるか。架空の場面・行動・効果の三点で示します。

場面

XYZ広告のプランナー（架空）が、毎週類似のレポート依頼に対して毎回プロンプトを書き直し、品質も毎回ぶれている。

こう使う

成功した会話を入力・処理・出力・品質基準・失敗時対応の形式へ整理し、Skill化する。

得られるもの

次週からは『あのSkillで』の一言で同品質の成果物が出る。教育コストもSkill共有で済む。

場面

ある制作チーム（架空）が、複数メンバーで同じSkillを書き散らし、似て非なるSkillが乱立して使い分けに迷っている。

こう使う

正本名・Owner・参照資料を持つSKILL.mdテンプレートを揃え、referencesに長文を逃がすProgressive Disclosureを徹底する。

得られるもの

Skillの重複が抑えられ、新規メンバーも『どれを使えばよいか』を迷わず選べる。
