第16章

正しいだけで、読まれない資料を卒業する MarkdownよりHTMLが向く仕事

『正しいことを長く書いたのに、読まれない』。 これは内容ではなく、出力形式の問題です。

ほとんどの人は、AIに『レポートを書いて』と頼んで、長文の Markdown を受け取って終わりにしています。 ですが、受け取った相手は、それを読み切る時間も読む構造も持っていません。

HTML は、見出し・図・色・前後関係を視覚で伝えられる出力形式です。 同じ情報を 1 枚に整え、結論を上に、根拠を下に、補足は折りたたみへ。読み手の意思決定速度が変わります。

この章を読み終わる頃には、『長く書く』ではなく『短く伝える』の方向で、AIに頼めるようになります。

この章の概観 ／ 何を扱い、何ができるか

この章で扱う機能と、それで何ができるようになるか。実装に入る前に、全体像をここでつかみます。

  • 形式選択（Markdown vs 単一HTML）

    執筆・差分管理が強いMarkdownと、視認性・操作性が強いHTMLを目的別に使い分ける判定基準。

    入力
    成果物の用途、読み手、共有経路
    出力
    形式選択の根拠と最終出力フォーマット

  • 単一HTML標準（自己完結レポート）

    外部CSS/JSに依存せず、1ファイルだけで開ける標準フォーマット。共有・印刷・スマホ表示も担保する。

    入力
    Markdown原稿、データ、画像
    出力
    外部依存なし1ファイルHTMLレポート

  • CSV分析Prompt

    CSVを読み込んで集計結果と図解を含むHTMLにする標準プロンプト。経営共有しやすい1枚物に変換する。

    入力
    CSVデータ、注目したい軸
    出力
    集計表・グラフ・要点カード入りHTML

  • HTML Slide / Discussion Board

    Markdownから1ページHTMLでスライド風や議論用ボードを作る派生形式。同じ正本から複数表現を出す。

    入力
    発表用骨子または議題リスト
    出力
    HTML Slide、Discussion Board

  • 品質Check（A11y/印刷/スマホ）

    公開前にアクセシビリティ・印刷・スマホ表示・装飾依存度を確認する標準チェック項目。

    入力
    完成したHTML
    出力
    A11y/印刷/スマホでの動作確認結果

実装リファレンス

16.1 形式選択

なぜここを読むか Markdownで何でも済ませようとすると、長文の正しさは保てても読み手の理解速度が出ません。

やる／やらないとどうなるか Markdown・HTML・PPT/PDF・Web Appの強みを並べて見ると、用途ごとに最短で伝わる形式を選べるようになります。

形式 強み 用途
Markdown 軽い、Diff、再利用 仕様、議事録、Runbook
HTML Graph、Filter、操作、情報密度 Report、Dashboard、教材
PPT/PDF 配布・発表互換 正式発表、顧客提出
Web App Data更新、入力、状態 継続業務Tool

原則は、正本をMarkdown/JSON/CSVで持ち、HTMLをViewとして生成する。

16.2 単一HTMLの標準

なぜここを読むか 外部CDN依存のHTMLは、共有先のネットワークやオフライン環境で簡単に崩れます。

やる／やらないとどうなるか 1ファイル完結・印刷・スマホ・JS無効でも主要情報が残る標準を持てば、配布経路を選ばずに同じ品質で届けられます。

これは何か 単一HTMLレポートが満たすべき8項目の標準で、配布性とアクセシビリティを並列で要求しています。

実行するとどうなるか 出力先の環境差に依存せず、同じHTMLがメール・印刷・スマホで同じように読める状態を作れます。

- 1 Fileで開ける
- 外部CDNに依存しない
- Mobile対応
- Print CSS
- 見出し/Keyboard/Contrast
- 更新日・出典・母数
- GraphとTableを併記
- JS無効でも主要情報が読める

16.3 CSV分析Prompt

なぜここを読むか CSVを渡して『集計して』と頼むだけでは、母数や定義が抜けた『それっぽい数字』が返りがちです。

やる／やらないとどうなるか 観点・出力・ルールを明示したテンプレを使えば、経営会議に出せる粒度のHTMLレポートに揃えられます。

これは何か CSVから経営会議向けHTMLを生成するプロンプトで、分析観点・出力ファイル・ルールを分けて指示するテンプレです。

実行するとどうなるか 母数・推測の明示・匿名化までを含んだ、そのまま会議に持ち込める粒度の1枚物が得られます。

`data/survey.csv` を分析し、経営会議向け単一HTMLを作る。

Analysis:
- 回答数、欠損率、KPI
- 設問分布
- 自由記述Theme
- Positive/Negative要因
- 優先度付き改善案

Output:
- output/report.html
- output/analysis.md
- output/metrics.json

Rules:
- 元Dataを改変しない
- 推測は明記
- 個人特定を匿名化
- 母数と計算式を表示
- JavaScript Errorなし
- Screenshotを保存

16.4 HTML Slide

なぜここを読むか 発表用と検討用で資料を作り分けると、同じ正本から二重管理が始まり、必ずどこかでズレます。

やる／やらないとどうなるか 1ページHTMLでSlide風やDiscussion Boardを派生させると、正本を保ったまま複数の表現を素早く出せます。

1. 表紙
2. 結論
3. 背景
4. 根拠Data
5. 選択肢
6. 推奨
7. 実行計画
8. Risk
9. 意思決定

これは何か Markdown原稿を1スライド1メッセージのHTMLスライドへ変換するための指示文です。

実行するとどうなるか 発表用と検討用を別資料化せず、同じ正本から発表向け表現を派生できます。

`output/analysis.md`を16:9 HTML Slideにしてください。
1 Slide 1 Message、本文最大6行。矢印Key/PageUp/PageDownで移動し、Print/PDF対応にしてください。

16.5 Discussion Board

なぜここを読むか 会議で出した付箋やToDoは、写真に撮っただけだと検索もコピーもできず後から効きません。

やる／やらないとどうなるか 列構造とLocal Storage保存・Markdown書き出しを備えたBoardにすると、議論結果を次の作業に直結させられます。

これは何か 今やる・次・調査・見送りの4列で構成するDiscussion BoardのHTML生成プロンプトです。

実行するとどうなるか 会議中の議論結果が、後から検索・コピー・Markdown化できる形でそのまま残ります。

会議用の単一HTML Boardを作る。
列は「今やる」「次」「調査」「見送り」。
CardはDragでき、担当、期限、根拠、決定者を編集可能。
Local Storageへ保存し、「決定事項をMarkdownでCopy」Buttonを付ける。

Local StorageへSecrets、原文、不要な個人情報を保存しない。

16.6 品質Check

なぜここを読むか 見栄えだけ整えて公開すると、印刷・スマホ・アクセシビリティのどこかで読めない読者を取りこぼします。

やる／やらないとどうなるか 8項目の品質チェックを通すことで、公開前に『誰でも開けて読める』状態をルーチンで担保できます。

これは何か 公開前の品質チェック8項目で、論理構造から技術的健全性まで一気に確認するリストです。

実行するとどうなるか 色覚・印刷・モバイル・Console Errorといった見落としを、公開前に潰し切れます。

[ ] 結論が最初にある
[ ] 元Dataへ追跡可能
[ ] 色だけで状態区別しない
[ ] 375/768/1440px確認
[ ] Keyboard操作
[ ] Print/PDF欠落なし
[ ] 外部Scriptなし
[ ] Console Errorなし

補助図の制作指示

こんな場面で使う

章の内容が実務でどう活きるか。架空の場面・行動・効果の三点で示します。

場面

ある経営企画（架空）が、毎月の予実報告をMarkdownで提出するが、役員が目を通さず会議で同じ説明を繰り返している。

こう使う

同内容を単一HTMLレポートへ変換し、結論カード・グラフ・強調点を1ページに集約する。

得られるもの

結論到達時間が短くなり、会議の冒頭で『見ました』前提で議論が始まる。

場面

中堅メーカーの開発者（架空）が、テストレポートをスクリーンショットで貼り、後日参照しても何が壊れていたか追えない。

こう使う

テスト結果を単一HTMLで保存し、印刷・スマホ・Alt textで読める形式に揃える。

得られるもの

障害後の振り返り会議で誰でも開けて確認でき、Markdown原本との二重管理にしなくて済む。
