第11章

同じ説明を、二度と書かない CLAUDE.md、Rules、Auto Memory

『うちのプロジェクト、AIに毎回ゼロから説明している』。 これは時間の問題だけでなく、再現性の問題でもあります。

ほとんどの人は、AIに毎回同じ前提を打ち込んでいます。 TypeScript を使っている、コメントは英語で書いている、テストは vitest で実行する──毎日10回、同じ説明をしている計算です。

CLAUDE.md は、プロジェクトの前提を一か所にまとめておくファイルです。AI は作業を始める前に必ずそこを読みます。 これだけで、毎回の前置きが消えます。

この章を読み終わる頃には、自分のプロジェクトに『専属の同僚』を一人配置している感覚を得られます。

この章の概観 ／ 何を扱い、何ができるか

この章で扱う機能と、それで何ができるようになるか。実装に入る前に、全体像をここでつかみます。

  • CLAUDE.md（プロジェクトの案内板）

    常時読まれる短い案内板。プロジェクトの目的・前提・最低限のルールだけを簡潔に置く。

    入力
    プロジェクトの目的、必ず守らせたい最低限ルール
    出力
    簡潔で肥大化しないCLAUDE.md

  • Rules（Path固有の指示）

    ディレクトリ単位で適用されるルール。CLAUDE.mdを膨らませず、効く場所だけに知識を置く仕組み。

    入力
    特定フォルダ・拡張子に限定したいルール
    出力
    対象Path下でだけ自動適用されるRule

  • /init による初稿生成

    プロジェクトを読み込みCLAUDE.mdの叩き台を作るコマンド。生成内容をそのまま使わず短く整える前提で使う。

    入力
    既存プロジェクト構造、目的の概略
    出力
    CLAUDE.md初稿（編集して短くする出発点）

  • Auto Memory（短い事実の自動保存）

    AIが見つけた短い事実だけを自動保存する記憶層。常時読み込み情報を短く安定させる役割を担う。

    入力
    会話中の決定事項、繰り返される好み
    出力
    短い形に圧縮された自動メモ

実装リファレンス

11.1 責務

なぜここを読むか CLAUDE.md・Rules・Memory・Settings・Hookは似て見えますが、書く人・効く範囲・強制力が違います。役割表で整理しないと、同じ知識を別の場所に重複して書く事故が起きます。

やる／やらないとどうなるか 誰が何を書きどこで効くかを最初に決めれば、ルール衝突や二重管理が消えます。曖昧なまま運用すると、強制したい設定が単なるお願い止まりになり安全装置として機能しません。

Layer 誰が書く 何を書く 強制力
CLAUDE.md 人間・Team Project概要、Command、方針 Guidance
Rules 人間・Team Pathや領域別の指示 Guidance
Auto Memory Claude Build、Debug、習慣の短い知見 Guidance
Managed Settings 管理者 Permission/Sandbox/MCP等 Enforced
Hook 管理者/開発者 Eventに対する決定的処理 Enforced

11.2 CLAUDE.mdのScope

なぜここを読むか CLAUDE.mdは管理者・User・Project・Localの4階層で読まれるため、どこに置くかで適用範囲と上書き順が決まります。Scopeを意識しないと組織ルールと個人設定が衝突します。

やる／やらないとどうなるか 適切なScopeへ置けば全社必須・個人好みを物理的に分離できます。誤って個人設定をProject層に置くと、他メンバーの環境まで巻き込んで挙動が変わってしまいます。

代表的なScope：

  • Managed policy: 組織共通、ユーザーが除外できない。
  • User: ~/.claude/CLAUDE.md。
  • Project: Repository RootのCLAUDE.md。
  • Local: Git管理しない個人差分。

全社の外せない指示はmacOSでは管理領域へ置く。

これは何か macOSの管理領域に置く全社共通CLAUDE.mdのPathです。ユーザーが除外できないManaged Policy層になります。

実行するとどうなるか ここへ配置すると全社員のClaude Code起動時に強制読み込みされ、組織共通の前提を全環境へ一斉適用できます。

/Library/Application Support/ClaudeCode/CLAUDE.md

ただし、ここもBehavior Guidanceであり、Tool実行の強制制御はSettings/Hookで行う。

11.3 良いCLAUDE.md

なぜここを読むか CLAUDE.mdは毎回読まれる常駐情報のため、長くするほど起動が重く、本当に伝えたい指示が埋もれます。良い形の雛形を最初に決めておくのが安全です。

やる／やらないとどうなるか Project概要・Architecture・Commands・Workflow・Constraints・Do notの6項目に絞れば、200行以内の短さを保てます。長文を許すと数か月で3,000行へ膨れ事故の温床になります。

これは何か 200行以内に収めるための良いCLAUDE.mdの骨格です。6見出しで責務を分けたMarkdown雛形になります。

実行するとどうなるか この構造を出発点にすれば、長文化を防ぎつつ必要な前提が網羅されます。逆にこの枠を超え始めたら別ファイル分割の合図です。

# Project overview
[目的と主要User]

# Architecture
[主要Directoryと責務]

# Commands
- install:
- dev:
- lint:
- typecheck:
- test:
- build:

# Required workflow
[調査→計画→実装→検証]

# Constraints
[Technology、Data、Security]

# Do not
[Config弱体化、本番接続、生成物直接編集]

短く、具体的、重複なし。200行未満を目安にする。

11.4 /init

なぜここを読むか `/init`は既存Projectを読み込んで初稿を生成してくれますが、出力は冗長で当たり前のことを並べがちです。そのまま採用するとCLAUDE.mdが最初から肥大化します。

やる／やらないとどうなるか Reviewして実Commandと一致しない記述・古いFile説明・Settings/Hookへ移すべき強制ルールを整理すれば、起点として安全に使えます。Reviewなしで採用すると初日から負債になります。

Projectを読み、初期CLAUDE.mdを作る起点にできる。ただし生成物をそのまま採用しない。

Review項目：

  • 実際のCommandと一致。
  • 古いFile説明がない。
  • 当たり前の文章を削除。
  • Security RuleをSettings/Hookへ移す。
  • 長い手順をSkillへ移す。

11.5 Rules

なぜここを読むか Path別の細かい指示までCLAUDE.mdへ書くと、本来関係ないファイルを編集するときにも全ルールが読み込まれ、雑音が増えます。領域別Ruleで効く場所を絞るのが筋です。

やる／やらないとどうなるか `.claude/rules/`配下にfrontend.md・api.md等を分け、Frontmatterで対象Pathを限定すれば、必要なときだけ該当Ruleが効きます。混ぜ書きすると効かせたい場所で見落とされます。

.claude/rules/へ領域別Ruleを分ける。

これは何か 領域別Ruleを置くディレクトリ構造です。frontend・api・database・testsなど扱う範囲ごとに分けます。

実行するとどうなるか このレイアウトに揃えると、関連ファイルを編集する文脈でだけ該当Ruleが効くようになります。CLAUDE.md本体を肥大化させずに済みます。

.claude/rules/
├── frontend.md
├── api.md
├── database.md
└── tests.md

Path条件を使えるVersionではFrontmatterで対象を限定する。

これは何か RuleのFrontmatterでPath条件を指定する例です。対象Globに一致する編集時だけRuleを適用させます。

実行するとどうなるか このFrontmatterを書けば`src/api/`配下を編集するときだけ「外部入力検証」「型付きエラー」等の指示が読み込まれ、無関係な領域では雑音になりません。

---
paths:
  - "src/api/**/*.ts"
---

- Validate all external input.
- Return typed errors.
- Do not log request bodies containing personal data.

11.6 AGENTS.mdとの共存

なぜここを読むか Claude CodeとCodexを併用する組織では、AGENTS.mdとCLAUDE.mdが重複しがちです。同じ文章を二重で編集すると、片方だけ更新される事故が必ず起きます。

やる／やらないとどうなるか 共通原則はAGENTS.md、Claude固有だけCLAUDE.mdに分ければ、更新ポイントが一本化されます。役割分担を決めずに併用すると、エージェント間で挙動が割れて再現性が失われます。

Claude CodeとCodexを併用する場合：

  • 共通原則はAGENTS.md。
  • Claude固有はCLAUDE.md。
  • 同じ文章を二重編集しない。
  • Build Scriptで共通Sectionを生成するか、Canonical Docを参照する。

11.7 Auto Memory

なぜここを読むか Auto MemoryはClaudeが学んだ事実を勝手に蓄える便利な機能ですが、放置すると古い情報・秘密値・端末固有情報が混ざります。常時読み込まれるだけに汚染リスクが大きいです。

やる／やらないとどうなるか `/memory`を定期確認して棚卸しすれば、品質を保てます。Teamで共有すべき知見はCLAUDE.md/Skillへ昇格、不要な短期情報は削除、というルールを決めないと信頼できない記憶になります。

ClaudeがBuild Command、Debug Insight、Architecture Note等を保存する。便利だが、次を定期確認する。

これは何か Auto Memoryの中身を確認するコマンドです。Claudeが自動保存した内容を一覧表示できます。

実行するとどうなるか 実行すると現在の自動メモが見え、古い情報・秘密値・端末固有値の混入を点検できます。定期実行しないとMemoryがゴミ箱化します。

/memory
  • 古い情報を削除。
  • 秘密・顧客情報を保存しない。
  • Teamで共有すべき知見はCLAUDE.md/Rule/Skillへ昇格。
  • 端末固有の一時情報を共通Ruleへ入れない。

11.8 改善Loop

なぜここを読むか 会話中に出た修正を毎回どこへ昇格させるか迷うと、知識が散らかります。再発頻度と影響範囲で行き先を決めるルートを持っておくのが実務的です。

やる／やらないとどうなるか 一時的→Sessionのみ、個人習慣→Memory、Project規約→CLAUDE.md/Rules、業務手順→Skill、必須制御→Hookと振り分ければ、知識が適切な階層に積み上がります。ルートが無いと毎回その場限りで終わります。

会話中の修正
  ↓
再発するか評価
  ↓
一時的 → Sessionのみ
個人習慣 → Auto Memory/User CLAUDE.md
Project規約 → CLAUDE.md/Rules
業務手順 → Skill
必須制御 → Hook/Managed Settings
