1枚に詰め込む、を諦める Design・資料・公開

『1 枚に詰め込み過ぎて、結局誰にも伝わらない』。 これは情報量の問題ではなく、設計の問題です。

ほとんどの人は、資料を作るときに『漏れなく書く』ことを優先しています。 ですが、伝わる資料は、書いた情報の量ではなく、捨てた情報の量で決まります。

この章で扱うのは、Design・資料・公開の3つを、AIと一緒に整える流れです。 1ページに収まる粒度を決め、補足はリンク先へ逃がし、公開先の媒体に合わせて翻訳します。

1 枚に詰め込む、を諦める。それが、読まれる資料の出発点です。

実装リファレンス

31.1 LP

なぜここを読むか LPは見た目だけ作って公開すると、Analytics・A11y・本番API差し替えが抜けがちです。Audience/Action→Inventory→Wireframe→Code→Previewと工程を分けると、各段で抜けを検知できます。

やる／やらないとどうなるか Designと公開を別工程として扱う運用が定着し、見せたい人に届くLPを再現性ある手順で出せます。

これは何か LP制作の標準パイプで、Analytics/SEO/A11yチェックをPreview直前に置き、本番Productionの前に独立Gateを設ける構成です。

実行するとどうなるか デザイナーが入れ替わっても工程の抜けが起きにくく、Audienceから出発した一貫性のあるLPが安定して出力できます。

それぞれの意味

Audience/Action
誰に何をさせたいLPかを最初に決める段で、これが曖昧なまま走ると全要素が中途半端になります。CV設計の土台です。
Content Inventory
掲載予定の素材・主張・証拠を棚卸しする段で、後から『この情報を入れ忘れた』『重複した』を防ぎます。
Wireframe
色やコピー前に骨格だけ確定する段で、構成議論とビジュアル議論を分離して手戻りを減らします。
Copy Review
コピー文言だけを独立に磨く段で、見た目に引きずられず言葉の説得力を評価できます。
Prototype
クリック可能な試作を作る段で、関係者が触って初めて気付く違和感を早期に拾えます。
Design System
色・余白・タイポグラフィをTokenで参照する段で、後続のSlide/Reportと一貫性を保ちます。
Code
実装フェーズで、Design Systemに沿って組むことで個別調整によるブランド揺れを抑えます。
Analytics/SEO/A11y
計測・検索性・アクセシビリティをPreview直前に確認する段で、公開後に気付く致命的抜けを潰します。
Preview → Production
本番前にPreview承認Gateを置く段で、生成の延長で即公開する事故を構造的に防ぎます。
Audience/Action → Content Inventory → Wireframe → Copy Review
→ Prototype → Design System → Code → Analytics/SEO/A11y → Preview → Production

31.2 Design System再利用

なぜここを読むか 成果物ごとに別の色や余白を増やすと、ブランドが揺れて顧客に違和感を与えます。LP・営業資料・Dashboard・Slideで共通Tokenを参照する運用にすれば、見た目の整合性が自動で保たれます。

やる／やらないとどうなるか デザイナーが毎案件ゼロから整えなくても、共通基盤に乗せた状態で各成果物が出るためレビュー時間が短縮されます。

営業資料、LP、Dashboard、HTML Report、Slideで共通Token。成果物ごとに勝手な色/余白を増やさない。

31.3 PPT/PDF Export

なぜここを読むか HTMLからPPT/PDFへExportすると、Font・改ページ・図欠落・解像度・選択可能Textで崩れが出ます。事前Checklist化することで、毎回手で気付くか運頼みかの工程を機械的に潰せます。

やる／やらないとどうなるか 提案直前にレイアウトが崩れている事故が減り、Export成果物を本番資料として安心して使えるようになります。

これは何か PPT/PDF Exportの直前に通すChecklistで、Font・改ページ・解像度・Note・機密Metadataまで横並びに確認します。

実行するとどうなるか Export崩れの原因が3観点（フォント置換・ページ分割・余白）に絞られ、毎回数時間の手動修正がなくなります。

それぞれの意味

Font置換
HTML上のフォントがPPT/PDFで別フォントに置換されていないかの確認項目で、ブランド指定字体が崩れる事故を防ぎます。
改ページ
意図しない位置でページが切れていないかの確認項目で、見出しと本文が分断される崩れを潰します。
図欠落
HTMLでは見えていた図がExportで消えていないかの確認項目で、SVG/Canvas由来の非互換崩れを検知します。
Link
リンク文字列がクリック可能なまま残っているかの確認項目で、PDF化で死んでいるリンクをそのまま提出する事故を防ぎます。
Note
発表者ノート等の補足が意図通り出力されているかの確認項目で、説明用メモが消える/外に出る両事故を防ぎます。
解像度
画像が荒くなっていないかの確認項目で、Retina前提素材を低dpiで吐いて提案資料が粗く見える事故を防ぎます。
選択可能Text
本文テキストが選択コピー可能なまま残っているかの確認項目で、画像化されて検索もコピーも不能になる劣化を検知します。
機密Metadata
作成者・コメント・社内パス等のMetadataが残っていないかの確認項目で、社外提出時の情報漏洩事故を防ぎます。

31.4 Deploy

なぜここを読むか Deployを生成の延長として扱うと、Secret混入・公開範囲ミス・Rollback不在の事故が一度に起きます。PreviewをDefault・本番をAsk Ruleにすると、公開という別Gateが明確に確保されます。

やる／やらないとどうなるか Preview承認とSmoke/Monitoring/Rollbackが必ず通る運用になり、即本番公開による事故が体系的に防げます。

これは何か Deploy工程をPreview→Approval→Production→Smoke/Monitoring/Rollbackの順に縛る標準フロー定義です。

実行するとどうなるか Owner承認とRollback前提がDeploy Skillに組み込まれるため、便利さで公開事故が起きる経路を物理的に閉じられます。

それぞれの意味

Preview
本番と同等の表示環境へまず出す段で、実機での見え方を承認前に確認できる状態を作ります。生成の延長で本番直行を防ぎます。
Functional
リンク到達・フォーム送信・主要動線が機能するかを検証する段で、見た目だけ整って動かない事故を潰します。
A11y/Security/Privacy
アクセシビリティ・秘密情報・個人情報の3観点を横並びで確認する段で、公開後に致命傷化する観点を一度に検査します。
Owner Approval
責任者が明示的に公開可否を判断する段で、Deploy Skillに承認なしでは進めない物理ガードを組み込みます。
Production
承認後にだけ本番へ反映する段で、Preview/本番の境界を曖昧にしない運用を担保します。
Smoke/Monitoring/Rollback
本番直後の動作確認・継続監視・即時切り戻し手順の3点セットで、事故時に最短で元へ戻せる経路を必ず確保します。
Preview → Functional → A11y/Security/Privacy → Owner Approval
→ Production → Smoke/Monitoring/Rollback

Deploy SkillはPreviewをDefault、本番をAsk Ruleにする。

実務に落とすときの補足

公開前に最低限揃えるもの
Preview URL、スマホ表示確認、内部リンクと外部リンクの到達確認、Analytics・APIキー等の秘密情報の有無、Rollback手順の5点を1枚の事前チェックに固定します。ここを言語化していない組織では、毎回担当者の記憶頼みで公開され、必ずいつか事故が起きます。
Export崩れの直し方の順序
PPT/PDFへExportして崩れた時、HTML側を直すかExport設定を直すかで悩む前に、まず正本はHTMLだと宣言します。崩れの再現箇所を1ページだけ抜き出して原因を見極め、フォント置換・ページ分割・余白の3観点で順に潰すと最短で安定します。
測ると効く指標
公開回数、公開前事故件数、Rollback回数、Export修正に費やした時間を月単位で記録します。事故ゼロを目指すのではなく、Rollback回数が一定範囲内に収まっているかを健全性の指標にすると、過剰な慎重さで止まらなくなります。
次の章への接続
公開Gateの感覚が掴めると、次章の毎日業務の自動化でも『失敗時の戻し方』を最初に決められるようになります。公開とは独立した品質ゲートだという意識が、自動化全体の安全装置として効いてきます。
