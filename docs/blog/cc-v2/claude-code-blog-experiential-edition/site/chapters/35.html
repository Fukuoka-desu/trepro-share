第35章

読む本を、操作するアプリに昇格させる 教科書アプリの情報設計

『教科書を読み終わったのに、自分の進捗が分からない』。 これは読書の限界ではなく、本という形式の限界です。

ほとんどの人は、教科書を『最後まで読んだら終わり』だと考えています。 ですが、教科書は本来、操作・証拠・保存・再開を含む実行アプリへ昇格させるべきものです。

この章で扱うのは、Lesson の JSON 化、進捗状態の保存、章番号での再開を含む、教科書アプリの情報設計です。 学習者は自分の現在地と次の一操作を、いつでも取り出せます。

読む本を、操作するアプリに昇格させる。これは本書全体の到達点でもあります。

実装リファレンス

35.1 最適形

なぜここを読むか 教科書をPDFで配ると、読了率しか測れず、誰が何を実践し、どこで止まったかが分かりません。学びの状態は中身を持つ必要があります。

やる／やらないとどうなるか Role別Path・Lesson進捗・Copy可能Prompt・Terminal Check・Quiz・Harness Statusを備えた実行支援Learning Systemに昇格します。

PDF Viewerではなく、次を持つ実行支援Learning Systemにする。

  • Role別Path。
  • Lesson進捗。
  • Copy可能なPrompt/Code。
  • Terminal Check。
  • Artifact確認。
  • Quiz/Hands-on。
  • Version差分。
  • Harness Status。
  • Skill Catalog/Score。
  • Admin Coverage Dashboard。

35.2 Content Model

なぜここを読むか 本文・状態・証拠・再開を一枚のHTMLに混ぜると、再利用も検証もできなくなります。データとして構造化する必要があります。

やる／やらないとどうなるか course/module/lessonの三段構造に前提・成果物・Check・Sources・change_notesを持たせた、機械が読めるContent ModelのJSONが土台になります。

これは何か 教科書アプリのContent Model例。course/module/lessonの3階層にstatus・level・prerequisites・outcomes・change_notesを持たせています。

実行するとどうなるか 1枚のJSONを起点に、表示UI・進捗保存・修了判定・改訂差分まで同じデータ構造で扱える状態に揃います。

それぞれの意味

course
教材全体の最上位単位で、IDとverson、対象audienceを持ちます。揃えないと改訂時の差分追跡ができません。
module
学習領域でまとめたLessonの束です。順序とテーマを定義しないとLessonが孤立し学習導線が崩れます。
lesson
前提・成果物・Check・所要時間を持つ実習最小単位です。これを構造化しないと進捗保存も修了判定もできません。
verified_at
教材内容の最終確認日です。書かないとどれが古びているか分からず、教材全体の信頼が落ちます。
audiences
非Engineer・開発者・管理者など対象読者の定義です。書かないと全員に同じ難度で出てしまい脱落者が出ます。
change_notes
Lesson単位の変更履歴です。残さないと「いつ何が変わったか」が追えず、現場の混乱を招きます。
{
  "course_id": "claude-code-ultimate",
  "version": "1.0.0",
  "verified_at": "2026-06-22",
  "audiences": ["non-engineer", "developer", "admin"],
  "modules": [
    {
      "module_id": "core-loop",
      "title": "標準作業ループ",
      "order": 2,
      "lessons": [
        {
          "lesson_id": "plan-implement-verify",
          "status": "stable",
          "level": "beginner",
          "estimated_minutes": 25,
          "prerequisites": ["project-folder"],
          "outcomes": ["計画と実装を分ける", "証拠で完了判定する"],
          "steps": [],
          "checks": [],
          "assets": [],
          "sources": [],
          "change_notes": []
        }
      ]
    }
  ]
}

35.3 Lesson JSON Schema

なぜここを読むか Lessonの定義が緩いと、人が手書きで足りない項目を埋め、再開時に検証もできません。形式は機械で検証可能であるべきです。

やる／やらないとどうなるか lesson_id・status・level・outcomes・steps・checks・verified_atを必須にしたJSON Schemaが、不完全なLessonを物理的に拒否します。

これは何か Lesson JSON Schema。lesson_idのパターン・statusのenum・stepsとchecksのオブジェクト構造・verified_at必須を定義しています。

実行するとどうなるか Schemaチェックを通った時点で、不完全なLessonは公開ラインに乗らず、人手の見落としが構造的に減ります。

それぞれの意味

lesson_id
Lesson一意識別子で命名規則を正規表現で固定します。揃えないと参照や進捗保存がIDの揺れで壊れます。
status
stable/beta/preview/suspended/deprecatedの状態enumです。書かないと公開可否や利用警告の出し分けができません。
level
beginner〜adminの難度区分です。書かないと初心者にいきなり管理者操作が出るなど誤配が起きます。
outcomes
Lessonで到達すべき学習成果の配列です。書かないと完了判定の基準が消え、verifiedの意味も曖昧になります。
steps
explain/command/prompt/edit/verify/quizなどの実行単位です。型付けしないとUI側で表示と操作を切り替えられません。
checks
manual/command/file/quiz/reviewなど証拠提示方法の配列です。書かないと「やった」を客観的に示せません。
verified_at
Lesson内容の最終確認日（日付形式必須）です。欠落させると古い手順が無自覚に公開され続けます。
{
  "$schema": "https://json-schema.org/draft/2020-12/schema",
  "$id": "https://trepro.example/schemas/lesson.schema.json",
  "type": "object",
  "required": ["lesson_id", "title", "status", "level", "outcomes", "steps", "checks", "verified_at"],
  "properties": {
    "lesson_id": {"type": "string", "pattern": "^[a-z0-9]+(?:-[a-z0-9]+)*$"},
    "title": {"type": "string", "minLength": 1},
    "status": {"enum": ["stable", "beta", "preview", "suspended", "deprecated"]},
    "level": {"enum": ["beginner", "intermediate", "advanced", "admin"]},
    "estimated_minutes": {"type": "integer", "minimum": 1},
    "outcomes": {"type": "array", "minItems": 1, "items": {"type": "string"}},
    "steps": {
      "type": "array",
      "items": {
        "type": "object",
        "required": ["step_id", "type", "body"],
        "properties": {
          "step_id": {"type": "string"},
          "type": {"enum": ["explain", "command", "prompt", "edit", "verify", "quiz"]},
          "body": {"type": "string"},
          "copyable": {"type": "boolean"},
          "risk": {"enum": ["low", "medium", "high"]}
        }
      }
    },
    "checks": {
      "type": "array",
      "items": {
        "type": "object",
        "required": ["check_id", "description"],
        "properties": {
          "check_id": {"type": "string"},
          "description": {"type": "string"},
          "evidence_type": {"enum": ["manual", "command", "file", "quiz", "review"]}
        }
      }
    },
    "verified_at": {"type": "string", "format": "date"}
  }
}

35.4 Feature Registry

なぜここを読むか モデル名・利用可能Plan・最小Versionなどは月単位で変わり、本文に埋めると一斉に古びます。可変情報は外出ししないと持ちません。

やる／やらないとどうなるか feature_idごとに状態・最小Version・Plan・verified_at・official_sourceを持つRegistryへ抜き出し、本文を直さず追随できる仕組みが組めます。

これは何か Feature Registryの1エントリ例。status・minimum_version・plans・verified_at・official_source・notesを持つ最小形です。

実行するとどうなるか 本文を一切触らずRegistryのverified_atとstatusだけを更新すれば、教材全体の最新性を維持できる運用に切り替わります。

それぞれの意味

feature_id
Feature識別子です。本文ではこのIDを参照し具体的な仕様はRegistry側に持たせる構造にします。
status
available/suspended/deprecatedなどの状態です。書かないと停止済み機能を本文が紹介し続ける事故が起きます。
minimum_version
そのFeatureが動く最小バージョンです。書かないと古い環境で動かない手順を読者に踏ませてしまいます。
plans
free/paid/enterpriseなど利用可能Plan一覧です。書かないとPlanで使えない機能の手順を初心者に提示しがちです。
verified_at
公式情報と突合した最終確認日です。3ヶ月超過の項目は再確認対象として警告できます。
official_source
一次資料のURLまたはID参照です。書かないとRegistryの記述が誰のソースか追えず信頼が落ちます。
{
  "feature_id": "dynamic-workflows",
  "status": "available",
  "minimum_version": "2.1.154",
  "plans": ["paid"],
  "verified_at": "2026-06-22",
  "official_source": "official-doc-id",
  "notes": "Pro may require enabling in config"
}

古いverified_atへWarningを出す。

35.5 Progress

なぜここを読むか 「読んだ」を進捗とすると、実践したか・現場で使えたかが見えません。学びの段階は出力可能な状態で区別する必要があります。

やる／やらないとどうなるか seen→practiced→verified→appliedの4段階で進捗を持ち、verified以上を認定にカウントすることで、修了判定が客観的になります。

これは何か 4段階進捗の表記。各段階の意味を一行で示し、認定はverified以上を数えるルールと一体で運用します。

実行するとどうなるか 学習者と管理者の双方から、誰がどこまで到達したかが同じ語彙で見え、修了判定の客観性が担保されます。

それぞれの意味

seen
Lesson本文を読んだ段階です。学習者が何かを「目にした」だけで、行動も証拠もまだありません。
practiced
実際にコマンドや操作を試した段階です。手は動いたが結果の正しさは未検証で、完了とは数えません。
verified
操作結果が期待通りかをCheckで検証した段階です。認定はこの段階以上を数えるため、修了判定の起点になります。
applied
実務案件で同じ手順を再利用した段階です。学んだ内容が現場で再現できたことを示し、定着の証拠になります。
seen → practiced → verified → applied

認定はverified以上を数える。

35.6 実行連携

なぜここを読むか ブラウザから任意のShellを直接叩ける設計は、監査でほぼ必ずNGになります。実行と表示は分離しておく必要があります。

やる／やらないとどうなるか Lesson IDを受けたLocal Companion CLIがAllowlist Commandだけ実行し、Exit Code/Artifact Pathを返す形で安全に実機連携できます。

  1. Copy Button。
  2. Local Companion CLIがLesson IDを受ける。
  3. Allowlist Commandだけ実行。
  4. Exit Code/Artifact Pathを返す。
  5. 高Riskは手動承認。

Browserから任意Shellを直接実行しない。

やる／やらないとどうなるか Home/Learning Path/Lesson Player/Template Library/Skill Catalog/Harness Status/Admin Coverage/Certificationを一画面群へまとめます。
