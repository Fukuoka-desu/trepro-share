90分で、動く成果物を1つ持って帰る 最初の90分実習：会議アシスタント

『AIで何か作って』と言われて、3週間止まってる人、結構います。 題材が大きすぎて、どこから始めればいいか分からないんです。

この章で扱うのは、最初の90分で『動いた』を体験するための題材。会議の文字起こしを入れたら、議事録・タスク一覧・HTMLレポートの3点セットが出てくる、小さくて完結する成果物です。

ポイントは派手さじゃありません。要件定義→計画だけ承認→実装→検証→次回呼び出せるSkill化、までを一周することです。 この一周を経験すると、Claude Codeの基本ループが身体感覚になります。

90分後には、『次の会議から、これ呼び出すだけで5分で終わる』という業務型が手元に残ります。

4.2 要件定義

なぜここを読むか 目的・入力・出力・Schema・制約・完了条件を曖昧にしたまま実装すると、AIは推測で埋めて手戻りが連鎖する

やる／やらないとどうなるか requirements.mdが埋まると、AIは何をしてはいけないかまで分かる。担当・期限の捏造を構造的に防げる

docs/requirements.md：

これは何か requirements.md。目的・入力・出力・Task Schema・制約・完了条件を1ファイルに固定する要件定義書

実行するとどうなるか AIへ渡す前に『何を作り、何を作らないか』が明確になる。担当・期限の捏造禁止まで仕様で縛れる

# Meeting Assistant 要件定義

## 目的
会議文字起こしから、決定事項と担当Taskを追跡可能にする。

## 入力
- `inputs/*.md` または `.txt`
- UTF-8
- 発言者表記が不完全でも処理する

## 出力
- 人間向けMarkdown
- Application向けJSON
- 閲覧用HTML

## Task Schema
- task_id
- title
- owner
- due_date
- status
- source_quote
- confidence
- needs_confirmation

## 制約
- 入力を変更・削除しない
- 不明な担当・期限を推測しない
- 決定事項と提案を分離する
- 外部送信しない

## 完了条件
- Sample 3件で出力できる
- JSON Schema検証が成功
- Owner/Status/DueでFilterできる
- READMEに起動手順がある

4.3 Project CLAUDE.md

なぜここを読むか Project単位のお願いがないと、AIは外部API追加・入力ファイル改変・担当の創作などを善意でやってしまう

やる／やらないとどうなるか CLAUDE.mdに『inputsを触らない』『null+needs_confirmationを使う』と書けば、暴走が起きにくい設計になる

これは何か Project CLAUDE.md。inputs改変禁止・担当捏造禁止・null+needs_confirmation活用・外部API無断追加禁止を書く

実行するとどうなるか Project全体で守るべきルールがAIに伝わる。会話のたびに同じ注意を書く必要がなくなる

# Meeting Assistant

- Respond in Japanese.
- Read `docs/requirements.md` before changing files.
- Never modify files under `inputs/`.
- Do not invent owners, deadlines, or decisions.
- Use `null` and `needs_confirmation: true` when uncertain.
- Run all checks before reporting completion.
- Do not add external APIs, authentication, or databases without approval.

4.4 議事録Skill

なぜここを読むか 成功した手順を会話の中に置いたままだと、翌日同じ依頼でまた1から説明が要る。Skill化で再利用できる

やる／やらないとどうなるか SKILL.mdに入力・処理・出力・品質チェックを書けば、次回からは1行呼び出しで議事録パイプラインが走る

.claude/skills/meeting-minutes/SKILL.md：

これは何か 議事録SkillのSKILL.md。入力・Workflow・Markdown出力・Quality checksを宣言する再利用可能な手順書

実行するとどうなるか 次の会議からは1コマンドで議事録パイプラインが走る。担当捏造や日付形式のブレが構造的に消える

---
name: meeting-minutes
description: Convert a meeting transcript into a traceable Markdown summary and structured task JSON without inventing owners, dates, or decisions.
argument-hint: "<transcript path>"
---

# Inputs
- transcript path
- meeting date/title when available

# Workflow
1. Read the transcript without changing it.
2. Identify agenda, facts, proposals, decisions, blockers, and actions.
3. Keep decisions separate from proposals.
4. For each task, preserve a short source quote.
5. Use null and `needs_confirmation` for missing owner/date.
6. Validate JSON against the project schema.

# Markdown output
- Overview
- Participants
- Agenda
- Decisions
- Actions
- Open questions
- Risks

# Quality checks
- No invented facts
- Every task has a source quote
- No duplicate tasks
- Dates are ISO 8601
- Markdown and JSON describe the same tasks

4.6 Plan依頼

なぜここを読むか いきなり実装に走らせると、ファイル構成もSchemaも一発で確定する。後で『そうじゃない』が大量に発生する

やる／やらないとどうなるか Plan依頼で構成・Schema・処理・UI・Test・Risk・Rollbackを先に文書化させる。実装前にレビューできる

これは何か Plan依頼Prompt。実装に入らずFile構成・Schema・処理・UI・Test・不確定・Risk/Rollbackだけを先に出させる

実行するとどうなるか 実装前に設計をレビューできる。後から『そうじゃなかった』で全書き直しになる事故が消える

`docs/requirements.md`、`CLAUDE.md`、Sample Inputを読んでください。
まだ実装せず、次を含む計画を作ってください。

- File構成
- Data Schema
- 処理手順
- UI構成
- Test Case
- 不確定事項
- RiskとRollback

4.7 実装依頼

なぜここを読むか 承認したPlanがあっても、実装段階の制約を毎回書かないと外部API追加・入力改変・Test省略が混入する

やる／やらないとどうなるか 禁止事項とDiff・Command・Test結果・残Riskの報告を求める形に固定できる。完了報告が証拠付きになる

これは何か 実装依頼Prompt。外部API/DB/認証禁止・入力不変・JSON Schema検証・完了時にDiff/Command/Test/残Risk報告

実行するとどうなるか 実装が承認済みPlanからブレない。完了報告が証拠付きで返り、後追い確認が要らない

承認した計画に従い、最小構成で実装してください。

- 外部API/DB/認証は禁止
- 入力を変更しない
- JSON Schema検証を追加
- Sample DataでTest
- HTMLはMobile対応
- 完了時にDiff、Command、Test結果、残Riskを報告

出力修正を会話だけに残さない。
