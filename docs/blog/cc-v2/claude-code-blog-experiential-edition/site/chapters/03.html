AIに、見せていい範囲を線で引く 導入とProject Folder

『デスクトップにAI作業フォルダひとつ』で全案件を回してませんか。 そこに顧客A・顧客B・自分の実験が全部入ってる。

これ、AIから見ると地獄です。 どの顧客の話か、どこを触っていいか、まったく区別がつかない。結果、誤って別案件のファイルを書き換える事故が起きます。

この章で扱うのは、AIに見せる範囲を決める『プロジェクトフォルダ』という考え方。 案件ごとに専用フォルダを切って、docs（資料）・input（素材）・output（生成物）・tests（検証）の4つに分けるだけ。これだけで、AIが見るべき範囲と、人が確認すべき範囲がはっきり分かれます。

設定は10分。一度切ると、案件取り違えの不安が消えます。

実装リファレンス

3.1 macOSへCLIを導入

なぜここを読むか Claude CodeはCLIから入るのが基本。ここを公式手順で揃えないと、後の章のコマンドが全部動かない

やる／やらないとどうなるか claude --versionとclaude doctorで導入確認まで取れる。会社環境ならURL・Hash・Versionも管理者で固定できる

公式Native Installerを使う。

これは何か macOSへClaude Code CLIを入れる公式インストール。インストール後にバージョン確認とdoctor診断まで実行する

実行するとどうなるか CLIがコマンドラインから動く状態になり、依存関係も自動診断される。会社環境では管理者がHashで固定できる

curl -fsSL https://claude.ai/install.sh | bash
claude --version
claude doctor

会社環境では、Install URL、Hash、Proxy、許可Versionを管理者が固定する。

3.2 VS Code

なぜここを読むか CLI単体だと変更の見通しが悪い。IDE側でDiffとファイルツリーを見ながら進めると人間のレビュー負荷が大きく減る

やる／やらないとどうなるか VS CodeでProjectを開き、Trust範囲を自社に限定できる。CLIとIDEが同じContextで動き、両刀使いに移行できる

  • 公式Anthropic提供のClaude Code Extensionを確認して導入する。
  • Project FolderをVS Codeで開く。
  • Trustするのは自社または確認済みRepositoryだけ。
  • CLIとIDEは同じProject Contextを使う。

IDEは見やすいが、すべての機能・診断はTerminalで先に提供される場合がある。両方使えるようにする。

3.3 Project Folderの原則

なぜここを読むか デスクトップ1フォルダに顧客A・顧客B・実験を全部入れると、AIが見るべき範囲が混ざる。誤って別案件を書き換える事故が起きる

やる／やらないとどうなるか 案件ごとにProject Rootを切れば、AIが見るのはその顧客分だけ。情報漏えいと取り違えのリスクが構造的に消える

相関のない仕事

これは何か 相関のない仕事を別Rootに分けるディレクトリ例。ライティングと会計が別フォルダに分かれる

実行するとどうなるか AIの探索範囲が無関係なファイルを巻き込まない。Context精度が上がり処理時間も短くなる

~/work/writing/
~/work/accounting/

別Rootへ分ける。

相関する仕事

これは何か 相関する仕事を共通Rootへまとめるディレクトリ例。研究・台本・素材・レポートを1つのProjectに収める

実行するとどうなるか 知識と成果物が同じRoot内で循環する。前回の調査を次回の台本へそのまま渡せる

~/work/youtube/
├── research/
├── scripts/
├── assets/
└── reports/

共通Rootを開き、知識と成果物を共有する。

3.4 推奨Project構造

なぜここを読むか docsもinputもoutputも同じ階層に放り込むと、AIも人間もどこを触ればいいか迷う。役割で分ける標準が要る

やる／やらないとどうなるか CLAUDE.md・docs・inputs・output・src・tests・scripts・.claudeの最小構造が決まる。初見の人もすぐ動ける

これは何か 推奨Project構造。CLAUDE.md・docs・inputs・output・src・tests・scripts・.claudeを役割で分ける標準ツリー

実行するとどうなるか 初見の人でもどこを触ればいいかが一目で分かる。AIにも『生成物はoutput/』と渡せて事故が減る

project/
├── CLAUDE.md
├── README.md
├── docs/
│   ├── requirements.md
│   ├── architecture.md
│   └── runbook.md
├── inputs/
├── output/
├── src/
├── tests/
├── scripts/
├── .claude/
│   ├── rules/
│   ├── skills/
│   ├── agents/
│   └── settings.json
├── .gitignore
└── package.json / pyproject.toml

3.5 CLIの基本

なぜここを読むか Claude Codeは対話Session・1回実行・継続・再開で動き方が変わる。最初の数コマンドを覚えないと『どう止めて再開する？』で詰まる

やる／やらないとどうなるか claude・-p・--continue・--resumeと、/clear・/compact・/contextを覚えれば、Sessionを能動的に管理できる

これは何か Claude Code CLIの4つの起動モード。対話・1回実行・継続・再開を切り替えるコマンド

実行するとどうなるか セッションを能動的に開始・継続・再開できる。長いタスクを途中で止めて翌日再開できる

claude                         # 対話Session
claude -p "依頼"              # 非対話の1回実行
claude --continue             # 直前Sessionを継続
claude --resume               # 保存Sessionを選んで再開

Session内の代表操作：

これは何か Session内の代表Slashコマンド。clear・compact・context・stats・rewind・memory・permissions・sandbox・mcp

実行するとどうなるか 履歴・権限・MCP接続を会話の中から直接確認・操作できる。問題が起きたら/rewindで戻せる

/clear      別Topicへ切り替える前に履歴をクリア
/compact    同じ目的のまま履歴を圧縮
/context    Contextの内訳を確認
/stats      利用状況を確認
/rewind     会話・編集Checkpointへ戻る
/memory     CLAUDE.mdとAuto Memoryを確認
/permissions 権限Ruleの出所を確認
/sandbox    Sandbox設定を確認
/mcp        MCP接続を確認

利用中Versionの /help を最優先にする。
