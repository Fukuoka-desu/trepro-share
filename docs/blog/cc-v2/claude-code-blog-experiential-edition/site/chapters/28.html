第28章

毎日の雑務を、AIに渡せる単位に切る 文書・ファイル・会議

『議事録・タスク管理・ファイル整理を、毎日手作業でやっている』。 これは仕事量の問題ではなく、AIに渡せる単位に切り出していないだけです。

ほとんどの人は、雑務を『自分でやる前提』で考えています。 ですが、文書・ファイル・会議の処理は、AIが最も得意とする領域のひとつです。

この章で扱うのは、毎日発生する雑務を、AIに渡せる単位に分けるための型です。 取り込み・分類・整形・保存・通知を、それぞれ独立した Skill として組み立てます。

毎日の雑務を、AI に渡せる単位に切る。それが、自分の時間を取り戻す最初の一歩です。

実装リファレンス

28.1 請求書Folderを一覧化

なぜここを読むか 請求書は金額・税・通貨が絡む高リスクデータです。元ファイルを触らずCSVへ書き出し、推測項目と低信頼行をReview表へ分けることで、自動化と人手確認の責任境界を最初に決められます。

やる／やらないとどうなるか Dry-run結果と例外一覧が先にあるため、自動化を有効化しても会計事故が起きにくくなり、経理が確認すべき行だけに集中できます。

Input：inputs/invoices/。Output：CSV、Review表、読取不能一覧。

これは何か 請求書PDFから10列のCSVを生成し、元ファイル無改変・推測禁止・total検算・重複候補抽出・低信頼Reviewまでを1プロンプトで縛る指示例です。

実行するとどうなるか 標準項目は自動入力されつつ例外と低信頼行は人へ回り、毎月数百枚規模でも会計事故ゼロのまま処理時間を圧縮できます。

それぞれの意味

source_file
元PDFのパスを記録し、抽出値が怪しい時に原本へ即座に戻れるようにする列です。これがないと根拠検証が不可能になります。
vendor_name
請求元の名称で、同一取引先を統一表記にしないと集計と支払先突合が狂います。表記揺れは別途辞書で補正します。
invoice_number
請求書を一意に識別する番号で、重複候補検知の主キーになります。空欄や重複は二重払い事故の温床です。
invoice_date / due_date
発行日と支払期限で、月次計上と振込スケジュールの根拠になります。混同すると遅延損害金が発生します。
subtotal / tax / total
小計・消費税・合計の三点で、total = subtotal + tax を検算しないと税抜誤計上による会計事故が起きます。
currency
通貨コードで、JPYとUSDの混在を見落とすと為替換算が抜けて利益が歪みます。必ず明示します。
extraction_confidence
AIが抽出した値の確度スコアで、低信頼行をReview表へ振り分ける閾値判定に使います。なければ全行が手動確認対象になります。
review_note
AIが迷った理由や曖昧箇所を人へ申し送るメモ列で、Reviewerが文脈ゼロから読まずに済みます。
元File無改変・推測禁止
原本を改変せず空欄も推測で埋めない縛りで、後で根拠を辿れる状態を維持します。便利を優先すると監査で詰みます。
支払/会計登録はしない
抽出と入金実行の責任を分離する境界で、AI誤抽出が即座に資金移動へ伝播する事故を物理的に止めます。
請求書を読み、source_file, vendor_name, invoice_number, invoice_date,
due_date, subtotal, tax, total, currency, extraction_confidence, review_note
の列でCSV化してください。

- 元Fileを変更/移動/削除しない
- 不明項目を推測しない
- total = subtotal + taxを検算
- 同一Invoice Numberを重複候補にする
- 低信頼行をReview表へ
- 支払/会計登録はしない

28.2 Folder整理

なぜここを読むか 整理という言葉は『削除』と混同されがちです。Explore→Plan→Human Review→Moveの順で進め、移動前後のPathとHashを残すことで、誤分類を必ず元へ戻せる状態を維持できます。

やる／やらないとどうなるか 整理で資産を失う事故が原理的に消え、AIに整理を任せても元データへ戻れる確証を持ったまま実運用へ進められます。

これは何か Folder整理を可逆な工程に分解し、Plan段階で人が見てからMoveへ進め、quarantine・Retention・Delete Approvalを段階配置する標準フローです。

実行するとどうなるか 整理という名の削除事故が物理的に起こらず、誤分類は移動前のPathとHashから即時にRollbackできます。

それぞれの意味

Explore
対象フォルダを変更せず走査して候補を作る段で、いきなりMoveしないことで誤分類の被害を未然に防ぎます。
organization-plan.csv
source/proposed_destination/category/reason等を持つ計画表で、人が見て判断できる単位に変更案を可視化します。
Human Review
計画表に人が承認印を入れる段で、ここを飛ばすと誤分類が本番フォルダへ流れます。自動化の責任分界点です。
Move
承認済みエントリだけを実際に移動する段で、移動前後のPath/Hashを必ず残してRollback可能性を確保します。
quarantine/YYYY-MM-DD
削除候補を即消さず日付付き隔離フォルダへ送る緩衝地帯で、誤判定を取り消せる猶予期間を作ります。
Retention / Delete Approval
保管期間を経て初めて削除し、最終承認も人が押す二段ゲートで、整理という名の削除事故を構造的に排除します。
Explore → organization-plan.csv → Human Review → Move
→ quarantine/YYYY-MM-DD → Retention → Delete Approval

`inputs/mixed-files/`を分析し、まだ操作せず、source, proposed_destination,
category, reason, duplicate_group, riskを持つPlanを作る。
Hidden、Package、Symlink、1GB超は対象外。

実行時は上書き禁止、移動前後のPath/HashをResultへ保存。

28.3 契約書論点

なぜここを読むか 契約書をAIに『有効か無効か』判断させると、法的責任が曖昧になります。条項・義務・期限・曖昧点・専門家質問を抽出するだけに役割を限定すれば、AIは下処理に徹し人が判断できます。

やる／やらないとどうなるか AIによる誤った法的結論で社内が動く事故が消え、法務レビュー時間も論点が整理された状態から始められます。

AIへ有効/無効を断定させず、条項、平易な説明、義務、金額/期限、曖昧点、専門家への質問を作る。最終判断は法務へ。

28.4 会議Assistant本番化

なぜここを読むか 会議AssistantをアドホックなChatのまま運用すると、Meeting IDの重複・Owner欠落・原文未参照で品質が下がります。冪等・JSON Schema・原文Quote・Retentionを固めると本番運用に耐えます。

やる／やらないとどうなるか 議事録からTaskへの変換が安定し、誰が・いつまでに・どの発言を根拠に動くかが、毎回同じ精度で出力されます。

  • Meeting IDで冪等。
  • Participant表記統一。
  • Decision/Proposal分離。
  • TaskにOwner/Due/Source Quote。
  • 不明はnull。
  • 原文Link。
  • JSON Schema。
  • Retention。

これは何か 会議AssistantのTask出力JSON Schema例で、source.quoteとconfidence・needs_confirmationを必須化することで根拠と確度を毎回検証可能にします。

実行するとどうなるか Task登録時に発言の原文へ即座に戻れ、AIが言い切ったか曖昧かが構造として可視化されるため再確認のループが短縮されます。

それぞれの意味

task_id
会議由来Taskを一意に識別するIDで、同じ議事録を再実行しても重複登録されない冪等キーになります。
title / owner / due_date / status
誰が何をいつまでに、現在どの段階かを表す基本4項目で、外部Task Boardへの取り込みに必須の最小スキーマです。
source.meeting_id
どの会議由来かを示す参照で、Task発生源を辿れることで議事録と実行の追跡可能性が成立します。
source.quote
Task根拠となった発言の原文を保持する項目で、AIの解釈ではなく一次発言まで戻って確認できます。
confidence
AIの抽出確度スコアで、閾値以下は人手レビューへ自動振り分けする判断軸になります。なければ全件確認になります。
needs_confirmation
曖昧な指示や条件付き発言にtrueを立て、人が確認すべきTaskを明示するフラグで、機械処理から外す境界線になります。
{
  "task_id": "meeting-20260622-001",
  "title": "新LPの見出し案を作成",
  "owner": "山田",
  "due_date": "2026-06-26",
  "status": "todo",
  "source": {
    "meeting_id": "meeting-20260622",
    "quote": "山田さん、金曜までに見出し案をお願いします"
  },
  "confidence": 0.96,
  "needs_confirmation": false
}

28.5 週報

なぜここを読むか 週報は『書く時間』のために遅延しがちです。過去3件から形式だけを抽出し、今週の議事録とTaskから素材を埋めるDraftに変えると、書く作業から見直す作業へ性質が変わります。

やる／やらないとどうなるか 送信前確認は残しつつ初稿の生成時間がゼロに近づき、上長は判断・修正に時間を使えるため週報の質が上がります。

過去3件から形式だけを抽出し、今週の議事録/Taskから実績、数字、進行、Blocker、来週、支援依頼をDraft化。誇張せず、送信しない。
