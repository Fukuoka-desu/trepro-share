第32章

朝起きたら、仕事が3つ終わっている 継続業務の自動化

『朝起きて最初にやる仕事を、AIに任せられたら』。 これは贅沢な願いではなく、設計次第で実現できる現実です。

ほとんどの人は、AIを『その都度呼ぶもの』として使っています。 ですが、定型業務に成功条件・最大回数・タイムアウト・承認地点を設けて自走させれば、AIは寝ている間にも動きます。

この章で扱うのは、継続業務を安全に自動化する設計です。 失敗時の戻し方、人の承認地点、ログと通知の設計。便利さと事故半径を同じ机の上で並べます。

朝起きたら、仕事が 3 つ終わっている。そういう運用は、設計の品質次第で安定します。

実装リファレンス

32.1 毎朝Brief

なぜここを読むか 朝の状況把握は属人化しやすく、担当が休むと止まります。Calendar・未返信・締切Task・重要MentionをBriefへ集約し、送信や予定変更はしないと縛れば、自動配信しても安全です。

やる／やらないとどうなるか 今日の動き方が10秒で把握でき、休んでも誰でも同じ精度でその日を立ち上げられる業務継続性が手に入ります。

Calendar、未返信候補、締切Task、重要Mentionから、今日の予定、準備Meeting、返信Draft、Riskを作る。送信/予定変更/Task更新はしない。

32.2 会議→Task Board

なぜここを読むか 議事録からTask登録までを一気通貫にすると、AIの誤抽出がそのまま外部Systemへ流れます。Human Reviewを必ず挟み、承認済みだけDraft登録する流れにすれば、AIと人の責任が分離されます。

やる／やらないとどうなるか Task Boardに不確かな情報が混入しなくなり、議事録から実行までの待ち時間も短縮されます。

これは何か 会議文字起こしからTask Boardへ流すパイプで、承認済みのTaskだけ外部SystemへDraft登録する境界を明示します。

実行するとどうなるか 議事録からTask登録までが半自動で成立しつつ、外部System側にはレビュー済みデータしか入らない安全境界が確保されます。

それぞれの意味

Transcript
会議の文字起こし原文で、抽出根拠を後から辿れる一次データになります。要約から始めると引用検証が不可能になります。
Meeting Skill
Transcriptを構造化Task JSONに変換するSkillで、抽出ルールをコード化することで担当者が変わっても結果がブレません。
Human Review
AIが抽出したTask候補を人が承認する段で、ここを省くと誤抽出がそのまま外部Systemへ流れます。責任分界点です。
Task JSON
承認済みTaskを定型JSONで保持する中間形で、複数の外部System（Notion/Jira等）へ同じデータから配布できます。
Dashboard
Task JSONを集約表示するUIで、議事録と実行状況を1画面で見られるためフォロー漏れを発見しやすくなります。
承認済みTaskだけ外部SystemへDraft登録
外部Systemには未承認データを流さず、登録もDraft状態にとどめる二重ガードで、AI誤抽出による外部事故を防ぎます。
Transcript → Meeting Skill → Human Review → Task JSON → Dashboard
→ 承認済みTaskだけ外部SystemへDraft登録

32.3 Skill改善Loop

なぜここを読むか Skillを会話中に即全社上書きすると、誰かの作業が突然壊れます。実行ログ・失敗Fixture・Fresh Evaluator・Canary・Metricsを経てStableへ昇格させると、影響範囲を測ってから配布できます。

やる／やらないとどうなるか Skillの改善がデータに基づき段階配布される運用に変わり、便利な変更が誰かの本番を壊す事故が消えます。

これは何か Skill改善Loopの標準パイプで、Canary配布とMetrics確認を経てStable昇格させる段階更新フローです。

実行するとどうなるか Skillの改修が個人の善意ではなくデータと評価器でゲートされ、配布事故と『便利な改悪』の両方を避けられます。

それぞれの意味

実行
Skillを実際の業務で動かして使う段で、机上ではなく現場負荷下のログを集める出発点になります。
User評価
利用者が結果に対して良し悪しを付ける段で、定量的改善の素材になります。なければ感想ベースの改修になります。
失敗Fixture
失敗ケースを再現可能な入力データとして固定する段で、修正後に同じ失敗が起きないか機械検証できます。
Generator
失敗Fixtureを踏まえてSkill改修案を生成する段で、人が手で書き直すのではなく差分の根拠が残ります。
Fresh Evaluator
改修案を独立コンテキストで評価する段で、改修者の願望バイアスを排除して品質判定を客観化します。
Canary
少人数へ先に配布する段で、全社破壊する前に本番で本当に壊れないかを限定範囲で確かめます。
Metrics → Stable
Canary実績が指標を満たした時だけStable昇格させる段で、便利な改悪が全社配布される事故を構造的に防ぎます。
実行 → User評価 → 失敗Fixture → Generator → Fresh Evaluator
→ Canary → Metrics → Stable

会話中に即全社上書きしない。

32.4 成熟度

なぜここを読むか 全業務を最初から監視付き自動へ持っていくと、運用負荷が爆発します。Level 0手動Chatから Level 6 Harness+Audit+Releaseまで段階化し、業務ごとに必要な最小Levelを選ぶと最適化できます。

やる／やらないとどうなるか 対象業務の現在地と次のターゲット段階が共通言語になり、過剰自動化と不足自動化のどちらも避けやすくなります。

Level 状態
0 手動Chat
1 Skill
2 File/Test
3 Hook/MCP/Scheduled
4 Subagent Workflow
5 Goal + Verifier + Human Gate
6 Harness + Audit + Release

必要な最小Levelを選ぶ。

実務に落とすときの補足

自動化候補の優先順位
週1回以上の頻度・入力フォーマットが安定・完了条件が機械検証できる・失敗しても人へ戻せる、の4条件を満たす業務だけを最初の自動化対象にします。条件を満たさない業務を強引にSkill化すると、最初の数週間は動いても運用フェーズで誤動作が出てきます。
Morning Brief の入力源を固定する
Calendar・メール・Slackメンション・締切Taskなど、毎朝の情報源を毎回違う方法で取りに行くと、ある日急に空欄になります。情報源と更新時刻を仕様として書き、対象が増えた時はSkillを直すのではなくRegistryへ追記する設計にします。
成熟度を測る尺度
個人手作業→再利用→半自動→監視付き自動の4段階で、対象業務がいまどこにいるかを月次で記録します。一気に最終段階を目指さず、半自動で1ヶ月動かしてから監視付き自動へ進めると、運用事故がほぼ起きません。
次の章への接続
自動化を増やすほど、過去の失敗を学習資産へ変換する次章のテーマが重要になります。動くものが増えるほど、止まったときの原因分析と再発防止の仕組みが品質を決めるからです。
