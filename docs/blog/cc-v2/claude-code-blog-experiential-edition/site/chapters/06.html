第6章

速いチームは、工程を省かない Explore → Specify → Plan → Implement → Verify → Review → Commit

『一度うまくいったプロンプトを、翌週も使えていますか』。 ほとんどの人は、また最初から書き直しています。 そして、また違う結果が返ってきます。

これは運の問題ではなく、仕事の順序が決まっていないことが原因です。 速いチームは、毎回同じ7つの段階を回しています。探す→仕様を決める→計画→実装→検証→レビュー→保存（Explore/Specify/Plan/Implement/Verify/Review/Commit、頭文字でESPIVRC）。

この章を読み終わる頃には、アプリ開発だけでなく、資料作成・調査・ファイル整理・Skill改善まで、同じ順序で進められるようになります。速いチームは工程を省いているのではなく、同じ工程を小さく明確に回している。それがこの章の核心です。

この章の概観 ／ 何を扱い、何ができるか

この章で扱う機能と、それで何ができるようになるか。実装に入る前に、全体像をここでつかみます。

  • ESPIVRC標準ループ

    Explore→Specify→Plan→Implement→Verify→Review→Commitの7段階を共通言語にし、業務種別を問わず同じ順で進める仕事の骨格。

    入力
    新しい依頼、作業対象、関係者
    出力
    7段階ごとの成果物（探索メモ・仕様・計画・実装・検証・レビュー・コミット）

  • Plan段階の変更範囲宣言

    実装の前に『何を触り何は触らない』を明示させ、AIの暴走と人のレビュー負荷を構造的に減らす設計。

    入力
    仕様、現状のコード/ファイル
    出力
    影響範囲・差分予定・前提条件を含むPlan

  • Verify→Review→Commit分離

    機械検証・別文脈レビュー・意味のある単位での保存を分けて、『動いたっぽい』を本物の完了に変える終盤フロー。

    入力
    実装済みの変更、テストケース、レビュー観点
    出力
    検証ログ、レビュー指摘、コミットメッセージ付き履歴

実装リファレンス

6.1 全業務を同じ型へ入れる

なぜここを読むか 場当たり的に進めると、同じ依頼でも毎回手順がぶれて再現できなくなります。最初に『仕事の型』を共通言語として置く必要があります。

やる／やらないとどうなるか 七段階の見出しを揃えるだけで、コードでも資料でも調査でも同じ順で進められ、抜けと作り直しが目に見えて減ります。

これは何か ESPIVRC七段階の各段階で何をするかを一望にしたテキスト表です。仕事の型を共通言語として机に並べます。

実行するとどうなるか 依頼テンプレや章運用の見出しとしてそのまま使え、業務種別に関わらず同じ順序で進められるようになります。

Explore    現状を読む
Specify    目的と完了条件を固定
Plan       変更範囲と検証方法を決める
Implement  小さく変更
Verify     機械的・視覚的に確認
Review     別文脈で欠陥を探す
Commit     良い状態を保存

このLoopはCodeだけでなく、資料、調査、業務自動化にも使える。

6.2 Explore

なぜここを読むか 現状を読まずに編集を始めると、AIは前提を勝手に想像して既存仕様を壊しにきます。最初に事実を集める段階が要ります。

やる／やらないとどうなるか 編集前にFile・既存挙動・Testを読ませて根拠付きでまとめさせると、後段のPlanとReviewが現実の側に着地します。

最初に読ませるもの：

  • README.md、CLAUDE.md、AGENTS.md。
  • Requirements、Architecture、Runbook。
  • Package/Build/Test設定。
  • 関係Codeと既存Test。
  • git status、最近の変更。

Prompt：

これは何か Explore段階で使うPromptで、編集を禁じたまま関係File・既存挙動・制約をEvidence付きで集めさせる指示文です。

実行するとどうなるか AIが現状を読まずに想像で書き始めるのを止め、後段のPlan/Verifyが事実に基づくものになります。

まだ編集しないでください。
この依頼に関係するFile、既存挙動、Test、制約、未知点を調査し、Evidence付きでまとめてください。

6.3 Specify

なぜここを読むか 『速くする』『見やすくする』のような動詞だけでは、AIも人もゴール判定ができません。完了を判定可能な状態に固定する必要があります。

やる／やらないとどうなるか p95やContrastなど数値や挙動で完了を定義すると、Verifyで合否を機械的に判断でき、『動いたっぽい』を排除できます。

完了条件は動詞ではなく、判定可能な状態にする。

曖昧 検証可能
速くする p95が500ms以下
見やすくする 375pxで横Scrollなし、Contrast基準を通す
Bugを直す 再現Testが修正前Fail、修正後Pass
整理する Manifestどおり移動しHash一致
調査する 全主要Claimに一次出典と取得日

6.4 Plan

なぜここを読むか Specifyだけでは『どこを触るか』が決まっておらず、AIは関係ない箇所まで書き換えがちです。変更範囲の宣言を挟む必要があります。

やる／やらないとどうなるか 変更File・整合・Rollback・対象外まで先に書かせると、AIの暴走が手前で止まり、レビュー側の負荷も大きく下がります。

Planで確認する項目：

これは何か Plan段階で確認すべき9項目のチェックリスト。変更範囲・整合・Migration・Side Effect・Rollbackなどを明示させます。

実行するとどうなるか 実装前に『触らない場所』が宣言され、AIの暴走と人のレビュー負荷を構造的に減らせます。

- 変更Fileと理由
- 既存設計との整合
- Data Migration
- External Side Effect
- Test/Verification
- Security/Privacy
- Rollback
- 対象外
- 承認が必要な点

複数File、Data変更、外部連携、高RiskはPlan Modeを使う。

6.5 Implement

なぜここを読むか 一度に大きく変更すると、原因切り分けができず、Testを通すために設定を緩めたくなる誘惑も生まれます。Implementには規律が要ります。

やる／やらないとどうなるか 小さく・無関係Refactorを混ぜず・Configを弱体化させないルールを守ると、戻し作業と事故調査の時間が劇的に短くなります。

  • 1回の変更を小さくする。
  • 無関係Refactorを混ぜない。
  • 既存規約を守る。
  • Config弱体化でTestを通さない。
  • 長時間Taskは途中成果物をFileへ保存する。
  • 同じFileを複数Agentが同時編集しない。

6.6 Verify

なぜここを読むか Implementの後に『たぶん動いた』で進めると、本番で初めて壊れます。機械的に確かめる段階を独立させる必要があります。

やる／やらないとどうなるか Static・Behavior・Visual・Data・Operationsを通して証拠を残させれば、未実行は未実行と分かり、合否が客観的になります。

これは何か Verify段階の検証カテゴリを5層に並べた一覧。静的・挙動・視覚・データ・運用の観点を漏らさず通します。

実行するとどうなるか どの層が未実行かが見える化され、『なんとなく動いた』ではなく機械的な合否に置き換えられます。

Static: format, lint, typecheck, schema, secret scan
Behavior: unit, integration, E2E, replay
Visual: screenshot, responsive, PDF/print
Data: row count, totals, hash, duplicate, null
Operations: logs, retry, timeout, rollback

Claudeへ証拠を要求する。

完了条件ごとに、実行したCommand、Exit Code、出力要約、Artifact Pathを表で示してください。
未実行は未実行と書いてください。

6.7 Review

なぜここを読むか 実装した本人の文脈は自分の判断に引っ張られ、欠陥を見落とします。Reviewは別の目で行う段階として分離する必要があります。

やる／やらないとどうなるか 新しいSessionやFresh ReviewerでDiffを独立に見させると、要件・Security・Edge caseの抜けが浮き上がり、品質が一段上がります。

作ったSessionは自分の判断へ引っ張られる。新しいSessionまたはFresh Reviewerを使う。

これは何か 別文脈Reviewer向けのPromptで、実装者の説明を鵜呑みにせずFile・Lineを根拠に独立Reviewさせる指示文です。

実行するとどうなるか 作成者バイアスから切り離された指摘が得られ、Edge caseやTest不足が事前に拾えるようになります

このDiffを、要件、Security、Data loss、Edge case、Test不足の観点で独立Reviewしてください。
実装者の説明を鵜呑みにせず、FileとLineを根拠にしてください。

6.8 Commit

なぜここを読むか 良い状態を保存しないと、次のセッションで何を直したか辿れなくなります。意味ある単位で履歴に残す段階が要ります。

やる／やらないとどうなるか Status・Diff・Statを確認してから1つの意図にまとめてCommitすると、後日の巻き戻しとレビューが現実的な作業になります。

Commit前：

これは何か Commit前に必ず通す4つの確認コマンド。状態・差分・統計・全文Diffを段階的に確認します。

実行するとどうなるか Secretsや他人の変更の混入、意図しないFileの混在を物理的に防ぎ、Commitが1つの意図にまとまります。

git status --short
git diff --check
git diff --stat
git diff

Commitは1つの意図にまとめ、Secrets、生成Noise、他人の変更を含めない。

補助図の制作指示

こんな場面で使う

章の内容が実務でどう活きるか。架空の場面・行動・効果の三点で示します。

場面

ABC物流の社内システム改修で、依頼→実装→『動いたっぽい』→本番→事故、という流れを毎月繰り返している。

こう使う

Explore→Specify→Plan→Implement→Verify→Review→Commitの七段階に依頼テンプレを揃え、各段階で証拠を残す。

得られるもの

本番事故の発生源を段階に切り分けて潰せるようになり、改修1本あたりの平均事故件数が減る。

場面

Webデザイナー（架空）が、AIに『修正お願い』と頼んだら関係ない部分まで書き換えられ、何が変わったか追えない。

こう使う

Plan段階で変更範囲をAIに宣言させ、Verifyで差分とテスト結果を見てからCommitする運用にする。

得られるもの

予期しない書き換えがPlanで弾かれ、レビュー時間が短く、巻き戻しも安全になる。
