第27章

新人の初日が、1人で完結する OnboardingとRunbook

『新人にAI環境を整えるのに、毎回1日かかっている』。 これは新人の問題ではなく、Onboarding の手順が型化されていないだけです。

ほとんどの人は、新人の初日を口頭の説明で進めています。 ですが、口頭は再現性が低く、後から困った時に戻る場所がありません。

この章で扱うのは、Onboarding を Runbook（手順書）として書き、新人が 1 人で初日を完走できる状態を作る方法です。 セットアップ・最小実習・確認チェック・困った時の連絡先まで、すべて文書で残します。

新人の初日が、1 人で完結する。そこまで作って、ようやく組織として AI を扱えるようになります。

実装リファレンス

27.1 30分Flow

なぜここを読むか 新人のOnboardingを口頭で進めると、初日の所要時間と詰まり所が毎回違ってしまいます。30分の時間割を区切って再現可能にする必要があります。

やる／やらないとどうなるか 0〜5分・5〜15分・15〜20分・20〜30分の4ブロックで前提確認→Install→Verify→Bronze実演までの導入Flowを全員が同じ手順で完走できるようになります。

0〜5分

なぜここを読むか Installスクリプトを叩く前に、対応端末・OS・GitHub組織・Claude権限・Network/Proxy・Data研修まで確認しないと、途中で進めなくなる事故が起きます。

やる／やらないとどうなるか 6項目の事前Checkboxで「ここで止まる人がいる」を早期検出でき、ScriptをPipe実行する前に環境差を潰せます。

これは何か 0〜5分で確認する前提条件（管理Mac・対応macOS・GitHub組織・Claude権限・Network/Proxy・Data研修）を6項目のChecklistにしたものです。

実行するとどうなるか Installへ進む前に止まる原因が早期に切り分けられ、Pipe実行の前段で組織側の不備を吸い上げられます。

[ ] 管理Mac
[ ] 対応macOS
[ ] GitHub組織
[ ] Claude権限
[ ] Network/Proxy
[ ] Data研修

5〜15分

なぜここを読むか 短期TokenをShell履歴やCommand引数に残すと、配布Repo経由で漏えいします。export→curl install→unsetの一連を型化する必要があります。

やる／やらないとどうなるか TREPRO_GITHUB_TOKENが10分以内に環境変数から消え、内部URL+署名/Hash確認の運用が成立します。任意Internet ScriptをPipe実行しないという原則も同時に守れます。

これは何か 短期Tokenを環境変数経由でinstall.shへ渡し、直後にunsetする5〜15分パートのInstall手順例です。

実行するとどうなるか Tokenが.git/configやShell履歴に残らず、内部URL+署名Hash確認とセットで安全なBootstrapが完走します。

export TREPRO_GITHUB_TOKEN='短期Token'
curl -fsSL https://INTERNAL_APPROVED_URL/install.sh | bash
unset TREPRO_GITHUB_TOKEN

内部URLと署名/Hashを確認し、任意Internet ScriptをPipe実行しない。

15〜20分

なぜここを読むか 「Install終わった」を主観で扱うと、claude本体やGitHub認証が壊れていても気付かれません。verifyを段階ごとに走らせる必要があります。

やる／やらないとどうなるか claude --version・claude doctor・gh auth status・verify.shの4コマンドで存在・健全性・認証・配布物が同時に確認でき、不具合がここで顕在化します。

これは何か claude本体・claude doctor・gh auth・verify.shを順に叩き、Install後15〜20分で配布全体の健全性を確認するVerifyシーケンスです。

実行するとどうなるか Installが終わっても壊れている要素があれば即座にFAILが出るため、新人がBronze実演へ進む前に問題を顕在化できます。

claude --version
claude doctor
gh auth status
$HOME/.local/share/trepro-harness/deploy/verify.sh

20〜30分

なぜここを読むか Install成功＝Onboarding完了ではありません。本人が安全なLabで実演し、Plan/Verify/Skill評価まで一周することが必要です。

やる／やらないとどうなるか training/meeting-assistantを使って Plan→実装→git diff→Test→/rewind→Skill評価まで通す体験となり、Bronze相当の合否を実演ベースで判定できます。

training/meeting-assistantを開く
Plan → 実装 → git diff → Test → /rewind → Skill評価

27.2 初日Safety

なぜここを読むか 初日に本番DBや顧客Fileに触らせると、AIが便利すぎる時に事故が必ず起きます。最小防御セットを初日から強制する必要があります。

やる／やらないとどうなるか 本番禁止・Bypass禁止・APIキー貼付禁止・Dry-run徹底・Diffなしcommit禁止・外部送信は承認・不明MCP禁止・Error/Log隠蔽禁止という8原則が初日から効きます。

これは何か 本番禁止・Bypass禁止・APIキー禁止・Dry-run・Diff必須・外部送信承認・不明MCP禁止・Log隠蔽禁止という8項目の初日Safetyルールです。

実行するとどうなるか 新人の初日からAIの暴走・誤操作・情報持ち出しが構造的に抑えられ、Sandboxの範囲内で安全に学習だけが進みます。

- 本番/顧客で練習しない
- Bypass禁止
- API Keyを貼らない
- 削除はDry-run/Manifest
- DiffなしCommit禁止
- 外部送信/公開/Deployは承認
- 不明MCP/Plugin禁止
- Error/Logを隠さない

27.3 認定

なぜここを読むか 認定を知識クイズで済ますと、現場でPlanやVerifyが回せない社員が量産されます。実演ベースで段位を分ける必要があります。

やる／やらないとどうなるか Bronze（安全・Plan・Diff・Checkpoint）からAdmin（Kandji・Marketplace・Audit・Release）までの4段階で、次に学ぶべき領域が個人ごとに見えるようになります。

Level 到達
Bronze 安全、Plan、Diff、Checkpoint
Silver CLAUDE.md/Skill、Fixture、HTML/MCP Read-only
Gold Agent/Hook/Policy、Evaluation、Incident
Admin Kandji、Marketplace、Audit、Release

27.4 運用Cadence

なぜここを読むか Onboardingで終わると、半年後にはVersionも知識も古くなります。Daily/Weekly/Monthly/Quarterlyの運用周期を入れる必要があります。

やる／やらないとどうなるか Daily=Update失敗・Weekly=Canary/Cost・Monthly=Harness診断/Deprecation・Quarterly=Policy/Portfolio/研修/KPIという周期で、環境と人の劣化を両方止められます。

Daily: Update失敗/SEV
Weekly: Canary/Skill候補/Cost
Monthly: Harness診断/Skill抜取/Deprecation/Version/Incident
Quarterly: Policy/MCP/Plugin/Portfolio/研修/KPI

27.5 障害表

なぜここを読むか 新人サポート問い合わせは認証・権限・Path・Versionに偏ります。症状→対処の対照表を一枚にしないと、情シスへ集中して業務が止まります。

やる／やらないとどうなるか claudeなし→Native Installer、Skill不明→Validator確認、Hook不動→Scope/Managed-only、MCP不明→Allowlistといった一対一対応で本人が解決できる比率が上がります。

症状 対処
claudeなし Native Installer
Skill不明 Name/Frontmatter/Manifest/Validator
Hook不動 Scope/Permission/Managed-only/Log
MCP不明 managed-mcp/Allowlist/OAuth
Update不動 Network/Lock/Auth/LaunchAgent
個人設定消失 Manifest所有範囲修正
Sandbox失敗 Domain/Socket/Apple Event最小例外
Context重い /context、分割、整理

27.6 Diagnostics

なぜここを読むか Support依頼時に毎回環境情報を聞き直すと、Onboarding直後の社員が心理的に詰まります。安全に拾える範囲を1コマンドで出す必要があります。

やる／やらないとどうなるか date・macOS・arch・claude/harness Version・git・gh authをRedactしながらtxtへ出力でき、会話・Secret・顧客Fileは自動添付しない安全な診断Logが成立します。

これは何か OS・arch・Claude/harness Version・git・gh authをRedact処理しつつ/tmpへ書き出すDiagnostics収集スクリプトです。

実行するとどうなるか Support問い合わせ時に必要な環境情報がワンコマンドで揃い、会話・Secret・顧客Fileを自動添付しない最小収集の原則が守られます。

{
  echo "date=$(date -Iseconds)"
  echo "macos=$(sw_vers -productVersion)"
  echo "arch=$(uname -m)"
  echo "claude=$(claude --version 2>&1)"
  echo "harness=$(cat ~/.claude/.trepro-harness-version 2>/dev/null || echo unknown)"
  echo "git=$(git --version)"
  echo "gh_auth=$(gh auth status 2>&1 | sed -E 's/(token:).*/\1 [REDACTED]/')"
} > /tmp/trepro-harness-diagnostics.txt

会話、Secret、顧客Fileを自動添付しない。
