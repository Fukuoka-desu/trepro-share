守らせたいルールは、お願いでは守られない HooksとPlugins

『AIに守らせたいルールを CLAUDE.md に書いたのに、守られない』。 これはAIの怠惰ではなく、お願いと強制を混同しているだけです。

ほとんどの人は、安全ルールを文章のお願い（Guidance）で済ませています。 ですが、本当に止めたい操作は、文章では止まりません。物理的な仕組み（Enforcement）が要ります。

Hooks と Plugins は、操作の前後に自前のチェックを差し込む仕掛けです。 危険コマンドは事前に止め、編集後に型チェックを走らせ、コミット時にテストを通す。お願いを強制へ昇格させる装置です。

この章を読み終わる頃には、『お願い』と『強制』の境界が、自分の言葉で説明できるようになります。

この章の概観 ／ 何を扱い、何ができるか

この章で扱う機能と、それで何ができるようになるか。実装に入る前に、全体像をここでつかみます。

  • PreToolUse Guard（危険操作の物理拒否）

    rm -rfや本番DB操作など、文章のお願いでは止められない操作を実行直前に拒否するHook。

    入力
    実行されようとしたコマンド、対象Path
    出力
    ブロック判定とログ、許可された操作のみの実行

  • PostToolUse Validation

    Edit/Write直後にtsc・lint・テスト等を自動実行し、編集の瞬間にエラーを検出する検証Hook。

    入力
    編集したファイル、プロジェクトの検証コマンド
    出力
    検証結果、エラー時のブロックと修正指示

  • SessionStart Hook

    セッション開始時に必要な前提・ルール・現在地ファイルを自動注入し、毎回ゼロから説明させない仕掛け。

    入力
    セッション起動イベント、注入したい文脈ファイル
    出力
    起動時自動ロードされた前提情報

  • Plugins（配布単位）

    Skills・Hooks・設定をまとめて配布する束ね単位。導入範囲と更新を組織的に管理する。

    入力
    配布したいSkill群、Hook群
    出力
    インストール可能なPluginパッケージ

  • Hook Test（Fixture検証）

    安全な入力と危険な入力のFixtureで、Hook自体の動作を単体テストする仕組み。Hookコードの品質を担保する。

    入力
    安全/危険のFixture、想定動作
    出力
    Fixture通過/拒否の検証結果

実装リファレンス

14.1 Hookは決定的な制御

なぜここを読むか CLAUDE.mdに書いた安全ルールは「お願い」止まりで、本当に止めたい操作には効きません。決定的に制御できるイベントを把握しておく必要があります。

やる／やらないとどうなるか SessionStart・PreToolUse・PostToolUse・Stop・SessionEnd・ConfigChangeを足場にすれば、危険Command拒否や編集後Typecheckなどを実行時に強制できます。Eventを知らないと強制と願望の境界が引けません。

代表Event：

これは何か Claude Codeで使える代表的なHook Eventの一覧です。起動から終了までのライフサイクルを縦に並べて把握できます。

実行するとどうなるか このEvent名を押さえれば、危険Command拒否・編集後Typecheck・公開可能Context注入など、どの瞬間に介入するかを設計時に選べます。Event未把握ではHookが書けません。

用途：

  • 危険Command拒否。
  • Edit後のTypecheck。
  • Session開始時の公開可能Context注入。
  • Goal未達時の継続。
  • 設定改変の監査。

14.2 PreToolUse Guard

なぜここを読むか `rm -rf`やForce push・Hard resetなどは、AIが文章のお願いだけでは止められない代表例です。実行直前に介入できる物理ガードが要ります。

やる／やらないとどうなるか PreToolUseでコマンド文字列を正規表現判定し、該当時に`deny`を返す構成にすれば、AIがいくら賢くてもその操作は通りません。Hookが無い環境では破壊的コマンドが繁忙期に必ず通ります。

global-config/hooks/pre-bash-guard.sh：

これは何か PreToolUse Guardの最小実装例です。jqでコマンドを抽出し、rm -rf・force push・hard resetなど危険文字列に対し`deny`を返します。

実行するとどうなるか この内容を`global-config/hooks/pre-bash-guard.sh`へ置けば、AIがどんな経緯で危険コマンドを出してもHookが拒否を返し、実行されません。文章のお願いから強制への昇格が完了します。

#!/usr/bin/env bash
set -euo pipefail

payload="$(cat)"
command="$(printf '%s' "$payload" | jq -r '.tool_input.command // empty')"

block() {
  jq -n --arg reason "$1" '{
    hookSpecificOutput: {
      hookEventName: "PreToolUse",
      permissionDecision: "deny",
      permissionDecisionReason: $reason
    }
  }'
  exit 0
}

[[ -n "$command" ]] || exit 0

if [[ "$command" =~ (^|[[:space:];|&])rm[[:space:]]+-rf([[:space:]]|$) ]]; then
  block "Recursive force deletion is blocked. Produce a dry-run manifest and request approval."
fi

if [[ "$command" =~ git[[:space:]]+push.*(--force|-f)([[:space:]]|$) ]]; then
  block "Force push is blocked by TrePro policy."
fi

if [[ "$command" =~ git[[:space:]]+reset[[:space:]]+--hard ]]; then
  block "Hard reset is blocked. Use a reversible branch or request approval."
fi

exit 0

Hook Input SchemaはVersionで変わる可能性があるため、実機Fixtureで確認する。

14.3 PostToolUse Validation

なぜここを読むか Edit/Write後にtsc・lint・テストを手動で回す運用だと、エラー検知が翌朝までずれ込み、月曜の手戻り会議の原因になります。

やる／やらないとどうなるか PostToolUseで対象拡張子にだけtypecheckを走らせれば、編集の瞬間に型エラーが検出されます。Debounce・対象Path・Timeoutを入れないと、毎Edit重くなって運用負担に振れます。

global-config/hooks/post-edit-validate.sh：

#!/usr/bin/env bash
set -euo pipefail
payload="$(cat)"
file="$(printf '%s' "$payload" | jq -r '.tool_input.file_path // empty')"
[[ -n "$file" ]] || exit 0

root="${CLAUDE_PROJECT_DIR:-$PWD}"
case "$file" in
  *.ts|*.tsx)
    if [[ -f "$root/package.json" ]]; then
      cd "$root"
      if npm run -s typecheck >/tmp/trepro-typecheck.log 2>&1; then
        exit 0
      fi
      summary="$(tail -n 30 /tmp/trepro-typecheck.log)"
      jq -n --arg context "Typecheck failed after editing $file:\n$summary" '{
        hookSpecificOutput: {
          hookEventName: "PostToolUse",
          additionalContext: $context
        }
      }'
    fi
    ;;
esac

重いCheckを毎Editで実行しすぎない。Debounce、対象Path、Timeoutを入れる。

14.4 SessionStart

なぜここを読むか 顧客案件ごとに必要な前提・公開可能Runbookが違うのに、毎回会話で説明していると忘れ・取り違えが起きます。SessionStart注入が有効です。

やる／やらないとどうなるか Pathで顧客判定し、公開可能なRunbookの場所だけをHook出力で注入すれば、起動の瞬間に必要な前提が揃います。秘密値を出力に埋め込むと別事故になるので分離が必須です。

顧客Pathを判定し、公開可能なRunbookの場所だけを注入する。秘密値をHook出力へ埋め込まない。

これは何か SessionStart Hookで顧客Pathを判定し、公開可能Runbookの場所だけを注入する例です。秘密値はHook出力に乗せません。

実行するとどうなるか この分岐を仕込めば、対象顧客ディレクトリで起動した瞬間に必要な前提と禁止事項がAIへ届きます。毎回口頭で前提共有する手間が消え、取り違えも防げます。

case "$PWD" in
  */clients/acme/*)
    echo "Client context: ACME. Read docs/clients/acme/public-runbook.md. Never access production credentials."
    ;;
esac

14.5 Stop Hookと/goal

なぜここを読むか テスト成功・型チェック成功・Build成功まで本当に走り切らせたい場面で、AIが途中で止まることがあります。`/goal`と停止判定を組むのが現実解です。

やる／やらないとどうなるか 「最大20ターン」「test/typecheck/build成功」「README整備」のような上限と完了条件を併記すれば、暴走と早期停止を同時に抑えられます。上限を入れないと無限ループの危険があります。

これは何か Stop Hookと組み合わせる`/goal`指示テンプレートです。継続条件と最大ターン上限を同時に明示します。

実行するとどうなるか この書式で渡せば、test/typecheck/build成功とREADME整備を満たすまでAIが自律的に継続します。上限が無いと無限ループ化し、上限だけだと早期撤退する事故が起きます。

/goal 次を満たすまで継続。ただし最大20ターン:
- npm test成功
- npm run typecheck成功
- npm run build成功
- READMEに起動手順

上限と失敗Reportを必ず入れる。

14.6 Hook Security

なぜここを読むか Hookは強力なだけにバグるとそれ自体が事故源になります。stdin汚染・Path Traversal・Secrets混入を防ぐSecurity項目を順守する必要があります。

やる／やらないとどうなるか stdin不信用・変数Quote・絶対Path化・Secrets除外・Timeout・Safe failure・Fixture Test・署名配布を全部入れれば、Hookを安全に長期運用できます。1つ抜けると環境破壊や情報漏えいに直結します。

これは何か Hook Securityチェックリストです。Quote・Path Traversal拒否・絶対Path化・Secrets除外・Timeout・Safe failure・Fixture Test・署名配布を網羅します。

実行するとどうなるか この9項目を全部潰したHookだけ配布対象にすれば、Hookそれ自体が事故源になるリスクを最小化できます。1つ抜けるだけで環境破壊や情報漏えいの入口になります。

- stdinを信頼しない
- Shell変数をQuote
- Path Traversalを拒否
- 絶対PathまたはCLAUDE_PROJECT_DIR
- Secrets/.gitを除外
- Timeout
- Safe failure
- Fixture Test
- 変更をReview/署名配布

14.7 Hook Test

なぜここを読むか Hookは「危険を拒否」と「正常を通す」の両方を満たす必要があります。誤検知も誤通過もFixtureテストで担保しないと本番で破綻します。

やる／やらないとどうなるか 正常Command許可・rm -rf拒否・force push拒否・誤検知防止・空Input安全・壊れたJSON安全の6ケースを揃えれば、Hookの品質が機械的に検証できます。Fixture無しで配布すると現場で初めて欠陥が見つかります。

これは何か Hook Fixture Testの最小ケース集です。許可・拒否・誤検知防止・空Input・壊れたJSON・破壊的引数の6パターンで動作を検証します。

実行するとどうなるか このFixtureを通したHookだけ配布すれば、本番で「危険を見逃す」「正常を止める」事故を構造的に防げます。テスト無しのHookは現場で初めて欠陥が露出します。

正常Command → 許可
rm -rf → 拒否
force push → 拒否
文字列中の単なるrm → 誤検知しない
空Input → Errorにしない
壊れたJSON → 安全側

14.8 Plugins

なぜここを読むか Skill単体・Skill+Agent+Hook・全社標準・顧客固有では、配布方法と更新管理の最適解が違います。状況別の選び方を持っておく必要があります。

やる／やらないとどうなるか 1 SkillならSkill単体、複合ならPlugin、全社標準ならManaged Marketplace、顧客固有なら顧客別Pluginと使い分ければ、配布範囲と更新ルートが整理されます。提供元・実行Code・MCP・Rollbackは必ずReviewします。

PluginはSkills、Agents、Hooks、MCP等を配布単位にまとめる。

状況 選択
1 Skillだけ Skill単体
Skill + Agent + Hook Plugin
全社標準 管理Marketplace / Force-enabled Plugin
顧客固有 顧客別PluginまたはProject設定

提供元、実行Code、MCP、更新方式、RollbackをReviewする。
