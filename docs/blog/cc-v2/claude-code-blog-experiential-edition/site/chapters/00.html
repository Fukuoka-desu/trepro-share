第0章

読むだけの学習を、卒業する 教科書を「実行システム」にする

『AI研修、受けたのに、自席で再現できない』。 これ、ほとんどの人が一度は通る挫折ポイントです。 便利なツールには触れた。でも、自分の仕事に持ち帰れない。

ほとんどの教科書は『知識』を渡してきます。 この章で作るのは、知識じゃなく『進める単位』です。 読んだ／使った／検証した、を1ステップに圧縮した『実行できる教科書』（Lesson）。役割を非エンジニア・開発者・管理者の3つに分け、最初に作るゴールと合格証拠を1行で決めます。

この章を読み終わる頃には、『何を作れば自分は今日進んだと言えるか』が一行で言えるようになります。

実装リファレンス

0.1 3つの学習ルート

なぜここを読むか 学習者の役割によって、必要な順序と最初に作るものが違うから。全員に同じカリキュラムを当てても、誰かは必ず脱落する

やる／やらないとどうなるか 自分のルートを最初に決めると、無関係な章を飛ばせて到達点も明確になる。学習が『今日進んだ』と言える単位に変わる

非エンジニアルート

これは何か 非エンジニア向けの章順マップ。全体像から始めて安全・Project・会議Assistantを経由する

実行するとどうなるか 業務担当者がどの章を飛ばさず読めば成果に届くか一発で分かる。学習のロードマップとして機能する

それぞれの意味

全体像
Claude Codeが何でどう動くかの俯瞰。最初に持つことで、後の各章が地図のどこに対応するか迷わなくなる
安全
事故を起こさない設定と練習場の話。ここを先に抑えないと、便利機能を試す前に本番データを壊す
Project Folder
案件ごとにAIの作業範囲を区切る単位。これがないとAIが別案件のファイルまで触る事故が起きる
会議Assistant
文字起こしから議事録・タスク・HTMLを出す最初の成果物。動くものを90分で持ち帰る体験用題材
HTML/Cowork
結果を共有できる形に整える段階。Cowork面でレポートやArtifactを作る型を覚える
実務レシピ
週報・調査・整理など定型業務のテンプレ集。会議Assistantで掴んだ型を他業務へ横展開する
全体像 → 安全 → Project Folder → 会議Assistant → HTML/Cowork → 実務レシピ

到達点：ローカル資料から議事録、週報、レポート、簡易アプリを作り、危険操作を避けられる。

開発者ルート

これは何か 開発者向けの章順マップ。標準ループ・権限・Git・拡張・MCP・自律化まで段階で並ぶ

実行するとどうなるか 実装者が再現可能な開発フローを身につける順序が固まる。MCPと自律化を後半に置く理由も明確になる

それぞれの意味

標準ループ
Explore→Plan→Implement→Verifyの繰り返し型。これを身体化しないと毎回違う手順で再現できない
Permissions/Sandbox
AIに許す操作と作業範囲の物理境界。設定で破壊コマンドを入口で止め、本番領域への波及を切る
Git/Context
戻れる仕組みと会話文脈の制御。コミットで巻き戻し点を残し、Contextを意図的に切って混入事故を防ぐ
Skills/Agents/Hooks
再利用手順・専門家分業・自動強制装置。会話の知見を会話の外へ昇格させて属人化を解消する
MCP
Claude Codeに外部システムを接続する規格。社内DocsやAPIを安全に呼べるようにする後半トピック
自律化
成功条件と停止条件を持つ無人実行。設計を誤ると無限ループになるため、Loopを身につけた後で扱う
標準ループ → Permissions/Sandbox → Git/Context → Skills/Agents/Hooks → MCP → 自律化

到達点：複数ファイルの実装、テスト、修正、レビュー、長時間タスクを再現可能に運用できる。

管理者ルート

これは何か 管理者向けの章順マップ。安全境界→ポリシー→配布→Governance→Telemetryで全社運用まで進む

実行するとどうなるか 情シス・AI推進が全社配布と監査をどの順で設計すればよいかが見える。個別操作より仕組み側に時間を使える

それぞれの意味

安全境界
全社で守る最小権限・隔離・監査の枠。これを先に決めないと、配布後にユーザーごとに穴が開く
Managed Policy
管理者が一括強制する設定層。ユーザーが外せないCLAUDE.mdや権限定義をここへ置いて全社統制する
Harness配布
設定・スキル・Hookをまとめて配るパッケージ運用。手作業配布だと環境差で再現性が崩れる
Skill Governance
Skillの登録・審査・更新ルール。野良Skill乱立を防ぎ、誰が何を変えたかを監査可能にする
Telemetry/Incident
利用ログと事故対応の仕組み。使用状況を観測し、問題発生時に原因と影響範囲を即特定できる体制
安全境界 → Managed Policy → Harness配布 → Skill Governance → Telemetry/Incident

到達点：全社員へ同じ環境を配布し、品質とセキュリティを強制・監査できる。

0.2 Lessonの標準形

なぜここを読むか 教科書を読み物のままにすると、再現も検証もできずに『分かった気』で終わる。Lessonの単位を共通化する必要がある

やる／やらないとどうなるか 前提・成果物・検証・版情報がひと塊で揃うから、自分も他人も同じ手順で再実行できる。資産化できる学習になる

将来アプリ化しやすいよう、各Lessonを次の単位で管理する。

これは何か Lesson標準形のYAML。id・level・前提・成果物・チェック・検証日・参照元を1ファイルで管理する

実行するとどうなるか Lessonをコードとして扱える状態になる。後でアプリ化・進捗管理・依存解決を機械的に処理できる

それぞれの意味

lesson_id
Lessonを一意に識別する文字列。後でアプリ化・依存解決・進捗追跡をするときの主キーになる
status
stable/draft/deprecatedなど運用状態。読者が安心して取り組んでいいかを一目で判断できる
level
beginner/intermediate/advancedの難易度。自分のルートに合うLessonを順序立てて選べるようにする
estimated_minutes
想定所要時間。スキマ時間に1Lesson回せるか、まとまった時間が要るかを事前に見積もれる
prerequisites
先に終えるべきLessonのid。順序を守らないと前提知識が無くて詰まる地点を明示する
outcomes
学習後にできるようになる行動。完了したかどうかを自分で判定する基準として使う
artifacts
Lesson終了時に残るファイル群。後日見返したり再開したりするときの物的証拠になる
checks
合格判定に使う観点。テスト結果や画面証拠など、提示すべき証拠の種類を先に決めておく
verified_at
最終検証日。半年で仕様が変わるので、いつ時点の動作確認かを残さないと古い手順で詰む
source_topics
出典となった動画・公式Doc・原論文。なぜこの手順なのかを後追いできる根拠リンクとして機能する
lesson_id: cc-core-plan-verify
status: stable
level: beginner
estimated_minutes: 25
prerequisites:
  - cc-core-project-folder
outcomes:
  - 計画と実装を分けられる
  - 完了条件を検証可能に書ける
artifacts:
  - requirements.md
  - implementation-plan.md
checks:
  - テスト結果または画面証拠を提示できる
verified_at: 2026-06-22
source_topics:
  - video-v01-plan-mode
  - video-v04-meeting-assistant
  - official-best-practices

0.3 各章の統一フォーマット

なぜここを読むか 章ごとに章の構成がバラバラだと、読者は毎章で『どこを見ればいい？』の迷子になる。全章を同じ流れに統一する

やる／やらないとどうなるか 目的→概念→手順→コピペ→検証→事故防止→発展の固定順だから、章を移動しても探す位置が変わらず学習速度が上がる

これは何か 章ごとの統一フォーマット。目的→概念→手順→コピペ→検証→事故防止→発展課題の7段で並ぶ

実行するとどうなるか どの章でも『次はどこを読めばいい？』が迷わない。読者の認知負荷が下がり、学習速度が一定になる

それぞれの意味

目的
この章で何ができるようになるか。最初にゴールが見えないと、読みながら『なぜ今これを読んでいる』が分からなくなる
概念
目的を達成するために必要な考え方の定義。手順だけ覚えると応用が利かず、半年後に陳腐化する
手順
概念を実際の操作へ落とした順序。ここを読めば手を動かす段取りが分かる
コピペ例
そのまま貼り付けて試せるコマンドや設定。最初の一歩を踏み出す摩擦をゼロにする役割
検証方法
うまくいったかを機械的に確かめる手順。『動いたっぽい』を排除して合否を客観化する
事故防止
踏みやすい地雷と避け方。先に知らないと初日に致命傷を負う領域をまとめる
発展課題
次に挑戦すべき応用。基本を終えた人が腐らず次のレベルへ進めるよう橋を渡す
目的
  ↓
概念
  ↓
手順
  ↓
コピペ例
  ↓
検証方法
  ↓
事故防止
  ↓
発展課題

0.4 変わりにくい原則と変わりやすい情報を分ける

なぜここを読むか ツールの仕様は半年で変わるが、安全や計画分離のような原則は変わらない。両者を混ぜると改訂のたびに全章が腐る

やる／やらないとどうなるか 本文には原則だけを残し、Model名やSlash Commandは別Registryへ逃がす。古い情報で迷わなくなり保守も楽になる

本文へ固定

  • Planと実装を分ける。
  • 完了条件を先に定義する。
  • 最小権限、隔離、可逆性、検証、監査。
  • 作成者と評価者を分ける。
  • GitとCheckpointを併用する。
  • 会話の知見をSkillやRuleへ昇格させる。

Feature Registryへ分離

  • Model名と利用可否。
  • 料金と上限。
  • UI名、Slash Command、対象Plan。
  • Beta/Preview/Suspended状態。
  • 最低Claude Code Version。
