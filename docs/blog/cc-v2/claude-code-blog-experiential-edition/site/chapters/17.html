第17章

デザインカンプを、その場で動くUIに変える Claude Designから実装へ

『デザインカンプを受け取ったのに、コードに起こすのに 3 日かかる』。 これは技術力の問題ではなく、橋渡しの仕組みが無いだけです。

ほとんどの人は、Figma のキャンプを手で読み、HTML/CSS を書き、JS をつなぎ、確認に何往復もしています。 ですが、Claude Design からそのまま実装へ渡せば、その往復は半分に圧縮できます。

この章で扱うのは、デザインのレイヤー・トークン・コンポーネントを、AIが解釈できる形で渡すための型です。 カンプを情報として渡せば、AI はそれを生きた UI へ翻訳します。

この章を読み終わる頃には、『デザインから実装』の距離が、1日へ縮まります。

この章の概観 ／ 何を扱い、何ができるか

この章で扱う機能と、それで何ができるようになるか。実装に入る前に、全体像をここでつかみます。

  • Wireframe / Design System / Prototype / Code 分離

    情報・見た目・動作・実装を別段階で確定する4分離設計。修正コストを段階で閉じ込める。

    入力
    作りたい画面の目的と利用者
    出力
    段階ごとに確定した情報設計・スタイル・プロトタイプ・コード

  • Design SystemとProject分離

    色・タイポ・余白の共通システムを別Projectに固定し、各画面はそれを参照する形に揃える運用。

    入力
    ブランドカラー、フォント、共通余白ルール
    出力
    再利用可能なDesign System Projectと参照Path

  • Claude Design 標準Flow

    白黒WireframeからDesign System適用、実装まで進める標準的な制作フロー。各段階の決定事項を記録する。

    入力
    ラフ案、要件、対象ユーザー
    出力
    段階別の決定記録と実装ドラフト

  • Codeへ渡すPromptとSync Rule

    Designの決定をコード側へ漏れなく伝える受け渡しプロンプトと、後で齟齬を起こさない同期ルール。

    入力
    Designでの確定事項、参照コンポーネント
    出力
    実装プロンプトとSync用ルール

  • 公開前チェック

    レスポンシブ・アクセシビリティ・リンク・秘密情報・ブランド整合をCode側で再検証する公開前ゲート。

    入力
    実装済みページ、公開予定
    出力
    5項目のチェック結果と公開可否判定

実装リファレンス

17.1 分離する

なぜここを読むか 情報構造・見た目・実装を同時に揺らすと、どこを直しても他の段階に揺り戻しが発生します。

やる／やらないとどうなるか Wireframe・Prototype・Design System・実装を別段階で確定すれば、修正コストをその段階の中に閉じ込められます。

これは何か デザインから実装までの段階を一列に並べ、検証で締める標準フローです。

実行するとどうなるか どの段階の決定で詰まっているかを、後から指差して指摘できる構造になります。

それぞれの意味

要件
誰が何のために使うか、何ができれば成功かを言葉で固める段階です。ここがぶれると、後段の作業がすべて作り直しになります。
Wireframe
装飾なしで情報の配置と順序だけを決める段階です。見た目に気を取られず、情報の抜けや順序ミスをここで潰せます。
Prototype
実装前に画面遷移と操作感を試せる試作品の段階です。動かして初めて気付く違和感を、コードを書く前に拾えます。
Design System
色・タイポ・余白・部品の共通ルールを当てる段階です。複数画面のばらつきを構造的に揃え、ブランドの一貫性を守れます。
実装
確定した要件・構造・スタイルをコードへ落とす段階です。前段で決まったことを翻訳する作業に集中でき、判断と実装が混ざりません。
視覚/Accessibility検証
見た目とアクセシビリティを最後に再検査する段階です。実装で見落とした崩れや読みにくさを、公開前にここで拾えます。
要件 → Wireframe → Prototype → Design System → 実装 → 視覚/Accessibility検証

Wireframeでは装飾より、情報の順序とActionを決める。

17.2 Design SystemとProject

なぜここを読むか 個別画面ごとに色やフォントが微妙に違うと、ブランドの一貫性が静かに壊れていきます。

やる／やらないとどうなるか Design SystemをProjectとは別物として扱えば、各画面はそれを参照するだけで揃い、横展開のコストが下がります。

概念 内容
Design System 色、文字、余白、Component、状態、Brand Rule
Project Design Systemを使う具体物

17.3 最小構造

なぜここを読むか Design Systemを概念だけで持つと、誰がどのトークンを更新するかがふんわりして運用が止まります。

やる／やらないとどうなるか tokens/・components/・assets/・examples/という最小フォルダ構造を持つことで、更新責任と参照経路が物理的に明確になります。

これは何か Design Systemの最小フォルダ構造で、tokens・components・assets・examplesを分けたレイアウトです。

実行するとどうなるか 更新責任と参照Pathが物理的に固定され、複数Projectから安全に再利用できます。

それぞれの意味

tokens/
色・タイポ・余白・角丸といった最小単位の値を置く場所です。ここを正本にすることで、各画面の調整が個別実装で散らばらなくなります。
components/
Button・Card・Form・Navigationなど再利用するUI部品を置く場所です。同じ部品を画面ごとに作り直す重複を構造的になくせます。
assets/
ロゴ・アイコン・画像などの静的素材を置く場所です。配布元が一箇所に固定されることで、古い素材が紛れ込む事故を減らせます。
examples/
Design Systemを使った実例画面を置く場所です。トークンや部品の使い方が文書だけでは伝わらない時の、生きた参照になります。
design-system/
├── README.md
├── tokens/
│   ├── color.json
│   ├── typography.json
│   ├── spacing.json
│   └── radius.json
├── components/
│   ├── button.md
│   ├── card.md
│   ├── form.md
│   └── navigation.md
├── assets/
└── examples/

17.4 標準Flow

なぜここを読むか いきなり実装から作ると、情報の欠落や順序ミスが見た目の上に固定されてしまいます。

やる／やらないとどうなるか Wireframe→Prototype→Design System→Code→差分確認まで標準フロー化すると、各段階の決定事項を後から追える形で残せます。

  1. Wireframe作成。
  2. 情報の欠落・重複・順序Review。
  3. Prototypeで遷移確認。
  4. Design System適用、Desktop/Mobile確認。
  5. Claude Codeへ渡す。
  6. Component対応表、Token、Test計画。
  7. 実装後のScreen差分確認。
  8. 汎用ComponentをDesign Systemへ戻す。

現行の /design と /design-sync を /help と公式情報で確認する。

17.5 Codeへ渡すPrompt

なぜここを読むか Designをそのままコード化させると、状態・レスポンシブ・アクセシビリティの抜けが実装後に露見します。

やる／やらないとどうなるか 実装計画の必要項目を先に挙げさせるプロンプトを使えば、Hover/Focus/Error/Loadingまで含めた完成度で着手できます。

これは何か Designを受け取った直後に、画面対応・Token・状態・Accessibility・Acceptanceを先に書かせる指示文です。

実行するとどうなるか 着手前に抜け項目が洗い出され、実装後にHover/Error/Loadingが見つかる手戻りを防げます。

それぞれの意味

画面/Component対応
Designのどの画面・部品が、実装側のどのComponentに対応するかを先に明示します。対応表がないと、同じ部品を別実装で二重に作る事故が起きます。
Token/CSS変数
色や余白を変数として一元化する設計を先に決めます。後からブランド変更が入っても、置換すべき場所が散らからずに済みます。
Responsive
画面幅ごとのレイアウト切り替え方針を実装前に決めます。スマホで崩れて作り直すような大きな手戻りを防げます。
Hover/Focus/Disabled/Error/Loading
見た目だけでなく操作中・無効・失敗・読み込み中の状態をあらかじめ列挙します。実装後に状態が抜けて発見されるパターンを潰せます。
Accessibility
コントラスト・キーボード操作・代替テキストなどの要件を先に書きます。あとから足すと構造改修が必要になる項目を、設計段階で確保できます。
Mock範囲
実データの代わりに使うダミーデータの範囲を決めます。本番接続前にUIだけ先に確認でき、依存待ちで止まらない開発になります。
Acceptance Test
完成と見なすための判定条件を先に書き出します。実装後に『どこまでやれば終わりか』が曖昧になり延々と直し続ける状態を防げます。
Designを写経せず、まず次を含む実装計画を作る。
- 画面/Component対応
- Token/CSS変数
- Responsive
- Hover/Focus/Disabled/Error/Loading
- Accessibility
- Mock範囲
- Acceptance Test
承認前に変更しない。

17.6 Sync Rule

なぜここを読むか Design側とCode側が両方とも『正本』を主張すると、トークンや共通部品が二重メンテで腐ります。

やる／やらないとどうなるか Git側を正本にし、採用変更だけCodeへ流す同期ルールを敷くと、デザインと実装の乖離が時間とともに発生しません。

  • Token/共通ComponentはGit側を正本。
  • Designで探索し、採用変更だけCodeへ。
  • Codeで生まれた汎用部品はReview後に戻す。
  • Sync前後にDiff保存。
  • Beta変換に備えGit Commit。

17.7 公開前

なぜここを読むか Designレビューを通っても、未実装ボタン・認証漏れ・Contrast不足が公開直前に出るのはよくあります。

やる／やらないとどうなるか Code側で再検査する8項目を通せば、見た目の確認ではなく動作と安全性の確認として公開判断ができます。

これは何か 公開前ゲートとして、遷移網羅・Responsive・Contrast・自動Accessibility検査まで通す8項目です。

実行するとどうなるか Design承認とは別軸で、Code側でしか分からない動作上の不備を最後に拾えます

それぞれの意味

全遷移
ページ間のすべての遷移が成立することを確認します。Designで承認されても、リンク切れや行き止まりは実装側でしか発見できません。
未実装Buttonの明示
押せるが何も起きないボタンが残っていないかを点検します。残っていると、利用者は『壊れている』と判断してしまいます。
Responsive
代表的な画面幅で破綻なく表示できることを確認します。実機で見て初めて気付く崩れを、公開前にここで止めます。
Focus/Error/Loading/Success
操作中・失敗・読み込み中・成功の各状態が正しく見えることの確認です。状態の見落としが、ユーザーの操作迷子を生みます。
Contrast
文字と背景の色差が読める水準にあることを確認します。明るい場所や色覚特性のあるユーザーが取り残されないラインを担保します。
自動Accessibility検査
ツールによるアクセシビリティ自動チェックを通すことです。人の目視では拾いにくい構造的な不備を機械的に検出できます。
Design差分Screenshot
Designと実装のスクリーンショットを並べて差分を残すことです。後から『どっちが正か』を確認する根拠として効きます。
環境変数/認証/Log
本番公開に必要な環境変数・認証設定・ログ出力が揃っていることの確認です。Design承認の枠外で発生しがちな、本番でだけ顕在化する不備を拾えます。
