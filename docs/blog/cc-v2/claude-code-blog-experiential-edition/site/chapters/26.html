第26章

誰が、何を、いつ。後から辿れる現場にする Managed Policy、Security、Observability

『AI が裏で何をしたか、誰も把握していない』。 これは便利すぎる時に起きる、組織にとって最も静かな事故です。

ほとんどの人は、AIの動きをログとして残していません。 ですが、暴走・誤操作・情報持ち出しは、ログがなければ後追いできません。

この章で扱うのは、Managed Policy・Security・Observability の3点で AI の動きを観測可能にする設計です。 誰が・何を・いつ・どこで実行したかを、後から辿れる形で残します。

AI の暴走は、ログから見える。可視化は、信頼を裏付ける土台です。

実装リファレンス

26.1 Threat Model

なぜここを読むか 想定する脅威を並べないまま安全策を積むと、抜けや過剰の判断ができません。誤操作・Secret漏えい・Injection・Supply chain・設定改変・過剰自律・古い配布・過収集の8脅威を出発点に置く必要があります。

やる／やらないとどうなるか 各脅威に対応するdeny・Hook・Sandbox・Managed Settings・Budget・Version Inventory等の対策がマトリクスで見え、抜けと重複を同じ表で議論できるようになります。

脅威 例 対策
誤操作 rm/force push deny、Hook、Git、Sandbox
Secret漏えい env読取/外部送信 denyRead、Network制限、Scan
Injection Web/Issue命令 Untrusted扱い、Write分離
Supply chain 悪性Plugin/MCP Marketplace Allowlist、Review
設定改変 Safety解除 Managed Settings、ConfigChange
過剰自律 Goal無限 Budget、Timeout、Kill
古い配布 端末差 Version Inventory
過収集 Log人物評価 Opt-in、Minimize、Retention

26.2 Managed Settings例

なぜここを読むか 「.envを読まない」「force pushしない」を文章で書いても、社員端末では外せます。managed-settings.jsonで実行時に物理強制する必要があります。

やる／やらないとどうなるか defaultMode plan・bypass無効化・denyにrm/force push/secret・askに本番Deploy・Sandbox network allowedDomainsまで一式が並び、文書ではなく実装で安全が立ち上がります。

これは何か Plan既定・bypass無効化・deny/ask/allow・Sandboxのfilesystem/network・managedRulesOnly・MCP Allowlistまでを1ファイルへ束ねたManaged Settingsの完全例です。

実行するとどうなるか 個人端末側でPermissionを緩めたり、危険コマンドを実行したり、許可外DomainへNetwork接続することが、実行時に物理ブロックされるようになります。

それぞれの意味

defaultMode: plan
全プロジェクト既定をPlan Modeへ固定する設定です。便利すぎる時の即時実行事故を初期状態で抑えられます。
disableBypassPermissionsMode
--dangerously-skip-permissionsを物理的に無効化する設定です。社員が自力でSafetyを外せなくなります。
deny
Read(./.env)やforce push・rm -rf・git reset --hardといった危険操作を実行前に止める拒否リストです。事故の主因を一括ブロックします。
ask
git push・gh pr merge・gcloud run deploy・terraform applyなど本番系を明示承認制にする設定です。誤Deployの最後の関門となります。
allow
lint/test/typecheck/diff/statusなど安全操作を確認なしで通すホワイトリストです。Prompt連発の摩擦を減らせます。
sandbox.filesystem
~/.ssh・~/.awsのReadや/etc・/usr/local/binのWriteを禁ずるFS隔離です。鍵流出と環境破壊を分離します。
sandbox.network / allowedDomains
github.com・api.github.com・registry.npmjs.org・claude.aiだけにNetworkを限定する設定です。許可外への外部送信を物理的に塞げます。
allowManagedPermissionRulesOnly
Project独自Ruleを禁じ、Managed Policyだけを有効化する強制です。組織方針が局所設定で骨抜きにされる事故を防ぎます。
allowedMcpServers
github・trepro-docsだけを承認するMCP Allowlistです。社員追加の任意MCPが起動段階で拒否されます。
cleanupPeriodDays / companyAnnouncements
ローカル蓄積データの掃除期間と組織からの注意文言です。長期保持リスク低減と注意喚起を同居させます。
{
  "$schema": "https://json.schemastore.org/claude-code-settings.json",
  "permissions": {
    "defaultMode": "plan",
    "disableBypassPermissionsMode": "disable",
    "deny": [
      "Read(./.env)",
      "Read(./.env.*)",
      "Read(./secrets/**)",
      "Read(//Users/*/.ssh/**)",
      "Read(//Users/*/.aws/**)",
      "Bash(rm -rf *)",
      "Bash(git push --force*)",
      "Bash(git reset --hard*)"
    ],
    "ask": [
      "Bash(git push *)",
      "Bash(gh pr merge *)",
      "Bash(gcloud run deploy *)",
      "Bash(terraform apply *)"
    ],
    "allow": [
      "Bash(git status)",
      "Bash(git diff *)",
      "Bash(npm run lint *)",
      "Bash(npm run test *)",
      "Bash(npm run typecheck *)"
    ]
  },
  "sandbox": {
    "enabled": true,
    "failIfUnavailable": true,
    "autoAllowBashIfSandboxed": false,
    "allowUnsandboxedCommands": false,
    "filesystem": {
      "denyRead": ["~/.ssh", "~/.aws", "~/.config/gcloud", "~/.kube"],
      "denyWrite": ["/etc", "/usr/local/bin"],
      "allowManagedReadPathsOnly": true
    },
    "network": {
      "allowedDomains": ["github.com", "api.github.com", "registry.npmjs.org", "claude.ai"],
      "deniedDomains": [],
      "allowManagedDomainsOnly": true
    }
  },
  "allowManagedPermissionRulesOnly": true,
  "allowManagedMcpServersOnly": true,
  "allowedMcpServers": [
    {"serverName": "github"},
    {"serverName": "trepro-docs"}
  ],
  "deniedMcpServers": [
    {"serverName": "filesystem-unrestricted"}
  ],
  "strictPluginOnlyCustomization": ["hooks", "mcp"],
  "cleanupPeriodDays": 20,
  "companyAnnouncements": [
    "本番変更・外部送信・削除は必ず人間承認を取得してください。"
  ]
}

注意：

  • allowManagedPermissionRulesOnlyはProject独自Ruleも止める。
  • Hooks/MCPをStrictにするならManaged領域またはForce-enabled Pluginへ移す。
  • allowManagedHooksOnlyは現行User配布Hookと両立しない。Plugin化後に有効化。
  • Network DomainをCanaryで洗い出す。

26.3 Drop-in

なぜここを読むか Managed Settingsを1ファイルに詰めると、Security・Telemetry・部署別を別チームが触りたい時に衝突します。Drop-inで分割する必要があります。

やる／やらないとどうなるか managed-settings.d/配下に10-security.json・20-telemetry.json・30-department-mm.jsonと番号付きで並べられ、レイヤ別に責任分担しつつ統合した結果として強制されるようになります。

これは何か managed-settings.d/配下に10-security.json・20-telemetry.json・30-department-mm.jsonを並べ、領域別Policyを分割管理するDrop-inの配置です。

実行するとどうなるか Security担当・Telemetry担当・部署別管理者が別ファイルを並走編集でき、統合結果として組織Policyが強制される運用が成立します。

それぞれの意味

managed-settings.json
土台となる最上位Policyファイルです。Drop-inファイルとマージされる起点になり、ここを置かないと後段配置が無効化します。
10-security.json
Security担当が触るdeny/ask/sandbox等のレイヤです。番号10で最初に読まれ、安全策が最優先で適用されます。
20-telemetry.json
Telemetry担当が触るLog/収集設定のレイヤです。Securityレイヤを上書きしない順序で観測項目を追加できます。
30-department-mm.json
MM部署など現場が触る軽度のカスタマイズレイヤです。最後に読まれ、安全と監視を壊さない範囲で部署色を加えます。
managed-mcp.json
MCP Allowlist本体ファイルです。settings側の制御と組み合わせて、承認済みServerだけ接続可能な状態を作ります。
/Library/Application Support/ClaudeCode/
├── managed-settings.json
├── managed-settings.d/
│   ├── 10-security.json
│   ├── 20-telemetry.json
│   └── 30-department-mm.json
└── managed-mcp.json

26.4 Managed MCP

なぜここを読むか MCPの追加を社員任せにすると、悪性ServerやFile直書きSecretが混ざります。Allowlistで固定し、Secretは環境変数経由にする必要があります。

やる／やらないとどうなるか managed-mcp.jsonでgithubとtrepro-docsだけ有効化し、TREPRO_DOCS_TOKENを環境変数参照に揃え、claude mcp addで追加しても拒否される運用が固まります。

{
  "mcpServers": {
    "github": {
      "type": "http",
      "url": "https://api.githubcopilot.com/mcp/"
    },
    "trepro-docs": {
      "type": "stdio",
      "command": "/usr/local/bin/trepro-docs-mcp",
      "args": ["--mode", "read-only"],
      "env": {"TREPRO_DOCS_TOKEN": "${TREPRO_DOCS_TOKEN}"}
    }
  }
}

FileへSecret直書き禁止。MCP無効化：

{"mcpServers": {}}

claude mcp list
claude mcp add --transport http test https://example.com/mcp

固定管理が有効なら追加は拒否される。

26.5 Hook Block Event

なぜここを読むか Hookで止めただけでは「何回・誰の端末で・どのRuleが止めたか」が分かりません。Block Eventを構造化Logにして観測可能にする必要があります。

やる／やらないとどうなるか timestamp・device_id（Hash済）・harness_version・rule_id・command_hashが揃い、Command全文や顧客名は中央へ送らないという観測と最小収集を両立できます。

{
  "timestamp": "2026-06-22T10:15:00+09:00",
  "device_id": "hashed-device-id",
  "harness_version": "1.8.2",
  "hook": "pre-bash-guard",
  "rule_id": "BLOCK_FORCE_PUSH",
  "action": "blocked",
  "command_hash": "sha256:...",
  "project_class": "internal"
}

Command全文や顧客名を中央へ送らない。

26.6 指標

なぜここを読むか Block件数だけ追うと、誤検知や利用低下が見えません。Update成功率・Skill失敗・Cost・Tokenまで広く測る必要があります。

やる／やらないとどうなるか Version分布・Update成功率・Permission Prompt・Skill失敗・MCP接続失敗・Token/Cost概算が並ぶことで、安全と利便性のバランスを数値で判定できます。

  • Claude/Harness Version分布。
  • Update成功率/遅延。
  • Permission Prompt/Hook Block。
  • Skill利用/失敗。
  • MCP接続失敗。
  • Goal/Workflow Timeout。
  • Token/Cost概算。

26.7 Retention

なぜここを読むか Logは長く広く持つほど、情報持ち出しと過収集のリスクが上がります。種別ごとの保持期間と権限を決め切る必要があります。

やる／やらないとどうなるか Update Log 30日・Hook Metadata 90日・Command全文は原則収集なし・Incidentは法務基準という保持基準が固定され、Retentionと閲覧権限が監査可能になります。

これは何か Update Log 30日・Hook Metadata 90日・Command全文は収集なし・Skill評価は存続中+1年・Incidentは法務基準というRetention表です。

実行するとどうなるか ログの保持期間と権限が種別ごとに固定され、長すぎる保持・広すぎる閲覧という過収集リスクが運用ルールで抑え込まれます。

それぞれの意味

Update Log 30日
Update成功・失敗履歴の保持期間です。1ヶ月分あれば異常検知に十分で、長期保持の過収集リスクを抑えます。
Hook Metadata 90日
Hook Block Eventの保持期間です。四半期分でトレンド分析が回せ、誤検知Ruleの改善判断に使えます。
Command全文 原則収集なし
Command本文を中央へ送らないという原則です。Hash運用とセットで、顧客名・Secretの流出窓口を塞ぎます。
Skill評価 Skill存続中+1年
評価結果の保持期間です。Skillが現役の間と廃止後1年は監査に使え、改廃判断の根拠を残せます。
Incident 法務/Security Policy
Incident記録は法務・社内Security Policyに従う原則です。事故対応の証跡を制度側の基準で扱えます。
Update Log 30日
Hook Metadata 90日
Command全文 原則収集なし
Skill評価 Skill存続中+1年
Incident 法務/Security Policy

26.8 Incident

なぜここを読むか 事故発生時の手順を口頭運用にすると、復旧と再発防止の両方が抜けます。隔離→Update停止→切分け→Rollbackまでを定型化する必要があります。

やる／やらないとどうなるか 端末/Project/Version特定→Update停止→Layer切分け→Credential失効→外部副作用確認→Rollback→Canary修正→Root Causeの8工程と、SEV1〜SEV4の重大度区分が共有されます。

これは何か Incident対応の8工程を、端末特定からRoot Cause/再発防止まで順に並べたRunbookの骨子です。

実行するとどうなるか 事故発生時に隔離・Rollback・通知・振り返りまでが定型で進み、属人的な深夜対応ではなくRunbook運転で復旧できるようになります。

それぞれの意味

1 端末/Project/Version特定
事故の発生端末・対象案件・配布版を最初に固める工程です。影響範囲を見誤ると過剰停止か被害拡大のどちらかが起きます。
2 Update停止
他端末への新規Updateを即時停止する工程です。同じ事故の連鎖発生を時間軸で止められます。
3 Layer切分け
Hook・Skill・MCP・本体・Policyのどの層が原因かを切り分ける工程です。手戻りなく対処層を特定できます。
4 Credential失効
漏えい疑いのTokenや鍵を取り消す工程です。原因不明でもまず信頼チェーンを切り、被害をその場で凍結します。
5 外部副作用確認
外部Deploy/Push/送信が発生したかを点検する工程です。社外向け被害の有無を把握し、二次対応の要否を決めます。
6 Rollback
直前Versionへ戻す復旧工程です。Update停止と組み合わせて、安定版での全社業務再開を最速で実現します。
7 Canary修正
修正版をいきなり全社配布せずCanaryで再検証する工程です。再発の同種事故を構造で防げます。
8 Root Cause/再発防止
根本原因分析とPolicy/Hookでの再発防止策を入れる工程です。同じ事故が二度起きない仕組み化までを含みます。
1 端末/Project/Version特定
2 Update停止
3 Layer切分け
4 Credential失効
5 外部副作用確認
6 Rollback
7 Canary修正
8 Root Cause/再発防止
Level 例
SEV1 Secret漏えい、本番破壊、全社停止
SEV2 複数端末阻害、誤配布
SEV3 Skill品質低下
SEV4 軽微文言/UI

26.9 KPI

なぜここを読むか 「全社AI基盤がうまく動いている」を主観で語っても経営判断には繋がりません。配布・信頼性・安全・品質・Adoption・効率・Costの数値目標が必要です。

やる／やらないとどうなるか Kit時間30分以内・Update成功99%+・Critical Incident 0・Stable Skill 85+・Active社員80%+などの目標値が並び、Block件数だけでなく誤検知率も含めた経営KPIになります。

区分 指標 目標例
配布 Kit時間 30分以内
配布 Stable反映 24h以内
信頼性 Update成功 99%+
安全 Critical Incident 0
品質 Stable Skill 85+
品質 Skill失敗 <5%
Adoption Active社員 80%+
効率 業務時間削減 業務別実測
Cost 1成果物Cost 基準内
