第29章

調べて、書いて、学ぶを、1つの机で回す 調査・Content・学習

『調査・記事執筆・学習を、別々のツールで往復している』。 これは効率の問題ではなく、机が分かれているだけです。

ほとんどの人は、調査用 AI と執筆用エディタと学習用ノートを別々に開いています。 ですが、Claude Code を中心に置くと、これらは同じ机の上で繋がります。

この章で扱うのは、調査結果を保存し、書きかけの原稿に接続し、学んだことをノート化するまでを 1 つの流れにする設計です。 情報源・草案・学習メモが、同じフォルダで同居します。

調べて、書いて、学ぶを、1 つの机で回す。情報の流れが滑らかになると、思考の質も変わります。

実装リファレンス

29.1 YouTube Research Skill

なぜここを読むか 動画調査は対象が膨大で、目的を持たずに集めると時間だけが消えます。テーマ・期間・Allowlist・既出題目を入力に固定したSkillへすると、調査が方針付きの構造化作業へ変わります。

やる／やらないとどうなるか 重複Cluster・Trend・Content Gap・出典付き提案が毎回同じ形で出るため、調査者が代わっても番組のラインが揺れなくなります。

これは何か Inputs/Workflow/Outputを構造化したYouTube調査Skillの定義例で、Verification noteまで必ず出力させることで一次情報確認を仕組みに組み込みます。

実行するとどうなるか 調査結果に出典と検証メモが揃い、後続の記事執筆や台本制作で『この主張の根拠は何か』を再度探す作業が消えます。

それぞれの意味

name / description
Skill名と用途説明で、AIがどの場面で呼び出すべきか判定する材料になります。曖昧だと別Skillと競合して呼ばれません。
argument-hint
呼び出し時に必要な引数のヒントで、ユーザーが何を渡せばよいかが構造化されます。なければ毎回プロンプトで聞き直しが発生します。
Inputs (audience / date range / allowlist / previous topics)
調査対象と禁止条件を固定する入力欄で、Skillが毎回同じ基準で動くことを保証します。曖昧な調査命令を構造化命令に変えます。
Workflow
Collect→Extract→Cluster→Verifyの7段で進める処理手順で、調査者が変わっても同じ品質の中間成果物が出ます。
Output (research table / trend clusters / content gaps / topic proposals / verification notes)
出力5点セットで、特にVerification notesを必須化することで出典確認が抜けにくくなります。
---
name: youtube-topic-research
description: Research videos in a defined niche, verify claims, identify content gaps, and propose evidence-backed topics aligned with a channel strategy.
argument-hint: "<theme> <date-range>"
---

# Inputs
- audience and positioning
- date range
- channel allowlist
- previous topics

# Workflow
1. Collect candidates and metadata.
2. Extract promise, structure, examples, claims.
3. Cluster duplicates.
4. Separate evergreen and volatile news.
5. Verify product claims with primary sources.
6. Compare with existing content.
7. Propose topics with novelty, evidence, and value.

# Output
- research table
- trend clusters
- content gaps
- topic proposals
- verification notes

29.2 番組Profile

なぜここを読むか 『らしさ』を会話の印象だけで運用すると、AIが書くたびにトーンが揺れます。Audience・Positioning・Tone・公開済みTopic・強い例・Do not doをファイル化すれば、外部評価可能な基準になります。

やる／やらないとどうなるか 新しい素材を選ぶ瞬間に基準を参照でき、人気の高い題材ではなく『自分の番組にとって価値ある題材』を選び続けられます。

これは何か 番組ProfileをFolder構造で分解し、Audience・Tone・Do not doなどを独立ファイルにする標準レイアウト例です。

実行するとどうなるか Profileの更新箇所がファイル単位で明確になり、AIがどの基準を参照すべきかが毎回同じ場所から取り出せます。

それぞれの意味

audience.md
視聴者像を定義するファイルで、想定読者の知識レベルと関心を明文化します。これが曖昧だとAIの口調と難易度が毎回揺れます。
positioning.md
他チャンネルとの差別化軸を書くファイルで、テーマ選定時に『どこに陣取るか』を一貫した基準で判断できます。
tone.md
口調・敬体・禁止表現を定義するファイルで、AIが書くたびにブレるトーンを外部評価可能な基準として固定できます。
published-topics.csv
公開済みTopic一覧で、同じ題材の重複公開を防ぐ参照表になります。なければAIが過去テーマを再提案しがちです。
strong-examples.md
自分の番組で評価が高かった事例集で、AIに『この方向で書け』と具体例で示せます。抽象指示より精度が上がります。
do-not-do.md
避けたい話題・表現・依頼を明示するファイルで、過去の炎上やNG事項を踏まないための安全弁になります。
content-profile/
├── audience.md
├── positioning.md
├── tone.md
├── published-topics.csv
├── strong-examples.md
└── do-not-do.md

公開成果物から「らしさ」を明文化し、会話の印象だけにしない。

29.3 SNS/動画から教材

なぜここを読むか SNSや動画の発言は重複と未検証情報の塊です。Topic ID→重複Cluster→公式確認→原則は本文・実例はRecipeという加工パイプにすると、教材化と事実確認が同じ流れで進められます。

やる／やらないとどうなるか 発信者が違うだけの同じ主張を一つに統合でき、要検証情報が分離されるため、誤情報を教材に混ぜる事故が防げます。

これは何か SNS/動画素材を教材化するための工程フローで、obsolete/currentとCoverage 100%を完了条件として最後に置きます。

実行するとどうなるか 発言の重複統合と公式確認が抜けない設計になり、教材ドラフトの品質が個人の注意力に依存しなくなります。

それぞれの意味

素材保存
SNS投稿や動画文字起こしを原文のまま保存する段で、後から発言の原典へ戻れる状態を作ります。要約から始めると検証不能になります。
Topic ID
発言や記事を主題単位でID付けする段で、後段の重複統合や引用管理の主キーになります。
重複Cluster
発信者違いの同じ主張を一つのClusterへまとめる段で、N人が同じことを言っている題材を1つに圧縮できます。
公式確認
Clusterの主張を公式情報源で裏取りする段で、SNS発言の又聞きをそのまま教材化する事故を防ぎます。
原則は本文/実例はRecipe
汎用化できる原則と一回性の実例を分けて格納する段で、教材本文の寿命を伸ばします。
obsolete/current
古びた情報を捨てずに状態管理する区分で、過去事例を残しつつ現行扱いだけを表へ出せます。
Coverage 100%
対象素材を全て処理し終えたことを確認する完了条件で、抜けたまま教材化される事故を構造的に塞ぎます。
素材保存 → Topic ID → 重複Cluster → 公式確認 →
原則は本文/実例はRecipe → obsolete/current → Coverage 100%

29.4 個人学習

なぜここを読むか 個人学習は『説明できる気がする』で終わりがちです。Hookを作りFixtureを通すことを完了条件にすると、理解の有無が機械的に判定できる学習ループへ変わります。

やる／やらないとどうなるか 復習問題・調査テーマ・進捗状態がINTERESTS.md連動で生成され、苦手と関心に沿った継続学習が個人ペースで成立します。

毎週：新概念3、復習10問、30分Hands-on、誤解、次の一次資料。

説明できる、ではなく、Hookを作り5 Fixtureを通す。

29.5 News Dashboard

なぜここを読むか ニュース巡回は重複と古い記事の再浮上で時間が削られます。期間・URL重複排除・公式上位・確度・古い記事検知を持つ単一HTMLへ集約すれば、毎朝の情報収集を10分の確認作業に変えられます。

やる／やらないとどうなるか 更新時刻・出典・未確認情報が常に表示されるため、見落としリスクを抑えながら情報取得時間を圧縮できます。

  • 対象期間。
  • URL重複排除。
  • 発表日/Event Date。
  • 公式上位。
  • 関心との関連。
  • 重要度/確度/業務影響。
  • 古い記事の再浮上検知。
