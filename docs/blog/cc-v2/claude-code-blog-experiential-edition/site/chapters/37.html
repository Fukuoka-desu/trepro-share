AIの古い情報を、自分の手で正す 素材と現行仕様の補正

『本書に書かれた手順が、半年後にはもう古い』。 これは著者の怠慢ではなく、AI 領域全体の性質です。

ほとんどの人は、教科書の情報を『そのまま信じる』前提で読みます。 ですが、本書はあえて、変動しやすい情報を本文の外（Feature Registry）へ逃がす設計にしています。

この章で扱うのは、素材と現行仕様のズレを補正する方法です。 古くなった部分を自分の手で正し、Registry を更新し、原則と詳細を切り分けて扱う作法を身につけます。

AI の古い情報を、自分の手で正す。それができる読者だけが、変化に強い実装者になります。

この章の概観 ／ 何を扱い、何ができるか

この章で扱う機能と、それで何ができるようになるか。実装に入る前に、全体像をここでつかみます。

  • 原則とFeatureの分離

    長く使える原則と、モデル名・料金・UIなど頻繁に変わるFeatureを別レイヤーで扱う設計方針。

    入力
    教材本文、変動情報の候補
    出力
    原則本文とRegistry分離後の構成

  • Feature Registry更新

    名前・状態・確認日・公式Source・代替手順をRegistryへ書き、本文を直さずに新仕様へ追随できる仕組み。

    入力
    現行Featureの最新情報、公式Source
    出力
    Registryに追加された確認日付きエントリ

  • Suspended / Deprecated 表示

    停止・非推奨を明示し、古い手順から新仕様への移行ガイドを必ず添えるバージョン管理。

    入力
    停止対象、代替Feature
    出力
    Suspended/Deprecatedラベルと移行ガイド

  • 素材→現行仕様補正表

    動画・記事など素材上の主張と、本書での現行扱いを対応付ける補正表。素材を否定せず使い続ける運用。

    入力
    素材の主張、現行仕様
    出力
    素材→現行扱いの対照表

実務に落とすときの補足

Registryに最低限書くべき項目
名前・状態（Active/Suspended/Deprecated）・確認日・公式Sourceの4項目は必須です。代替手順は任意ですが、Deprecated項目には必ず入れます。確認日のない項目はRegistryに置く意味がないため、新規追加時の必須欄として強制する仕組みにします。
本文とRegistryの境界線
『3ヶ月以内に変わりそうか』を判断基準にすると、本文へ書くか、Registryへ書くかが迷わず決まります。モデル名・料金・利用上限・UI・Beta状態はほぼ全てRegistry側です。本文に書いてよいのは、変わっても古びない設計思想と原則のみと割り切ります。
確認日の運用
確認日は『前回見たから動くと推定する有効期限』であり、過ぎたら必ず再確認します。3ヶ月を上限とし、過ぎたものはダッシュボードに赤で出します。確認日が形骸化したRegistryは、ない方がマシです。
